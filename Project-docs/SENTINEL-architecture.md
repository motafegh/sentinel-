# SENTINEL — Architecture & Project Reference

## What SENTINEL Is

A decentralized platform where AI agents autonomously audit smart contracts, generate zero-knowledge proofs of their analysis, coordinate via multi-agent systems, and serve results through blockchain oracles — continuously learning from new exploits.

**The one-sentence pitch:** "I built an AI security oracle that uses GNNs and CodeBERT to detect smart contract vulnerabilities, generates ZK proofs of the AI inference on-chain, and coordinates five specialized agents to produce verifiable audit reports."

---

| Stakeholder | Problem Solved |
|---|---|
| DeFi Protocols | Trustless, verifiable security audits without revealing audit methodology |
| Developers | Instant vulnerability detection before deployment |
| Insurance Providers | Cryptographic proof that contract was audited by certified AI model |
| Security Researchers | Platform for contributing to and learning from exploit database |

### Key Differentiators (for interviews)
1. **Verifiable AI** — ZK proofs ensure model predictions are correct without revealing model weights
2. **Multi-Modal Analysis** — GNN (structure) + Transformer (semantics) dual-path architecture
3. **Continuous Learning** — Model updates as new vulnerabilities emerge, with drift detection
4. **Cross-Chain** — Audit results synced across Ethereum, Polygon, Arbitrum via CCIP (stretch)
5. **Gasless UX** — ERC-4337 Account Abstraction enables audits without ETH (stretch)

## System Architecture — How All 6 Modules Connect

```
USER
  │ Upload .sol contract
  ▼
MODULE 6: API GATEWAY (FastAPI)
  │ POST /v1/audit → job_id
  │ Celery task queue (Redis)
  ▼
MODULE 4: AGENT ORCHESTRATION (CrewAI)
  ├─ Agent 1: Static Analyzer (Slither/Mythril)
  ├─ Agent 2: ML Intelligence ──────────────── calls MODULE 1
  ├─ Agent 3: RAG Researcher (FAISS)
  ├─ Agent 4: Code Logic Reviewer
  └─ Agent 5: Synthesizer → structured report
  ▼
MODULE 2: ZK PROOF GENERATION (EZKL)
  │ Proxy model (distilled from MODULE 1)
  │ Generates proof π + public risk_score
  ▼
MODULE 5: BLOCKCHAIN (Ethereum Sepolia)
  │ ZKMLVerifier.verify(proof, signals) → true
  │ AuditRegistry.submitAudit(...)
  │ Emit AuditSubmitted event
  ▼
MODULE 3: MLOps LAYER
  │ MLflow: tracks all MODULE 1 experiments
  │ DVC: versions MODULE 1 datasets
  │ Evidently: monitors MODULE 1 model drift
  │ Dagster: orchestrates retraining pipeline
  ▼
USER RECEIVES
  Vulnerability report + risk score (0-100)
  + on-chain verifiable ZK proof link
  + remediation suggestions
```

### Module Dependency Map

```
MODULE 1 (ML Core)
  ← no dependencies, builds first
  → output: trained model + inference API
  → feeds: Module 2 (needs model to distill), Module 4 (Agent 2 calls inference)

MODULE 2 (ZKML)
  ← depends on: Module 1 (trained model required)
  → output: ZK proof pipeline + ZKMLVerifier.sol
  → feeds: Module 5 (verifier contract deployed here)

MODULE 3 (MLOps)
  ← depends on: Module 1 (something to track/version)
  → output: experiment tracking + retraining pipeline
  → runs parallel to: Module 1 training

MODULE 4 (Agents)
  ← depends on: Module 1 (inference API), Module 3 (FAISS index)
  → output: multi-agent audit pipeline
  → feeds: Module 6 (agents called by API)

MODULE 5 (Solidity)
  ← depends on: Module 2 (ZKMLVerifier.sol generated by EZKL)
  → output: deployed contracts on Sepolia
  → feeds: Module 6 (API calls contracts)

MODULE 6 (Integration)
  ← depends on: all modules working
  → output: running system end-to-end
```

## Interview Talking Points

Use these as story templates. Fill in your actual numbers and specific decisions.

### "Tell me about a challenging ML architecture decision"
"In SENTINEL, I chose a dual-path architecture combining a GNN with CodeBERT. The GNN captures structural relationships in contract control flow — function call graphs, state variable access patterns. CodeBERT understands code semantics. I implemented a Gated Multimodal Unit to fuse them dynamically — the gate learns which path is more informative per contract. The trade-off was training complexity and compute cost vs. richer representation. Starting with CodeBERT-only MVP first let me validate the pipeline before adding GNN complexity."

### "How did you handle class imbalance?"
"Our dataset had ~60% safe contracts and under 5% reentrancy examples. I stacked three techniques: Focal Loss down-weights easy safe examples so the model focuses on hard vulnerability cases. Class weights in the loss function amplify rare classes — reentrancy weight was ~12x safe. Stratified sampling via WeightedRandomSampler ensures rare classes appear in every training batch. At inference, I used class-specific thresholds tuned on validation set — lower thresholds for rare classes to catch more positives."

### "What is ZKML and why does it matter?"
"ZKML uses zero-knowledge proofs to cryptographically verify that an ML model produced a specific output — without revealing the model's weights. In SENTINEL, any user can verify our vulnerability score came from the certified SENTINEL model, not a fabricated one. I used EZKL to convert a distilled proxy model to a ZK circuit, generate proofs per audit, and verify them on-chain in Solidity. The hardest part was size: the full model (~100K params) is too large for ZK circuits, so I built a 3K-parameter proxy using knowledge distillation that agrees with the full model over 95% of the time."

### "Describe a multi-agent system you built"
"SENTINEL uses a 5-agent CrewAI system for contract auditing. Agent 1 runs Slither and Mythril for static analysis. Agent 2 calls our GNN+CodeBERT inference API. Agent 3 does RAG retrieval over a FAISS index of 1000+ historical exploits — this grounds the analysis in real precedent. Agent 4 reviews business logic and access control patterns. Agent 5 synthesizes all findings into a structured report. The challenge was coordination — agents producing conflicting severity assessments. I used LangGraph for conditional flow: if risk score > 70, route to deeper code review before synthesis."

### "What advanced Solidity patterns have you used?"
"In SENTINEL: UUPS upgradeable contracts via OpenZeppelin — chosen over transparent proxy for gas efficiency. A staking/slashing tokenomics system where agents must stake SENTINEL tokens to submit audits, with slashing for dishonest submissions. Integration with an auto-generated ZK verifier from EZKL — the verification key is embedded as constants in the contract. For testing: fuzz tests in Foundry that discovered [specific edge case] that manual tests missed. Invariant tests that enforce 'no audit is ever stored without a valid ZK proof' across thousands of random action sequences."

### "How do you manage ML models in production?"
"SENTINEL's MLOps stack: MLflow tracks every experiment — hyperparameters, metrics, artifacts — and manages model versions in its registry. DVC versions datasets so every model is reproducible from a specific data snapshot. Evidently AI runs monthly drift detection on incoming contracts — if more than 10% of features drift, it triggers automated retraining via a Dagster pipeline. This means the model stays current as new vulnerability patterns emerge without manual monitoring."

## Fallback Decision Heuristics

### When to Persist vs Pivot

```
Persist (debug 1-2 more days):
  ✓ Error messages are clear and searchable
  ✓ Solution seems close (>80% working)
  ✓ Learning value is high — this is core technology
  ✓ Problem is in your code, not the external tool

Pivot to fallback:
  ✗ No clear error messages — mystery bugs
  ✗ Problem is inside the external tool/library itself
  ✗ Blocking all progress on other modules
  ✗ Alternative achieves 80% of the learning goal
  ✗ Time spent > 2 days with no visible progress
```

### Priority When Time Is Short

```
MUST HAVE (core learning — do not skip):
  Module 1: Transformer path (CodeBERT fine-tuning)
  Module 5: Basic contracts (AuditRegistry + ZKMLVerifier placeholder)

SHOULD HAVE (impressive, high interview value):
  Module 2: ZKML (even if slow/inefficient)
  Module 4: Basic RAG agent (single agent + 3 tools)

NICE TO HAVE (resume boosters):
  Module 3: MLflow + DVC (basic setup is enough)
  Module 6: Docker Compose (local is fine)

STRETCH (only if time allows):
  GNN dual-path, multi-agent CrewAI, Kubernetes, Next.js frontend,
  Chainlink CCIP, ERC-4337, Feast, full drift detection
```
