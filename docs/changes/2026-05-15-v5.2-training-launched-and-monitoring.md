## Trainer Fixes Applied (this session)

### Fix: Per-run log file handler (loguru, not stdlib logging)

**Problem:** `ml/logs/v5.2-jk-20260514.log` was created but stayed 0 bytes. trainer.py
uses `loguru` for logging, but the file handler was added via `logging.FileHandler()` from
the stdlib `logging` module — a different logging system that loguru does not write to.

**Fix:** Replaced stdlib `logging.FileHandler` with `logger.add()` (loguru API):
```python
logger.add(
    str(_run_log_path),
    level="DEBUG",
    format="{time:YYYY-MM-DD HH:mm:ss} | {level:<8} | {name} | {message}",
    mode="a",
    encoding="utf-8",
)
```
Removed the now-unused `import logging` from the top of the file.

Takes effect from the NEXT run (the current v5.2-jk-20260514 run was already started with
the broken handler; its output goes to nohup.out or the terminal only).

### Feature: Smoke auto log_interval

**Problem:** With `--smoke-subsample-fraction 0.1` and `--gradient-accumulation-steps 4`,
there are only ~49 optimizer steps per epoch. The default `log_interval=100` is never
reached, so the GNN-share gate cannot be observed in smoke runs.

**Fix:** In `train()`, if `steps_per_epoch < log_interval`, automatically lower to
`max(5, steps_per_epoch // 4)`. For the standard smoke config this gives ~12 steps,
producing 4 log points per epoch instead of zero.

### Feature: Per-eye loss at each log line

**Problem:** The step-level log showed total loss and per-eye grad norms, but not per-eye
auxiliary loss values. This made it impossible to see which path (GNN / TF / fused) was
struggling early in training.

**Fix:** Save per-eye losses during the forward pass and accumulate running averages
between log intervals. Each step log now shows:
```
Step 12/487 | loss=0.8863 [eyes: gnn=0.9341 tf=0.8892 fused=0.9021] | \
  grad: gnn=0.031 tf=0.018 fused=0.024 | GNN share=32.9%
```

### Feature: gnn_grad_share logged to MLflow per epoch

`train_one_epoch` now returns `(avg_loss, nan_count, last_gnn_share)`. The share from the
last log interval of each epoch is logged as `mlflow.log_metric("gnn_grad_share", ...)`.
This makes the GNN share trend visible in the MLflow chart without scanning log files.

### Feature: Resume command printed at startup

At the start of each `train()` call, the exact resume command for this run is logged:
```
TRANSFORMERS_OFFLINE=1 PYTHONPATH=. python ml/scripts/train.py \
    --resume ml/checkpoints/v5.2-jk-20260514_best.pt \
    --run-name v5.2-jk-20260514-resumed \
    --experiment-name sentinel-v5.2 \
    --epochs 60 \
    --gradient-accumulation-steps 4
```

## Naming Convention (established)

All runs must include the date in `--run-name` to ensure checkpoints, log files, and
MLflow runs are unique and never overwrite each other:

| Artifact | Path |
|---|---|
| Checkpoint | `ml/checkpoints/<run-name>_best.pt` |
| Patience sidecar | `ml/checkpoints/<run-name>_best.state.json` |
| Log file | `ml/logs/<run-name>.log` (append — safe for resume) |
| MLflow run | experiment `sentinel-v5.2`, run named `<run-name>` |

Examples: `v5.2-smoke-20260515`, `v5.2-jk-20260515`, `v5.2-no-jk-20260515`
