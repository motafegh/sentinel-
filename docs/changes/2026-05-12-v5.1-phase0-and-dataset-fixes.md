#### Fix 0a — `_select_contract()` interface bug (graph_extractor.py)

**Before:** `candidates[0]` picked interfaces when they appeared first in the `.sol` file.
Result: ~10% of 68K training graphs were ghosts (2 nodes, 0 edges).

**After:**
```python
non_iface = [c for c in candidates if not c.is_interface]
if non_iface:
    return max(non_iface, key=lambda c: len(c.functions))
```

Note: `is_abstract` does not exist in Slither 0.10.0. Using `is_interface` only is correct.
Abstract contracts are naturally de-prioritised by the `len(c.functions)` sort.

**Verified:** 11_external_bug.sol (was ghost → 19 nodes, 28 edges, contract=LendingPool),
13_multilabel_complex.sol (was ghost → 24 nodes, 40 edges, contract=MultiVulnStaking).

Also added `sol_path = Path(sol_path)` coercion at function entry to prevent AttributeError
when callers pass a string.

#### Fix 0b — Function-level GNN pooling (sentinel_model.py)

**Before:** `global_max_pool + global_mean_pool` over ALL node embeddings.
77% of CFG node mass = CFG_RETURN nodes (uninformative exit-point markers).
GNN eye gradient share collapsed to 6.7% by epoch 43.

**After:** Pool only over FUNCTION / MODIFIER / FALLBACK / RECEIVE / CONSTRUCTOR nodes
(type IDs {1, 2, 4, 5, 6}). After Phase 3 reverse-CONTAINS aggregation, these nodes
carry CFG ordering signal propagated up from their CFG children.
Ghost-graph fallback to all-node pool prevents crashes on interface-only contracts.

```python
_FUNC_TYPE_IDS = {1, 2, 4, 5, 6}
node_type_ids = (graphs.x[:, 0] * 12.0).round().long()
pool_mask = torch.zeros(node_embs.size(0), dtype=torch.bool, device=node_embs.device)
for tid in _FUNC_TYPE_IDS:
    pool_mask |= (node_type_ids == tid)
if pool_mask.any():
    pool_embs, pool_batch = node_embs[pool_mask], batch[pool_mask]
else:
    pool_embs, pool_batch = node_embs, batch   # ghost-graph fallback
```

#### Fix 0c — aux_loss_weight 0.1 → 0.3 (train.py)

Default for `--aux-loss-weight` raised from 0.1 to 0.3. At 0.1 the auxiliary gradient
signal was insufficient to prevent GNN eye dominance collapse when fused eye became
dominant from epoch 23 onward.

#### Fix 0e — CFG failure counter (graph_extractor.py)

`_build_control_flow_edges()` previously swallowed all exceptions silently. Added
`_cfg_failure_count` counter per contract; logs `logger.error` if failure rate > 5%,
`logger.debug` otherwise. Benign (single-statement functions) remain debug-level.

#### Phase 1 Skip Analysis — Critical Finding

12,447 contracts (18.2%) raised `SolcCompilationError`. All from BCCC
`SourceCodes/`, all declaring 0.4.x / 0.5.x pragmas.

**Root cause:** `_detect_solc_version()` used the exact version from the
pragma (e.g. `^0.4.18` → `solc-0.4.18`). BCCC contracts frequently declare
an old minimum but use syntax introduced in later patches:

| Error type | Share | Example |
|---|---|---|
| `emit EventName()` with solc < 0.4.21 | ~80% | emit requires ≥ 0.4.21 |
| `constructor()` with solc < 0.4.22 | included above | |
| `calldata` explicit location in 0.4.x | ~7% | requires 0.5.x |
| Multi-pragma version conflict | ~5% | two conflicting pragmas in one file |

**Fix (commit 65617e1):** `_detect_solc_version()` now maps the detected minor
to the latest installed patch: `0.4.x → 0.4.26`, `0.5.x → 0.5.17`, etc.
Added `_LATEST_PATCH` dict; default fallback changed from `0.8.20` to `0.8.31`.

### Dataset Deduplication — Unexpected Critical Finding (Phase 0f)

During split verification, content-hash analysis revealed a structural problem with the
original 68K dataset that invalidated all v5.0 (and v4) validation metrics.

#### Root Cause

BCCC-SCsVul-2024 stores the **same `.sol` file in multiple category directories** — one
copy per vulnerability class a contract exhibits. For example, a contract with
IntegerUO + Reentrancy + Timestamp vulnerabilities appears as:
```
SourceCodes/IntegerUO/abc.sol     → path-MD5: aaa111...  label: IntegerUO=1
SourceCodes/Reentrancy/abc.sol    → path-MD5: bbb222...  label: Reentrancy=1
SourceCodes/Timestamp/abc.sol     → path-MD5: ccc333...  label: Timestamp=1
```
Each copy has a different path-MD5 (the filename used for graph/token `.pt` files).
Random splitting scattered these copies across train/val/test.

#### Measured Impact

| Metric | Value |
|--------|-------|
| Original rows | 68,523 |
| Unique content hashes | 44,420 |
| Cross-split leaking groups | 7,630 |
| Leaky rows (% of total) | 34.9% |
| Contracts leaking train→val | 4,169 |
| Contracts leaking train→test | 4,178 |
| Contracts leaking val→test | 353 + 1,070 (all-3) |

**Consequence:** v5.0 (and v4) val/test F1 scores were inflated — model saw identical
contract source during training that appeared in validation. The degree of inflation is
unknown but non-negligible given 34.9% exposure.

**Label quality also degraded:** each copy carried only its category label. A contract
in three categories had three rows each with a single `1`. The model received
contradictory training signal: same graph, different labels. Deduplication with OR-merge
produces the correct multi-label annotation.

#### Fix Applied

**Script:** `ml/scripts/dedup_multilabel_index.py`

1. Scans all source dirs, builds path-MD5 → content-MD5 map (vectorised, 22s).
2. Groups rows by content hash; OR-merges labels (`max()` per class).
3. Keeps alphabetically-first path-MD5 as canonical (its `.pt` files reused).
4. Writes `ml/data/processed/multilabel_index_deduped.csv` (44,420 rows).
5. Rebuilds stratified 70/15/15 splits with `seed=42` into `ml/data/splits/deduped/`.

## Implications for v5.1 Gates

The dataset deduplication means v5.1 metrics will NOT be directly comparable to v5.0.
The v4 per-class floors (used as the v5.0 gate) were also measured on the leaky dataset.
New gates should be set relative to v5.1 baseline metrics.

Recommended v5.1 gates:
- Tuned F1-macro > 0.55 (relaxed from 0.60 due to honest/smaller dataset)
- DoS tuned F1 > 0.30 (relaxed from 0.45 — true count is 377, not 995)
- Behavioral: 70% detection rate, 66% safe specificity
- Mandatory CEI pair A (vulnerable) fires, B (safe) silent — absolute requirement
