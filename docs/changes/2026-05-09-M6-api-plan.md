## 2. Plan A — Security Design (must precede any route)

Per `SENTINEL-M5-M6-PLATFORM.md` §"Security Design":

| Concern | Decision |
|---|---|
| Auth | `Authorization: Bearer <key>` header, FastAPI `Depends()` |
| Key storage | Env var `SENTINEL_API_KEYS` (comma-separated); never hardcoded |
| Rate limit | `slowapi` token bucket (Redis-backed); 10 audits/min/key |
| Input size | 500 KB max contract source; reject before Slither/preprocess |
| Encoding | UTF-8 only; reject non-UTF-8 with 400 |
| Confidentiality | Audit payloads logged at DEBUG only (never INFO/WARN) |
| TLS | Terminate at reverse proxy (nginx/caddy); not in-app |
| CORS | Allow-list specific origins; no `*` |
| Error surfaces | Never leak internal stack traces; generic 500 + correlation id |

New file:

```
api/SECURITY.md       NEW — single-source-of-truth for the table above,
                      keyed off correlation IDs, and incident-response
                      ladder. Reference from api/main.py docstring.
```

Acceptance: ADR (append to `SENTINEL-ADR.md`) recording the auth scheme
and rate-limit storage choice.

## 4. Plan C — Endpoint Contracts

### 4.1 `POST /v1/audit`

Request:
```json
{ "contract_source": "<solidity utf-8>", "language_version": "0.8.20" }
```
Response (202):
```json
{ "job_id": "<uuid>", "status": "queued" }
```
Behaviour:
- Reject if size > 500 KB or non-UTF-8 (400)
- Hash payload (SHA-256), store in `AuditJob.payload_hash`
- Enqueue `audit_task` with `job_id`
- Audit task: build graph → predictor → if high-risk, RAG + on-chain
  history → synthesizer → write `result_jsonb` → optionally enqueue
  `proof_task`

### 4.2 `GET /v1/audit/{id}`

Response (200):
```json
{ "job_id": "...", "status": "queued|running|done|error",
  "result": { "vulnerabilities": [...], "thresholds": [...],
              "rag_evidence": [...], "audit_history": [...] } }
```

Note `thresholds` (list) per `SENTINEL-INDEX.md` Critical Rules.
**Never** emit a `confidence` field — explicitly removed in Track 3.

### 4.3 `GET /v1/proof/{id}`

Response: `{ "proof": "<hex>", "publicSignals": [...uint256],
"on_chain_tx": "0x..." | null }`. 404 until proof_task completes.

### 4.4 `GET /health`

Aggregates: M1 `/healthz`, MCP servers SSE reachable,
Postgres SELECT 1, Redis PING. Returns 200 only if all green; otherwise
207 with per-service detail.

## 6. Plan E — Phased Build

Suggested phasing so the team always has something runnable:

1. **Phase 1: scaffold** — `api/main.py`, `/health`, auth dep, no Celery yet.
   Synchronous call into `predict()` for `/v1/audit`. Goal: prove auth +
   rate-limit + 500 KB guard work end-to-end.
2. **Phase 2: Celery** — move audit work to background, add `GET
   /v1/audit/{id}` polling. Postgres for `AuditJob`.
3. **Phase 3: graph integration** — replace direct `predict()` with
   `agents.build_graph().ainvoke()` so RAG + audit_check fire.
4. **Phase 4: proof** — wire `proof_task` (gated on M2 Option A).
5. **Phase 5: docker-compose** — full local stack.
6. **Phase 6: CI/CD** — GitHub Actions; spec §M6 has the matrix.

Each phase ends with a green pytest run and a dated changelog.

## 7. Operational Considerations

- **Idempotency** — `POST /v1/audit` returns the existing `job_id` if
  the SHA-256 of the payload matches an existing non-error job within
  the last 24 hours. Prevents accidental double-billing.
- **Replay attacks** — none in v1 (no signing); document this and add
  request-signing to a future sprint.
- **Cost cap** — per-key daily quota on top of the 10/min token bucket.
  Configurable via env var.
- **Backpressure** — Celery queue depth > N → return 429 instead of 202.
