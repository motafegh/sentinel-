# v5.1 Smoke Run Fixes — 2026-05-12

## Summary

Two blocking bugs prevented the v5.1 smoke run from completing even one batch.
Both root causes were in the training data pipeline, not the model itself.
After fixes, the smoke run is live with GNN gradient share 27–75% across
the first 400 batches (gate ≥ 15% ✅).

---

## Bug 1 — PyG Collation Crash: `KeyError: 'contract_hash'`

### Root Cause

The v5.1 re-extraction (`reextract_graphs.py`) correctly follows
`graph_extractor.py`'s design contract:

```
# graph_extractor.py line 48:
# Caller-specific metadata (.contract_hash, .contract_path, .y) is NOT set
```

However, **280 stale canonical graphs** were not re-extracted in Phase 1
(0.6% of 44,420 — contracts with `^0.4.x` pragma + 0.5-syntax that
genuinely fail compilation). These were originally produced by
`ast_extractor.py` which sets `data.contract_hash = md5` at line 233.

When PyG's `Batch.from_data_list()` collates a batch containing both fresh
(no `contract_hash`) and stale (has `contract_hash`) graphs, it discovers
the attribute on some objects, tries to collate it across all objects in the
batch, finds it absent on others, and raises `KeyError: 'contract_hash'`.

Same issue applies to: `contract_path`, `contract_name`, `node_metadata`,
`num_edges`, `num_nodes`, `y` — all present on stale graphs, absent on fresh.

### Fix Applied

**`ml/src/datasets/dual_path_dataset.py` — strip metadata in `__getitem__`:**

```python
# Strip stale v5.0 metadata so PyG collation never sees mixed attributes
for _k in ["contract_hash", "contract_path", "contract_name",
           "node_metadata", "num_edges", "num_nodes"]:
    if hasattr(graph, _k):
        delattr(graph, _k)
```

Runs in the DataLoader worker process before any collation — safe for both
fresh (no-op) and stale (strips attribute) graphs.

**`ml/src/datasets/dual_path_dataset.py` — `exclude_keys` in collate fn:**

```python
_EXCLUDE = ["contract_hash", "contract_path", "contract_name",
            "node_metadata", "num_edges", "num_nodes", "y"]
batched_graphs: Batch = Batch.from_data_list(graphs, exclude_keys=_EXCLUDE)
```

Defence-in-depth: even if a stale attribute survives `__getitem__`,
the collator will not attempt to batch it.

## Bug 2 — Custom `collate_fn` Silently Ignored

### Root Cause

`trainer.py` imported `DataLoader` from `torch_geometric.loader`:

```python
# Before fix
from torch_geometric.loader import DataLoader
```

PyG's `DataLoader` subclass overrides the collation mechanism and ignores
the `collate_fn` kwarg entirely — it always uses its own internal `Collater`.
So `dual_path_collate_fn` (with `exclude_keys`) was passed but never called.

### Fix Applied

**`ml/src/training/trainer.py`:**

```python
# After fix
from torch.utils.data import DataLoader
```

`torch.utils.data.DataLoader` respects `collate_fn`. PyG's
`Batch.from_data_list()` is still called inside `dual_path_collate_fn`,
so graph batching behaviour is identical. The only change is that the
custom collate function now actually executes.

## Impact Analysis

- **Re-extraction not needed**: stale 280 graphs are still valid (correct 12-dim
  features, correct edges); only their metadata attributes were extra
- **Cache valid**: `cached_dataset_deduped.pkl` stores raw graph objects;
  metadata stripping happens at collation time, not at cache-build time
- **Token files unaffected**: completely independent of graph attributes
- **inference/preprocess.py unaffected**: sets `graph.contract_hash` intentionally
  for its own cache-key logic — that path does not go through `dual_path_collate_fn`
