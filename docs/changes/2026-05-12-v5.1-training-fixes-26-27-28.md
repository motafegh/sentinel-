# v5.1 Training Fixes #26/#27/#28 — 2026-05-12

## Context

After the smoke run was unblocked (see `2026-05-12-v5.1-smoke-run-fixes.md`),
a pre-Phase-3 audit of the training loop identified three more issues:
one correctness bug (grad norms always logged as 0.000) and two efficiency
improvements (VRAM leak between epochs, unnecessary MHA weight computation).
All three were fixed before the 60-epoch Phase 3 run was launched.

---

## Fix #26 — `need_weights=False` on MHA in `fusion_layer.py`

### Root Cause

`CrossAttentionFusion.__init__` creates two `nn.MultiheadAttention` modules:
`self.node_to_token` and `self.token_to_node`. Both were called without
`need_weights=False`:

```python
# Before — default need_weights=True
attn_out, _ = self.node_to_token(query, key, value)
attn_out, _ = self.token_to_node(query, key, value)
```

When `need_weights=True` (the default), PyTorch computes the full attention
weight matrix and materialises it — even though we discard it with `_`.
This blocks the fused efficient-attention kernel (FlashAttention-style path)
and wastes ~12.6 MB VRAM per forward pass at the current `attn_dim=256, heads=8`
configuration.

### Fix Applied

**`ml/src/models/fusion_layer.py`:**

```python
# After — suppresses weight computation; enables fused kernel
attn_out, _ = self.node_to_token(query, key, value, need_weights=False)
attn_out, _ = self.token_to_node(query, key, value, need_weights=False)
```

Zero behaviour change — the returned weight tensor was never used.
Commit: `df75466`

## Fix #27 — `gc.collect()` + `torch.cuda.empty_cache()` Between Epochs

### Root Cause

After `evaluate()` returns, Python's garbage collector has not necessarily
freed all intermediate tensors allocated during validation. The CUDA caching
allocator holds onto freed blocks in its own pool, which is not released back
to CUDA until explicitly requested. Over a long run (60 epochs), this causes
the "reserved but not allocated" VRAM wedge to grow, leaving less headroom
for the next epoch's training and increasing the risk of OOM on smaller GPUs.

### Fix Applied

**`ml/src/training/trainer.py`:**

```python
import gc  # added at module level

# Inside epoch loop, after evaluate() returns:
val_loss, val_f1, val_metrics = self.evaluate(val_loader)
gc.collect()
torch.cuda.empty_cache()
```

This runs between epochs only (not mid-batch), so there is no throughput impact.
Commit: `df75466`

## Fix #28 — Grad Norm Logging Always 0.000

### Root Cause

This was a correctness bug — all three grad norm readings (`gnn_eye`,
`tf_eye`, `fused_eye`) were logged as `0.000` for the entire v5.1 smoke run,
making it impossible to detect gradient collapse.

The logging call was positioned **after** `optimizer.zero_grad(set_to_none=True)`:

```python
# Before — wrong order
scaler.step(optimizer)
scaler.update()
scheduler.step()
optimizer.zero_grad(set_to_none=True)   # ← wipes .grad to None

# ... log_interval check ...
gnn_norm = _grad_norm(self.model.gnn_eye)  # ← .grad is None → returns 0.0
```

With `set_to_none=True`, `.grad` is set to `None` (not zeroed), so
`_grad_norm()` computes `sqrt(sum(None))` which evaluates to `0.0`.

Additionally, `log_interval` was counting **micro-batches** rather than
**optimizer steps**, so with `accum_steps=4` the reading was firing 4× more
often than intended — every 100 micro-batches = every 25 optimizer steps.

### Fix Applied

**`ml/src/training/trainer.py`:**

```python
# After — correct order inside is_accum_step block
if is_accum_step:
    scaler.unscale_(optimizer)           # 1. grads → fp32
    clip_grad_norm_(params, max_norm)    # 2. clip
    optimizer_step += 1                  # 3. increment BEFORE logging
    if optimizer_step % log_interval == 0:
        gnn_norm   = _grad_norm(self.model.gnn_eye)   # 4. READ — grads exist
        tf_norm    = _grad_norm(self.model.tf_eye)
        fused_norm = _grad_norm(self.model.fused_eye)
        logger.info(f"Optimizer step {optimizer_step} ...")
    scaler.step(optimizer)               # 5. update weights
    scaler.update()
    scheduler.step()
    optimizer.zero_grad(set_to_none=True)  # 6. wipe grads — AFTER logging
```

A new `optimizer_step` counter (initialized to 0 before the epoch loop)
ensures `log_interval` counts optimizer steps, not micro-batches.
Commit: `db2277f`

### Confirmation

First reading at optimizer step 100 (batch 400/1947) on Phase 3 run `v5.1-fix28`:

```
gnn_eye=0.601  tf_eye=0.170  fused_eye=0.191
```

All three eyes non-zero. GNN/TF ratio = 3.5× — GNN is the dominant gradient
contributor, confirming the Phase 0b function-level pooling fix is working.
