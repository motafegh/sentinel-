## 4. Root Cause Analysis

### RC1 — Fusion Module LR Dominance (confirmed)

The CrossAttentionFusion (821,888 params) and transformer_eye_proj (98,432 params),
classifier (3,850), and aux heads (3,870) all ran at the full learning rate (lr=2e-4) in v5.2.

This created a 4-way imbalance in effective parameter-weighted gradient scale:
- GNN: 70,272 params at lr=2e-4 (but with 2.5× multiplier = lr_eff=5e-4)
- LoRA: 589,824 params at lr=1e-4
- Fusion+classifier+proj: 927,040 params at lr=2e-4 (the problem group)

The fusion module has 13× more parameters than the GNN and was running at 2× the GNN's
base LR. In early training, fusion gradients dominated the shared optimizer step, pulling
the entire model toward CodeBERT-driven predictions before the GNN had time to learn.

**Evidence:** Gradient norms logged at epoch 1 showed fused=0.136–0.307 vs gnn=0.027–0.074.
The GNN was updating at ~4-5× lower gradient scale than the fusion path.

### RC2 — BCE pos_weight + Data Co-occurrence (confirmed, root cause of Reentrancy collapse)

v5.2 used `BCEWithLogitsLoss` with sqrt-scaled pos_weight. For Reentrancy:
- Train samples: 5,000 positive / ~26,000 negative → pos_weight ≈ 2.82
- Every Reentrancy false negative was penalised 2.82× harder than a false positive

The BCCC-SCsVul-2024 dataset has high co-occurrence between external calls and Reentrancy
labels: contracts labelled Reentrancy frequently contain `call{}`, `.transfer()`, or
`.send()`, which also appear in non-Reentrancy contracts (e.g., contracts labelled
CallToUnknown, ExternalBug, or safe transfer helpers).

The 2.82× FN penalty + external-call co-occurrence taught the model:
**"any contract with an external call is probably Reentrancy"**

This is not fixable by threshold tuning — the model's internal probability for safe contracts
with external calls (0.95–0.97) exceeds its probability for some true Reentrancy cases.

### RC3 — No Label Smoothing (confirmed contributing factor)

With hard binary targets (0.0 / 1.0), BCE drives outputs toward log(0) and log(1).
The model had no penalty for extreme confidence. Once the Reentrancy association was learned,
the loss rewarded pushing probabilities toward 0.97+ on any contract with external calls.

### RC4 — Data Starvation for Timestamp and DoS (ongoing, not fixed)

```
DoS:       Train=257  Val=54   Test=54
Timestamp: Train=322 (estimated)  Val=322
```

These two classes had insufficient training signal regardless of loss function.
Timestamp recall was 0.124 (val) and 0/1 (behavioral). DoS was 0/1 (behavioral).
No amount of loss reweighting compensates for 257 training examples across 31K.

## 5. v5.3 Changes Made (2026-05-16)

The following changes were made to `ml/src/training/trainer.py` and `ml/scripts/train.py`
after the behavioral failure:

### RC1 Fix — Four-Group Optimizer (fusion at 0.5× LR)

**Before (v5.2):** Three param groups — GNN (2.5×), LoRA (0.5×), Other (1.0×).
Fusion, transformer_eye_proj, classifier, and aux heads all fell into "Other" at 1.0×.

**After (v5.3):** Four param groups:
```
GNN (gnn.* + gnn_eye_proj.*):       lr × 2.5
LoRA (params with "lora_" in name): lr × 0.5  weight_decay=0
Fusion (fusion.* + transformer_eye_proj.* + classifier.* + aux_*): lr × 0.5
Other:                              lr × 1.0  (empty in current config)
```

Fusion now trains at the same rate as LoRA, preventing it from dominating early gradient steps.

**Code change:** `ml/src/training/trainer.py` — `_fusion_params` list and param group routing.  
**CLI:** `--fusion-lr-multiplier` (default 0.5) added to `ml/scripts/train.py`.

### RC3 Fix — Label Smoothing ε=0.05

Soft targets: positive → 0.95, negative → 0.05. Applied per-batch before loss computation:
```python
if label_smoothing > 0.0:
    labels = labels * (1.0 - label_smoothing) + 0.5 * label_smoothing
```

Prevents extreme confidence by keeping gradients nonzero even for high-probability predictions.

**Code change:** `train_one_epoch()` signature gains `label_smoothing: float = 0.0` param.  
**CLI:** `--label-smoothing` (default 0.05) added to `ml/scripts/train.py`.

### RC2 — Focal Loss (under review)

**What was done:** Default `loss_fn` changed from `"bce"` to `"focal"` (γ=2.0, α=0.25).
The reasoning given was: "focal loss down-weights confident predictions and forces focus on
hard uncertain examples; v4 used focal and didn't show Reentrancy collapse."

**Why this reasoning is wrong:**

1. **v4 did not use focal loss.** v4 used `BCEWithLogitsLoss`. The statement "v4 used focal"
   was incorrect. v4 performed better behaviorally while using BCE, which means the loss
   function is not the primary differentiator.

2. **Focal loss is not binary-only, but the rationale was wrong.** FocalLoss applied
   element-wise is valid for multi-label classification. But the motivation here was wrong:
   focal loss addresses easy/hard example imbalance. The Reentrancy collapse is a
   *data co-occurrence problem*, not an easy/hard problem. Focal loss would not prevent the
   model from associating external calls with Reentrancy — it would simply modulate the
   gradient scale, which does not change the learned association direction.

3. **FocalLoss is AMP-unsafe.** The `focalloss.py` implementation uses
   `F.binary_cross_entropy` which PyTorch explicitly disallows inside `torch.cuda.amp.autocast`.
   The v5.3 training run crashed at batch 1 of epoch 1 with:
   ```
   RuntimeError: torch.nn.functional.binary_cross_entropy and torch.nn.BCELoss are unsafe
   to autocast. ... use binary_cross_entropy_with_logits instead.
   ```

**Current status:** v5.3 with focal loss is crashed and not running.

**What actually addresses RC2:** The data co-occurrence problem requires one or more of:
- Reducing pos_weight for Reentrancy specifically (it has 5,000 samples — not actually rare)
- CEI contrastive pairs (Phase 3-1): safe contracts with external calls labelled explicitly
  as not-Reentrancy, to teach the distinction
- Inspecting BCCC Reentrancy labels for false positives (contracts labelled Reentrancy
  that merely contain external calls but have no state-change-before-call pattern)

## 6. What v5.3 Should Actually Be

Based on the above analysis, the changes that are well-motivated and should be kept:

| Change | Motivated | Keep |
|--------|-----------|------|
| RC1: Fusion LR 0.5× (4-group optimizer) | Yes — gradient imbalance confirmed | ✅ |
| RC3: Label smoothing ε=0.05 | Yes — prevents extreme overconfidence | ✅ |
| RC2: Focal loss | No — wrong rationale, wrong problem, AMP crash | ❌ revert to BCE |

The correct v5.3 configuration is **BCE + RC1 + RC3**. Additionally, the Reentrancy collapse
requires addressing the data problem (RC4-equivalent for Reentrancy): the pos_weight for
Reentrancy should be reviewed given 5,000 training samples make it a moderately represented
class, not a rare one. The sqrt-scaled pos_weight gives it 2.82× — a 3× false negative
penalty on a class with 5,000 samples may be too aggressive.

### Recommended v5.3 Launch Command

```bash
TRANSFORMERS_OFFLINE=1 PYTHONPATH=. nohup python ml/scripts/train.py \
    --run-name v5.3-bce-smooth-20260516 \
    --experiment-name sentinel-v5.3 \
    --loss-fn bce \
    --epochs 60 \
    --gradient-accumulation-steps 4 \
    --early-stop-patience 20 \
    --eval-threshold 0.35 \
    > ml/logs/v5.3-bce-smooth-20260516-stdout.log 2>&1 &
```

This uses: BCE loss (same as v4) + fusion_lr_multiplier=0.5 (new RC1) + label_smoothing=0.05
(new RC3) without the AMP-unsafe focal loss.

## 7. Gates for v5.3 Pass/Fail

### Quantitative gates (val set, tuned thresholds)
- Tuned macro F1 ≥ 0.5422 (v4 baseline on deduped val)
- Per-class F1 ≥ v4_F1 − 0.05 for all 10 classes

### Behavioral gates (mandatory — all must pass)
- Detection rate ≥ 70% (14/19+ behavioral contracts)
- Safe specificity: 3/3 safe contracts correctly clean
- Reentrancy probability on safe contracts < threshold (no false Reentrancy on 12/18/19)
- Timestamp fires on contracts 04 and 13
- DoS fires on contract 05

### Hard fails (any one = reject)
- Reentrancy fires on safe_contract (12), safe_no_calls (18), or safe_with_transfer (19)
- IntegerUO does not fire on 03 and 17
- More than 3 classes fire on any single safe contract
