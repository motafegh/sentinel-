## 0. Learning Context (read this first)

SENTINEL is a learning sandbox for becoming a hybrid engineer who
understands ML, ZK, smart contracts, and the interfaces between them.
The point is **exposure, understanding, and informed design choices** —
not protecting a fixed stack.

What that means for the plans here:

- **The "locked" constants are locked for the v4 sprint only.** They keep
  v4 numbers comparable to v3's 0.5069 baseline. They are not a
  permanent freeze. After v4 lands, the Architecture Playground (§5)
  is the place to break them deliberately.
- **The downstream rebuild is already due.** `zkml/` and
  `contracts/` were built for the **binary** detector and were never
  ported to the 10-class multi-label model: `CIRCUIT_VERSION` was bumped
  from v1.0 (binary) to v2.0 (multi-label) in source, but the EZKL
  pipeline has never been run, no `proving_key.pk` exists, and
  `ZKMLVerifier.sol` has not been regenerated. So the conventional
  "changing the architecture forces a full downstream rebuild" argument
  is much weaker for us — that rebuild is already needed regardless.
  This actively *lowers* the cost of upstream architectural exploration
  later.
- **FocalLoss is an experiment in our setting, not a proven default.**
  It was originally written for the binary classifier; the multi-label
  wrapper in `trainer.py` exists but has never been validated end-to-end
  in production. v4 is the first real test of whether it helps for our
  10-class problem. Treat it as a hypothesis, not a recipe.
- **The autoresearch harness is a stepping stone.** v4 uses it for
  hyperparameter search over a fixed model. Post-v4, the same harness
  should grow a `--model-file` argument so it can compare *different
  model files* (different GNN backbones, fusion designs, encoders).
  See §3.3.

The v4 sprint is intentionally conservative because the goal is to
beat 0.5069 on the existing architecture before going wild. Everything
exploratory lives in §5 and after.

## 2. Plan A — v4 Retrain (Primary)

Goal: tuned val F1-macro **> 0.5069** on `ml/data/splits/val_indices.npy`,
no per-class drop > 0.05 F1 from v3 (per-class floors in
`SENTINEL-EVAL-BACKLOG.md` §"v3 Per-Class Floor Values").

### 2.1 Pre-flight checklist (must all pass)

- [ ] `ml/data/splits/val_indices.npy` is the **same file** used by v3 (do NOT regenerate)

- [ ] No uncommitted changes to: `trainer.py`, `sentinel_model.py`,
      `gnn_encoder.py`, `transformer_encoder.py`, `dual_path_dataset.py`,
      `graph_extractor.py` (preserves comparability with v3)
- [ ] GPU free: `nvidia-smi` shows zero existing python processes
- [ ] `TRANSFORMERS_OFFLINE=1` set at shell

### 2.2 Hyperparameters (recommended starting point)

Recorded in `docs/changes/2026-05-05-v3-training-complete.md`. Repeated
here for traceability:

```bash
poetry run python ml/scripts/train.py \
    --run-name multilabel-v4-focal-r16 \
    --experiment-name sentinel-retrain-v4 \
    --epochs 60 \
    --batch-size 16 \
    --lr 3e-4 \
    --loss-fn focal --focal-gamma 2.0 --focal-alpha 0.25 \
    --lora-r 16 --lora-alpha 32 \
    --early-stop-patience 8 \
    --num-workers 2
```

Class-imbalance handling for DenialOfService (137 train support vs
IntegerUO 5,343) is the second knob to tune; see §2.4.

> **FocalLoss is experimental in our setting, not a proven default.**
> The implementation was written for the original binary classifier;
> the multi-label wrapper in `trainer.py` (FP32 cast applied per
> Audit #13) has never been validated end-to-end against BCE in our
> 10-class setup. v4 is the first real comparison. Concretely:
>
> - The `--focal-alpha 0.25` value is inherited from binary defaults
>   and may not be appropriate when each of 10 classes has its own
>   prevalence. Per-class α (a length-10 vector) is a future option.
> - If `loss_fn=focal` underperforms `loss_fn=bce` with `pos_weight`,
>   that's a finding — log it and use BCE.
> - The autoresearch search space (see §3) puts `loss_fn ∈ {bce, focal}`
>   exactly so this gets tested rather than assumed.

### 2.3 Per-class floor watch

After training, run `tune_threshold.py` and compare each class to the v3
floor. Rollback rule: if any class breaches its floor, abandon the
checkpoint regardless of macro F1. Floors:

| Class | v4 Floor (F1) |
|---|---|
| CallToUnknown | 0.3436 |
| DenialOfService | 0.3500 |
| ExternalBug | 0.3845 |
| GasException | 0.5001 |
| IntegerUO | 0.7714 |
| MishandledException | 0.4416 |
| Reentrancy | 0.4862 |
| Timestamp | 0.4289 |
| TransactionOrderDependence | 0.4270 |
| UnusedReturn | 0.4360 |

### 2.4 If v4 misses gate

In priority order:
1. Add weighted sampler for DoS/CallToUnknown (highest underrepresentation).
   Implementation: extend `dual_path_dataset.py` with a sampler builder
   keyed off `multilabel_index.csv` row sums.
2. Increase LoRA `r` to 32 (memory permitting; current 8 GB VRAM tight).
3. Try `--loss-fn bce_pos_weight` with computed per-class `pos_weight`
   tensor (already supported by `trainer.py`).
4. Only then: revisit data — re-extract graphs with a different
   `multi_contract_policy` (out of scope unless §6 is approved first).

## 3. Plan B — Autoresearch Harness (parallel, unblocks v4+)

Goal: turn v4 from a hand-launched run into a programmatic sweep.

### 3.1 New files

```
ml/scripts/auto_experiment.py        NEW — thin CLI that:
  • parses --focal-gamma --lora-r --weighted-sampler etc.
  • invokes the same TrainConfig path as train.py
  • after training, runs tune_threshold.py and reads tuned F1
  • prints SENTINEL_SCORE=<float> as the LAST stdout line (autoresearch contract)
  • exits non-zero if pre-flight checks fail

ml/autoresearch/program.md           NEW — declarative search spec:
  metric:          tuned F1-macro (sentinel-retrain-v4)
  constraints:     no per-class F1 below v3 floor
  knobs:           focal_gamma, lora_r, weighted_sampler, lr, batch_size
  ranges:          gamma ∈ [1.0, 3.0], r ∈ {8,16,32}, lr ∈ [1e-4, 5e-4]
  budget:          12 runs total, sequential
  rollback:        any run with score < 0.5069 is discarded
```

### 3.2 Validation

A 1-epoch dry-run must:
- emit `SENTINEL_SCORE=<float>` on the last stdout line
- create exactly one MLflow run under `sentinel-retrain-v4`
- not modify any file in `ml/checkpoints/` if `--dry-run` is passed

### 3.3 Future direction — architectural search (post-v4)

After v4 lands and the v4-sprint freeze is lifted, extend
`auto_experiment.py` so the agent can vary not only hyperparameters but
also *which model file is used*. Rough sketch:

```bash
poetry run python ml/scripts/auto_experiment.py \
    --regime smoke \
    --model-file ml/src/models/sentinel_model_gin.py  \
    --fusion-output-dim 96 \
    --gamma 2.0 ...
```

This unlocks the Architecture Playground (§5) without rewriting the
harness. The skill file (`program.md`) gets a corresponding new section
describing the model-file search space (e.g. one variant per GNN
backbone, one per fusion design, one per encoder). See the autoresearch
plan §13 for the matching out-of-scope list and §14 for the future
direction.

This is **not** in scope for the v4 sprint — it's recorded here so the
harness is built with the extension in mind from day one (e.g.
`auto_experiment.py` should accept arbitrary keyword overrides and
forward them to the constructed `TrainConfig` rather than hard-coding
each knob).

### 4.bis Data-extraction tooling (`ml/data_extraction/`)

The dataset built by `ast_extractor.py` + `tokenizer.py` will be regenerated
only if a feature-schema change forces it (P0-D bumps `feature_schema_version`).
Two improvements worth tracking but not blocking on:

- **Reproducibility manifest** — at end of `ast_extractor.py` run, write
  `ml/data/graphs/_manifest.json` containing: solc versions used, feature
  schema version, total files written, run timestamp, source parquet hash.
  Currently this information is implicit in DVC pointers only.

  read `_manifest.json` from graphs dir and assert it matches the tokenizer's

  paired with new graphs (silent training accuracy regression).

These belong to a *future* re-extraction (next time the schema bumps), not
to the v4 retrain — v4 reuses v3's data unchanged.

## 6. Plan E — Architecture Playground (post-v4)

This is the deliberate exploration phase. **It opens only after the v4
sprint closes** (winner promoted or sprint formally abandoned).

### 6.1 Spirit

The v4 freeze keeps things comparable to 0.5069. The playground exists
for the opposite reason: **break things on purpose to learn what
actually matters**. No checkpoint from this phase is deployed to the
audit chain unless it decisively beats the production model on full
data and per-class floors.

### 6.2 Why this is cheap to do (downstream cost is already due)

The argument that "changing the architecture forces a full downstream
rebuild" is much weaker than it looks for SENTINEL today, because the
rebuild is *already* required:

| Component | Built for | Current status |
|---|---|---|
| `zkml/src/distillation/proxy_model.py` | Already updated to multi-label (`CIRCUIT_VERSION="v2.0"`, 128→64→32→10) | Source-complete, **never run** |
| EZKL `proving_key.pk` / `verification_key.vk` | n/a | Do not exist on disk |
| `contracts/src/ZKMLVerifier.sol` | n/a | Will be auto-generated by EZKL when pipeline runs |
| `contracts/src/AuditRegistry.sol` | Multi-label-aware (`scoreFieldElement` + `publicSignals[64]`) | Source-complete, **never deployed** |

So the "cascade" cost of bumping `fusion_output_dim` (or any other
locked dimension) post-v4 is essentially the same cost as running the
ZKML pipeline for the first time — which we owe anyway. The playground
makes that cost visible, not extra.

### 6.3 Knobs to explore (not exhaustive)

The list below is a **starting menu, not a constraint**. New ideas are
welcome during the phase — the whole point is open-ended exploration.

| Axis | Concrete options to try | What it teaches |
|---|---|---|
| **GNN backbone** | GAT (current), GCN, GIN, GraphSAGE, GAT-v2, EdgeConv | Inductive bias for control/data flow; expressiveness vs. trainability |
| **Fusion design** | concat-MLP (current `FusionLayer`), bilinear, cross-attention (current), gated, no-fusion (graph-only / text-only ablations) | Which modality carries which class; whether fusion adds anything for cheap classes |
| **Encoder** | CodeBERT (current), GraphCodeBERT, CodeT5-small, distilled CodeBERT, no-LoRA full-fine-tune (if VRAM allows) | Tokeniser quality, AST-aware pretraining, parameter efficiency |
| **`fusion_output_dim`** | 64, 96, 128 (current), 192 | Capacity vs. ZKML circuit size — measure proof-time vs. F1 trade-off directly |
| **Classifier head** | linear (current), 2-layer MLP, multi-task with auxiliary class-correlation loss | When extra capacity at the head is cheaper than bigger trunk |
| **Loss** | BCE (+pos_weight), focal (γ, α — scalar or per-class), asymmetric loss, multi-label margin | FocalLoss is unverified in our setting — playground is where we settle the BCE vs focal vs asymmetric question |
| **Edge-attr handling** | embed (current), one-hot, learned per-type weights, no-edge-attr ablation | Whether 5-type edge information actually helps |
| **Sampler** | uniform, weighted (DoS-only, all-rare), curriculum (easy→hard), MixUp on graph features | Class-imbalance treatment families |
| **Multi-contract** | `"first"` (current), `"all"` with max-agg, `"all"` with mean-agg | Cross-cuts with Move 9 — same exploration |
| **Anything else** | any idea that surfaces during the phase | The list is a seed, not a fence |

### 6.4 Operating rules (light-weight)

The playground is exploratory but not chaotic.

1. **One branch per experiment**: `playground/<topic>-<idx>` (e.g.
   `playground/gin-fusion96-002`). Cheaper to throw away.
2. **One model file per experiment**: copy `sentinel_model.py` to a
   variant name (e.g. `sentinel_model_gin.py`) and edit there. Don't
   mutate the v4-frozen file.
3. **Subset training is the default**: 10–20 % stratified subsample,
   1–3 epochs. Goal is *signal direction*, not gate-passing.
4. **Same eval split**: keep `val_indices.npy` constant so numbers
   compare across experiments. (The split is locked because the *test
   harness* is fixed; the *model* is not.)
5. **Comparable metrics, always**: tuned F1-macro, per-class F1, peak
   VRAM, wall-clock, parameter count. Logged to MLflow under a
   dedicated experiment `sentinel-playground-<phase>`.
6. **Notes**: short markdown lab notebook per experiment in
   `ml/playground/notes/<branch>.md` — what you tried, what surprised
   you, what you'd test next. The point of the phase is the notebook
   as much as the numbers.
7. **No production deploy from playground**: a playground winner can
   only graduate via a fresh `sentinel-retrain-v5` run on full data
   with the v4 floors honoured.

### 6.5 ZKML interaction

Many playground knobs (anything that changes `fusion_output_dim`,
parameter count, or graph schema) require the proxy model to be
retrained and the EZKL pipeline to be re-run if we ever want to deploy
the experiment. **Do not run the EZKL pipeline from the playground.**
That happens once, against the chosen winner, in the M2 plan. The
playground's job is to surface trade-offs (e.g. "GIN-128 is +0.01 F1
but +40 % proxy parameters → +X% proof time") so the M2 decision is
informed.

### 6.6 Output of this phase

- A short report (`docs/changes/<date>-architecture-playground-summary.md`)
  with the comparison table, observations, and a recommendation:
  - either "production stays on v4 / v5", or
  - "promote architecture X, plan v5 retrain on full data, plan ZKML
    rebuild against new fusion_output_dim".
- A list of "things I learned about hybrid ML+ZK+contract systems" — the
  whole reason this phase exists.

## 7. Acceptance Criteria

### v4 sprint (must close before the playground opens)

- v4 checkpoint saved as `ml/checkpoints/multilabel-v4-<runname>_best.pt`
- `<...>_thresholds.json` written by `tune_threshold.py`
- MLflow run shows tuned F1 > 0.5069 and all per-class floors held
- `promote_model.py --stage Staging --val-f1-macro <v4>` succeeds
- `docs/STATUS.md` "Active Checkpoint" block updated; new dated changelog
  written summarising the run
- A short note in the changelog states which of `bce` / `focal` actually
  won, so the FocalLoss-as-experiment hypothesis has a logged outcome.

If the v4 sprint exhausts its budget without a winner (3 below-gate
confirms in autoresearch), close it formally with a
`<date>-v4-sprint-closed.md` changelog. The playground may still open;
it just opens against v3 as the comparison baseline.

### Architecture Playground (open-ended; closes only when you decide)

- At least 3 experiments per axis tried (GNN backbone, fusion design,
  encoder, fusion_output_dim) — not for completeness, for variety
- Each experiment leaves a `ml/playground/notes/<branch>.md` lab note
- Comparison summary written
- A clear recommendation made (stay on v4 / v5 / promote candidate X)
