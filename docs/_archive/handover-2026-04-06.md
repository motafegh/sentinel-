## 1. What SENTINEL Is

SENTINEL is a decentralised smart contract security oracle. Its purpose: automatically analyse any Solidity smart contract for vulnerabilities, produce a cryptographically verifiable score, and register that audit permanently on-chain.

The end-to-end flow:

```
Developer uploads .sol contract
          ↓
  Dual-path ML model (GNN + CodeBERT)
  → vulnerability score ∈ [0, 1]
          ↓
  Distilled proxy model (2,625 params)
  + EZKL generates ZK proof of inference
  → proof.json + public score
          ↓
  AuditRegistry.sol (Sepolia testnet)
  → submitAudit(contractAddress, score, zkProof, publicSignals)
  → on-chain record: score + proof hash + timestamp
```

Anyone can verify that:
1. The score came from the real SENTINEL model (ZK proof)
2. The audit happened at a specific time (blockchain timestamp)
3. The score was not tampered with (proof hash)

This is the core value proposition: **trustless, verifiable AI-driven security audits.**

## 3. Module 1: Deep Learning Core

### 3.1 Why This Architecture

Two ways to look at a smart contract:
1. **Structural** — which functions call which, which state variables are read/written, the control flow graph. This is graph-shaped data → best handled by a Graph Neural Network.
2. **Semantic** — what the code actually says, variable names, logic patterns, comments. This is text → best handled by a pre-trained code language model.

Using both paths together gives the model more signal than either alone. A contract with a complex call graph but innocuous code is different from one with simple structure but dangerous patterns in the logic.

### 3.2 Model Architecture (LOCKED — do not change)

```
Input: Solidity .sol file
         │
         ├──────────────────────────┬──────────────────────────
         │                          │
    GNN PATH                   TRANSFORMER PATH
    ─────────────────          ──────────────────────────
    Slither runs on .sol        CodeBERT tokenizer
    Extracts:                   microsoft/codebert-base
    - functions                 Max 512 tokens
    - state variables           (contracts >512 are truncated)
    - modifiers, events         FROZEN — weights never updated
    - call edges
    - read/write edges
         │                          │
    8-dim float32 node         [1, 512] token IDs
    features per node          [1, 512] attention mask
         │                          │
    GNNEncoder                 TransformerEncoder
    3× GAT layers              forward pass
    global_mean_pool           CLS token ([B, 768])
    output: [B, 64]            (first token = contract summary)
         │                          │
         └──────────┬───────────────┘
                    │
               FusionLayer
               concat([B,64], [B,768]) → [B, 832]
               Linear(832 → 256) → ReLU → Dropout(0.3)
               Linear(256 → 64) → ReLU
               output: [B, 64]
                    │
               Classifier
               Linear(64 → 1) → Sigmoid → squeeze(1)
               output: [B,] float in [0, 1]
```

**Total parameters:** 239,041 trainable + 124,645,632 frozen (CodeBERT)

**Output:** A single float per contract. Values closer to 1.0 = more likely vulnerable. Threshold 0.50 → binary label.

### 3.3 Node Feature Vector (8 dimensions, float32)

This is defined in `ml/scripts/ast_extractor_v4_production.py` and replicated exactly in `ml/src/inference/preprocess.py:_extract_graph()`. These MUST stay in sync — any drift causes inference to not match training.

| Index | Feature | How encoded |
|---|---|---|
| 0 | `type_id` | 0=STATE_VAR, 1=FUNCTION, 2=MODIFIER, 3=EVENT, 4=FALLBACK, 5=RECEIVE, 6=CONSTRUCTOR, 7=CONTRACT |
| 1 | `visibility` | 0=public/external, 1=internal, 2=private |
| 2 | `pure` | 1.0 if pure function, else 0.0 |
| 3 | `view` | 1.0 if view function, else 0.0 |
| 4 | `payable` | 1.0 if payable, else 0.0 |
| 5 | `reentrant` | Slither's `is_reentrant` flag |
| 6 | `complexity` | `float(len(func.nodes))` — CFG node count |
| 7 | `loc` | `float(len(source_mapping.lines))` — lines of code |

### 3.4 FocalLoss — Why and How

The dataset is imbalanced: 64.3% vulnerable, 35.7% safe. Standard BCE loss would let the model learn "just predict vulnerable always" and still get 64% accuracy.

**Focal Loss** adds a modulating factor `(1 - pt)^gamma` that down-weights well-classified examples and forces the model to focus on the hard cases.

```python
FL(pt) = -alpha * (1 - pt)^gamma * log(pt)
```

Parameters used: `gamma=2.0`, `alpha=0.25`.

**Important:** `alpha=0.25` is correct here. Alpha is applied to the **positive class** (vulnerable). Since vulnerable is the majority (64.3%), we down-weight it with alpha=0.25. The safe (minority) class gets weight `1-alpha=0.75`. This was questioned and then confirmed during development — do not change it.

### 3.5 Offline Data Pipeline (already complete — do not re-run)

The 68,555 training samples were built by a 5-step offline pipeline:

**Step 1: Source** — `contracts_metadata.parquet` lists all contract paths with their detected Solidity version and extraction success flag.

**Step 2: Graph extraction** — `ml/scripts/ast_extractor_v4_production.py`
- Runs Slither on every `.sol` file (11 parallel workers, version-grouped)
- Builds PyG `Data(x=[N,8], edge_index=[2,E], edge_attr=[E,1], y, contract_hash, contract_path)`
- Saves as `ml/data/graphs/<md5_hash>.pt`
- 68,556 files produced

**Step 3: Tokenisation** — `ml/scripts/tokenizer_v1_production.py`
- Tokenises with CodeBERT (`microsoft/codebert-base`, max 512 tokens, padding to max_length)
- Saves `input_ids [512]` + `attention_mask [512]` as `ml/data/tokens/<md5_hash>.pt`
- 68,570 files (13 extra with no matching graph — silently dropped by DualPathDataset)

**Step 4: Label index** — `ml/scripts/create_label_index.py`
- Scans all graph `.pt` files, reads `.y` attribute
- Writes `ml/data/processed/label_index.csv` (hash → label) — lightweight 3 MB index

**Step 5: Splits** — `ml/scripts/create_splits.py`
- Stratified 70/15/15 split (seed=42) from label_index.csv
- Saves `ml/data/splits/{train,val,test}_indices.npy`
- These are position indices into the sorted paired-hash list, NOT boolean masks

**Pairing mechanism:** Both graphs and tokens use `MD5(absolute_file_path)` as their filename stem. `DualPathDataset` sorts all hashes that appear in BOTH directories — that intersection is the dataset.

### 3.6 DualPathDataset

```python
# Init: find paired hashes
graph_hashes = {f.stem for f in graphs_dir.glob("*.pt")}
token_hashes = {f.stem for f in tokens_dir.glob("*.pt")}
paired = sorted(graph_hashes & token_hashes)       # intersection, sorted for determinism
# Apply split indices if provided
self.pairs = [paired[i] for i in indices]

# __getitem__: load lazily
graph = torch.load(graphs_dir / f"{hash}.pt", weights_only=False)   # PyG Data — complex object
token = torch.load(tokens_dir / f"{hash}.pt", weights_only=True)    # safe dict

label = graph.y.long()   # 0 or 1
# label normalisation: scalar from [1] tensor
if label.dim() > 0 and label.shape[0] == 1:
    label = label.squeeze(0)

return graph, {"input_ids": token["input_ids"], "attention_mask": token["attention_mask"]}, label
```

**Collate function (`dual_path_collate_fn`):**
```python
graphs_batch = Batch.from_data_list(graphs)          # PyG batch object
input_ids    = torch.stack([t["input_ids"] for t in tokens])     # [B, 512]
attn_mask    = torch.stack([t["attention_mask"] for t in tokens]) # [B, 512]
labels       = torch.stack(labels).squeeze(1)        # [B, 1] → [B,] — MUST be squeeze(1) not squeeze()
```

**Why squeeze(1) not squeeze():** If batch size happens to be 1 (can happen with last batch), `squeeze()` would collapse `[1,1]` → `[]` (scalar). `squeeze(1)` only removes the dimension at index 1, giving `[1,]` — safe for all batch sizes.

### 3.7 Training Loop

**Config (`TrainConfig` dataclass):**
```
checkpoint_path, graphs_dir, tokens_dir, splits_dir, labels_csv,
epochs=30, batch_size=64, lr=1e-3, focal_gamma=2.0, focal_alpha=0.25,
weight_decay=1e-4, num_workers=4, device="cuda" (auto-detected),
run_name="sentinel", mlflow_experiment="sentinel-training",
resume_from=None
```

**Training loop per epoch:**
1. `model.train()` → forward → `loss = FocalLoss(output, labels)` → backward → `optimizer.step()`
2. `model.eval()` → `evaluate()` → val_f1_macro, val_f1_safe, val_f1_vulnerable, val_precision_vulnerable, val_recall_vulnerable
3. If `val_f1_macro > best_f1` → save checkpoint
4. MLflow: log all metrics + params

**Checkpoint format (new, April 2026+):**
```python
{
    "model": model.state_dict(),
    "optimizer": optimizer.state_dict(),
    "epoch": epoch,
    "best_f1": best_f1,
    "config": config.__dict__
}
```
Old checkpoints (before this format) are plain state dicts. `Predictor` handles both formats.

**AdamW:** only trainable params:
```python
optimizer = AdamW(filter(lambda p: p.requires_grad, model.parameters()), lr=config.lr)
```
This skips the frozen CodeBERT parameters — they have `requires_grad=False`.

### 3.9 Inference Pipeline

**`ContractPreprocessor`** (`ml/src/inference/preprocess.py`)
Two entry points:
- `process(file_path)` → reads `.sol` file, runs `_extract_graph()` + `_tokenize()`
- `process_source(source_code_str)` → same but from in-memory string

`_extract_graph()` replicates the exact 8-dim feature vector from the training data extractor:
```python
slither = Slither(str(path), solc_remaps=..., solc_args=...)
# ... build PyG Data with same 8 node features ...
assert data.x.shape[1] == 8   # hard assertion — will crash loudly if drift
```

`_tokenize()` replicates the tokenisation exactly:
```python
encoded = tokenizer(code, max_length=512, padding="max_length", truncation=True, return_tensors="pt")
assert encoded["input_ids"].shape == (1, 512)  # hard assertion
```

**`Predictor`** (`ml/src/inference/predictor.py`)
```python
predictor = Predictor(checkpoint_path, threshold=0.50, device="cuda")

# From file:
result = predictor.predict("path/to/Contract.sol")
# result = {"score": 0.73, "label": "VULNERABLE", "threshold": 0.50}

# From source string:
result = predictor.predict_source(source_code_str)
```

The Predictor:
1. Loads checkpoint (handles both new dict format and old plain state_dict)
2. Calls `ContractPreprocessor.process()`
3. Builds `Batch.from_data_list([graph])` for single-item inference
4. Runs forward pass with `torch.no_grad()` + `model.eval()`
5. Returns score (float) + label (str) + threshold

## 4. Module 2: ZKML Pipeline

### 4.1 The Core Problem

ZK circuits have a parameter limit (~10K params). The full SentinelModel has 239,041 trainable params — 24× too large. The solution: **knowledge distillation** — train a tiny proxy model to mimic the full model's outputs.

### 4.2 ProxyModel

```python
class ProxyModel(nn.Module):
    def __init__(self):
        super().__init__()
        self.net = nn.Sequential(
            nn.Linear(64, 32), nn.ReLU(),
            nn.Linear(32, 16), nn.ReLU(),
            nn.Linear(16, 1),  nn.Sigmoid()
        )
    def forward(self, x):
        return self.net(x).squeeze(1)
```

Input: **64-dim fusion features** extracted from the SentinelModel's FusionLayer output (not raw contract features). The proxy sits downstream of the full model — it operates in the model's learned embedding space.

Parameters: 2,625 (well within ZK circuit limits).

**Training (distillation):** Feed contracts through the full SentinelModel, collect `fusion_output [B, 64]` as input and `score [B,]` as soft target. Train proxy with MSE loss. Result: **99.82% agreement** (epoch 1) — the proxy essentially memorises the full model's decision boundary in this feature space.

### 4.3 EZKL Pipeline (8 Steps)

**Steps 1–5 (one-time setup):** `zkml/src/ezkl/setup_circuit.py`

```
Step 1: gen_settings(proxy.onnx)
  → zkml/ezkl/settings.json
  Configures scale, lookup tables, accuracy vs. speed tradeoff

Step 2: calibrate_settings(proxy.onnx, calibration.json)
  → updates settings.json with witness sizes + scale factors
  Calibration data: 100 real contract feature vectors

Step 3: compile_model(proxy.onnx, settings.json)
  → zkml/ezkl/model.compiled
  Converts ONNX → Halo2 R1CS arithmetic circuit

Step 4: gen_srs(settings.json)
  → zkml/ezkl/srs.params
  Structured Reference String (~4 MB) — public randomness for the proof system

Step 5: setup(model.compiled, srs.params)
  → zkml/ezkl/proving_key.pk    (PRIVATE — prover only)
  → zkml/ezkl/verification_key.vk  (PUBLIC — embedded in Solidity verifier)
```

**Steps 6–8 (per audit):** `zkml/src/ezkl/run_proof.py`

```
Step 6: gen_witness(model.compiled, proof_input.json)
  → zkml/ezkl/witness.json
  Runs the circuit forward with real inputs to generate the witness

Step 7: prove(model.compiled, witness.json, pk)
  → zkml/ezkl/proof.json
  The actual ZK proof — takes ~30–60 seconds, output ~2 KB

Step 8: verify(proof.json, settings.json, vk)
  → bool (True = proof valid)
  Off-chain verification (~seconds)
```

### 4.4 BN254 Field Element Encoding

EZKL represents floating-point values as integers in the BN254 scalar field. The `instances` array in `proof.json` contains the public outputs (risk score) as 32-byte little-endian hex strings.

To decode:
```python
def decode_field_element(hex_str: str, scale: int = 13) -> float:
    raw = int.from_bytes(bytes.fromhex(hex_str), byteorder='little')
    # Check if in upper half of field (negative value)
    FIELD_MOD = 0x30644e72e131a029b85045b68181585d2833e84879b9709143e1f593f0000001
    if raw > FIELD_MOD // 2:
        raw = raw - FIELD_MOD
    return raw / (2 ** scale)
```

Scale=13 means: `integer_value / 8192 = float_value`. Example: `scoreFieldElement=4497` → `4497/8192 ≈ 0.5490`.

## 5. Module 5: Solidity Contracts

### 5.1 Contract Architecture

```
SentinelToken.sol
  ├── ERC20 (OpenZeppelin)
  ├── ERC20Permit (gasless approvals via EIP-2612)
  ├── ERC20Votes (governance-ready)
  └── Ownable

  Supply: 1,000,000,000 SENT (minted to deployer)
  Role: staking token for audit submission eligibility

AuditRegistry.sol
  ├── UUPSUpgradeable (proxy pattern)
  ├── OwnableUpgradeable
  └── PausableUpgradeable

  State:
    mapping(address → AuditResult[]) auditsByContract
    mapping(address → uint256) stakedBalance
    IZKMLVerifier public verifier
    IERC20 public sentinelToken
    uint256 public minStake = 1000 * 10**18  (1000 SENT)

  Key functions:
    stake(amount)        → transferFrom + update stakedBalance
    unstake(amount)      → check balance + transfer back
    submitAudit(contractAddress, score, zkProof, publicSignals)
      requires: stakedBalance[msg.sender] ≥ minStake
      requires: verifier.verifyProof(zkProof, publicSignals) == true
      stores:   AuditResult{score, proofHash, timestamp, agent, verified: true}
      emits:    AuditSubmitted(contractAddress, proofHash, agent, score)
    getLatestAudit(addr)   → AuditResult
    getAuditHistory(addr)  → AuditResult[]

IZKMLVerifier.sol
  Interface: verifyProof(bytes calldata proof, uint256[] calldata publicSignals) → bool
  Implemented by the EZKL-generated verifier contract
```

### 5.2 UUPS Pattern — Why

UUPS (Universal Upgradeable Proxy Standard) keeps the upgrade logic in the implementation contract, not the proxy. This is more gas-efficient than Transparent Proxy (no admin check on every call).

Key pattern:
```solidity
// In implementation constructor — prevents direct initialisation attacks
constructor() { _disableInitializers(); }

// Called once via proxy — sets up state
function initialize(address tokenAddr, address verifierAddr) external initializer {
    __Ownable_init(msg.sender);
    __Pausable_init();
    __UUPSUpgradeable_init();
    sentinelToken = IERC20(tokenAddr);
    verifier = IZKMLVerifier(verifierAddr);
}

// Only owner can upgrade — stored in implementation, called via proxy
function _authorizeUpgrade(address) internal override onlyOwner {}
```

**Deployment:** always through `ERC1967Proxy(implementation, initData)` — never call the implementation directly.

## 10. Architecture Decisions Made (ADRs)

| # | Decision | Chosen | Rejected | Reason |
|---|---|---|---|---|
| 001 | ML output type | Binary (vuln/safe) | 13-class multi-label | Simpler, working system. Multi-class adds dataset labelling complexity. |
| 002 | Node features | 8-dim semantic features | 79-dim one-hot | Richer features with fewer dimensions. 79-dim was one-hot categorical. |
| 003 | Fusion | concat → MLP | Gated Multimodal Unit | Simpler, same results. GMU would require more tuning. |
| 004 | UUPS proxy | UUPS | Transparent proxy | Gas efficient; upgrade logic in implementation, not proxy. |
| 005 | ZK library | EZKL | Custom circuits | Production library, active community, Python bindings, proven. |
| 006 | ZK proxy size | 2,625 params | Up to ~10K params | Smallest proxy that achieves ≥99% agreement — minimises circuit size. |
| 007 | Focal alpha | 0.25 (vulnerable=majority) | 0.75 | Alpha down-weights majority class. Vulnerable=64.3% → use 0.25. |
| 008 | Agent LLM | Ollama (local) | GPT-4 / Claude API | Free, no API cost during development. Swap for demo if quality insufficient. |

## 12. Interview Talking Points (fill in as you build)

**"Walk me through your ML architecture"**
"Dual-path model: a GNN captures the structural control flow graph of the contract — function calls, state variable reads/writes — encoded as 8-dimensional node features through 3 GAT layers. CodeBERT processes the raw source code and extracts a CLS-token semantic embedding. Both paths are fused via a 3-layer MLP to a 64-dimensional representation, then classified with sigmoid. The full model has 124M frozen parameters (CodeBERT) + 239K trainable. Val F1-macro hit 0.6686 on 10,283 contracts."

**"How did you make the AI audit verifiable?"**
"Knowledge distillation to a 2,625-parameter proxy that agrees with the full model 99.82% of the time. EZKL compiles the proxy to a Halo2 ZK circuit, generates a proof per audit (~30 seconds, ~2 KB), and we verify on-chain in a Solidity verifier contract auto-generated by EZKL. Anyone can verify the score without seeing the model weights. Last proof submitted on Sepolia at block 10595257."

**"What Solidity patterns did you use?"**
"UUPS upgradeable proxy via OpenZeppelin — chosen over transparent proxy for gas efficiency; upgrade logic lives in the implementation contract. Staking system: agents must stake 1000 SENT tokens to submit audits. The ZK verifier is referenced via IZKMLVerifier interface — swappable if EZKL updates the verifier format. Foundry tests: 20 tests including fuzz over score values and invariant tests that enforce no audit is ever stored without a valid ZK proof."
