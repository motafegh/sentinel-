## IMP-D: Data Quality Improvements

### IMP-D1 — G-04: return_ignored Temporal Ordering

**Audit finding:** G-04 (CRITICAL)
**File:** `ml/src/preprocessing/graph_extractor.py:218–267`
**Affects:** UnusedReturn labels, MishandledException labels

#### Problem

`_compute_return_ignored()` collects all variable names ever read anywhere in the function
(`all_read_names` set) and checks if the call's lvalue name is in that set. This has a
false-negative path: if a `TemporaryVariable` auto-generated by Slither (e.g., `"ret"`, `"tmp"`,
`"tmp_0"`) happens to share a name with an unrelated `LocalVariable` that IS read later in the
function, the function incorrectly concludes the call return was captured.

Result: contracts where the return IS discarded get `return_ignored=0.0` (safe), producing mislabeled
training samples for UnusedReturn and MishandledException.

#### Fix

Replace the global read-name membership check with a sequential scan that verifies the read occurs
**after** the call in Slither's node/IR order.

Slither returns `func.nodes` in CFG topological order — the order in which nodes execute. Walking
nodes and their IR operations sequentially allows tracking: "has the lvalue been read in any IR
operation AFTER this call?"

**Implementation in `graph_extractor.py:_compute_return_ignored()`:**

```python
def _compute_return_ignored(func: Any) -> float:
    try:
        from slither.slithir.operations import LowLevelCall, HighLevelCall, Send

        nodes = getattr(func, "nodes", None) or []
        all_ops_ordered = [
            (node, op)
            for node in nodes
            for op in (getattr(node, "irs", None) or [])
        ]

        for call_idx, (_, op) in enumerate(all_ops_ordered):
            if not isinstance(op, (LowLevelCall, HighLevelCall, Send)):
                continue
            lval = op.lvalue
            if lval is None:
                return 1.0
            lval_name = getattr(lval, "name", None)
            if lval_name is None:
                return 1.0
            # Check if lval_name is read in any IR AFTER this call
            used_after = False
            for _, later_op in all_ops_ordered[call_idx + 1:]:
                for rv in (getattr(later_op, "read", None) or []):
                    if getattr(rv, "name", None) == lval_name:
                        used_after = True
                        break
                if used_after:
                    break
            if not used_after:
                return 1.0  # lval never used after the call → return discarded

        return 0.0
    except AttributeError:
        logger.warning(
            "return_ignored: Slither IR unavailable for %s — using sentinel -1.0",
            getattr(func, "canonical_name", "?"),
        )
        return -1.0
```

#### Gate Criteria

Write unit test at `ml/tests/test_graph_extractor.py` (or new test file) with three synthetic cases:

| Test | Contract pattern | Expected `return_ignored` |
|------|-----------------|--------------------------|
| CAPTURED | `uint ret = externalContract.call(); use(ret);` | `0.0` |
| DISCARDED | `externalContract.call(); // return ignored` | `1.0` |
| NAME_COLLISION | Local variable named `"ret"` is read, but call return also stored in `"ret"` and never used afterward | `1.0` (collision case must NOT give 0.0) |

All three tests must pass. Additionally: run dry-extraction on 50 known-UnusedReturn contracts from
the dataset and verify that `return_ignored=1.0` appears on at least 70% of them (baseline: check
current rate before fix).

#### Expected Impact

- UnusedReturn F1: targeted by H-GCB-2 (current PLAN-3A baseline 0.195). Removing false negatives
  means cleaner training signal for the weakest class.
- MishandledException F1: currently 0.289. Cleaner labels should reduce confusion at the boundary
  between "exception handled" and "exception ignored."
- Requires full re-extraction on all 41K contracts — same protocol as Phase 2.

#### Ordering Constraints

- Must re-extract all graphs after applying this fix (same as any extractor change).
- Triggers: update `FEATURE_SCHEMA_VERSION`, rebuild cache, re-run label_cleaner.py.
- Run BEFORE Phase DATA-1 training run so the improved labels are used.

### IMP-G1 — N-01: Phase 2 Layer-Specific Edge Subsets

**Audit finding:** N-01 (HIGH)
**File:** `ml/src/models/gnn_encoder.py:489–497`
**JK evidence:** Phase 2 IQR range 0.064 (v8-AB), weight decays from 0.365 to 0.048 by ep67 (PLAN-3A)

#### Problem

All three Phase 2 conv layers (`conv3`, `conv3b`, `conv3c`) process the identical `cfg_ei`/`cfg_ea`
edge set. Three passes over the same edges produce highly correlated updates — the model learns each
successive pass adds marginal information. JK analysis: Phase 2 weight's per-node std collapses from
0.152 (ep3, genuine routing) to effectively constant by convergence. The model implicitly treats all
three Phase 2 layers as one averaged pass.

This directly explains why adding CALL_ENTRY/RETURN_TO/DEF_USE in v8-AB did not improve over v7 —
new edge types get diluted across three layers all processing the same combined set. The ICFG signal
competes with CONTROL_FLOW signal in every layer rather than building on top of it.

#### Fix

Assign each Phase 2 layer a distinct, focused edge subset so each layer builds on the previous:

| Layer | Edge subset | What it learns |
|-------|-------------|----------------|
| `conv3` (Layer 3) | `CONTROL_FLOW(6)` only | Intra-function execution order (CEI within a function) |
| `conv3b` (Layer 4) | `CALL_ENTRY(8)` + `RETURN_TO(9)` | Cross-function call structure (who enters whom, who returns where) |
| `conv3c` (Layer 5) | `CONTROL_FLOW(6)` + `CALL_ENTRY(8)` + `RETURN_TO(9)` | Integration: nodes with both local and cross-function context |

Layer 5 as an integration layer allows the model to learn "a node that has absorbed both local CFG
ordering AND cross-function call structure" — a richer representation than any single-type pass.

If `--phase2-edge-types` is used for ablations (PLAN-3B style), each layer receives the intersection
of its defined subset and the configured types.

**Implementation in `gnn_encoder.py` forward pass:**

```python
# In forward(), after building cfg_ei / cfg_ea (line ~438):

# IMP-G1: layer-specific edge subsets for Phase 2
# Subset 1: CONTROL_FLOW only
cf_only_mask  = (edge_attr == _CONTROL_FLOW)
cf_only_ei    = edge_index[:, cf_only_mask]
cf_only_ea    = e[cf_only_mask] if e is not None else None

# Subset 2: CALL_ENTRY + RETURN_TO only
icfg_only_mask = (edge_attr == _CALL_ENTRY) | (edge_attr == _RETURN_TO)
icfg_only_ei  = edge_index[:, icfg_only_mask]
icfg_only_ea  = e[icfg_only_mask] if e is not None else None

# Phase 2 with distinct layer subsets:
x2 = self.conv3(x, cf_only_ei,   cf_only_ea)    # Layer 3: CF only
x2 = self.relu(x2)
x  = x + self.dropout(x2)

x2 = self.conv3b(x, icfg_only_ei, icfg_only_ea) # Layer 4: ICFG only
x2 = self.relu(x2)
x  = x + self.dropout(x2)

x2 = self.conv3c(x, cfg_ei, cfg_ea)             # Layer 5: joint integration
x2 = self.relu(x2)
x  = x + self.dropout(x2)
x  = self.phase_norm[1](x)
```

Note: when `phase2_edge_types` ablation config excludes certain types, the corresponding subset
`edge_index` will be empty — `GATConv` on an empty edge set returns zeros, which the residual
handles correctly. No special casing needed.

#### Gate Criteria

| Check | Expected | Fail action |
|-------|----------|-------------|
| Smoke ep1: JK Phase 2 weight per-node std | > 0.08 (vs 0.064 IQR in current design) | Subsets still too similar — check mask logic |
| Smoke ep2: Phase 2 weight NOT decaying immediately | > 0.10 still at ep2 | Immediate decay = layers not differentiated |
| ep3 JK Phase 2 std | > 0.10 (genuine per-node routing) | If ≤ 0.05: architectural separation insufficient |
| ep20: Phase 2 JK weight | > 0.12 and stable | If < 0.08: same collapse as before |

### IMP-G2 — N-03: Phase 1 Input Projection Skip

**Audit finding:** N-03 (HIGH)
**File:** `ml/src/models/gnn_encoder.py:467–474` (Phase 1 Layer 1)
**GNN share evidence:** Gradient share collapses from ~70% at ep1 to ~7% by ep8-43

#### Problem

Phase 1 Layer 1 transforms 11-dim raw node features → 256-dim hidden. There is no residual
connection because the dimensions differ (11 ≠ 256). The comment reads "No residual - dims differ."

If first-layer GAT attention weights are poorly initialized (or the optimizer pushes them toward
uniform attention), raw feature information can be effectively lost before reaching Phase 2. Once
lost, it cannot be recovered — all later phases operate on a degraded representation.

The observed GNN gradient share collapse is consistent with this: if Phase 1 representations are
noisy, JK aggregation learns to down-weight Phase 1, which reduces its gradient, which further
degrades its representation — a feedback collapse.

#### Fix

Add a learned input projection as a skip connection: `self.input_proj = nn.Linear(NODE_FEATURE_DIM, hidden_dim)`.
Apply it to the original raw features and add to Phase 1 Layer 1 output before the residual chain:

```python
# In GNNEncoder.__init__():
self.input_proj = nn.Linear(NODE_FEATURE_DIM, hidden_dim, bias=False)
# bias=False: the GAT conv already has bias; adding bias to skip would double-count it

# In forward(), after Phase 1 Layer 1:
# x_init is the raw [N, 11] node feature tensor (before any conv)
x_skip = self.input_proj(x_init)           # [N, hidden_dim]

# Phase 1, Layer 1 (existing):
x = self.conv1(x_init, struct_ei, struct_ea)  # [N, hidden_dim]
x = self.relu(x + x_skip)                 # ADD projected skip BEFORE relu

# Layers 2+ continue with normal residuals as before
```

Parameters: `11 × 256 = 2,816` (trivial).

Note: `x_init` must be saved before the first conv call. Currently the forward pass overwrites `x`
immediately. Store `x_init = x` at the top of the forward pass before any convolutions.

#### Gate Criteria

| Check | Expected | Fail action |
|-------|----------|-------------|
| GNN gradient share at ep5 | > 25% (vs 7–10% collapse without fix) | Input projection not reaching gradient — check skip connection wiring |
| Phase 1 JK weight at ep5 | > 0.08 (vs 0.050 mean in v8-AB collapse) | Phase 1 still being down-weighted; may need higher LR on input_proj |
| GNN gradient share stable through ep30 | > 15% (not collapsing to < 5%) | Collapse still happening — investigate LayerNorm interaction |
| Phase 1 JK weight at convergence | > 0.07 (vs 0.065 in best PLAN-3A run) | No improvement from skip |

### IMP-G3 — N-04: Phase 3 Bidirectional Context Pass

**Audit finding:** N-04 (HIGH)
**File:** `ml/src/models/gnn_encoder.py:503–513` (Phase 3)
**Impact:** CrossAttentionFusion queries all nodes but CFG nodes carry only Phase 1+2 context

#### Problem

Phase 3 (reverse-CONTAINS) propagates Phase-2-enriched CFG signal **upward** to FUNCTION nodes.
After Phase 3 completes:
- FUNCTION nodes: carry Phase 1 + Phase 2 + Phase 3 context (fully enriched)
- CFG nodes: carry Phase 1 + Phase 2 context only (never received Phase 3 messages)

`CrossAttentionFusion` queries ALL nodes including CFG nodes. CFG nodes form the majority of each
graph (~70% of nodes are CFG-type). Since they lack Phase 3's function-level context, the fusion's
cross-attention between "what code token X means" and "what CFG node Y encodes" is operating on
a representation gap.

The pooling in the GNN eye is over FUNCTION nodes (correct). But the fusion eye queries ALL nodes
for the token→node attention, so it's mixing Phase-3-enriched FUNCTION embeddings with
Phase-2-only CFG embeddings — two different levels of information depth.

#### Fix

After Phase 3 reverse-CONTAINS (upward), add a forward CONTAINS pass (downward) that distributes
the enriched FUNCTION-node context back to CFG children:

```python
# In GNNEncoder.__init__():
# Add one more GATConv for the downward pass (same config as conv4)
self.conv4c = GATConv(
    hidden_dim, hidden_dim,
    heads=1, concat=False,
    dropout=self.dropout_p,
    add_self_loops=False,
    edge_dim=edge_emb_dim if use_edge_attr else None,
)
# (already has phase_norm[2] — use a new norm or reuse — see note below)

# In forward(), after Phase 3 reverse-CONTAINS:
# ... (existing Phase 3 up):
x2 = self.conv4(x, rev_contains_ei, rev_contains_ea)
x2 = self.relu(x2)
x  = x + self.dropout(x2)
x2 = self.conv4b(x, rev_contains_ei, rev_contains_ea)
x2 = self.relu(x2)
x  = x + self.dropout(x2)

# NEW: Phase 3 down — forward CONTAINS (FUNCTION → CFG children)
x2 = self.conv4c(x, fwd_contains_ei, fwd_contains_ea)  # fwd edges already computed above
x2 = self.relu(x2)
x  = x + self.dropout(x2)

x  = self.phase_norm[2](x)  # LayerNorm covers full Phase 3 (up + down)
```

`fwd_contains_ei` / `fwd_contains_ea`: the original CONTAINS edges before reversal. Already
computed in the existing forward pass as the source for `rev_contains_ei`. Save them:
```python
fwd_contains_ei = edge_index[:, contains_mask]
fwd_contains_ea = e[contains_mask] if e is not None else None
rev_contains_ei = fwd_contains_ei.flip(0)   # existing flip
```

This increases the model depth from 7 to 8 GNN layers and from 3 to 4 Phase 3 hops.

Note: `_live` list for JK aggregation should include ALL phase outputs. Currently `_live` has 3
entries (after Phase 1, after Phase 2, after Phase 3). With bidirectional Phase 3, the "after Phase 3"
entry in `_live` reflects the enriched state (both up and down passes) — no change to JK structure.

#### Gate Criteria

| Check | Expected | Fail action |
|-------|----------|-------------|
| CrossAttentionFusion loss | Lower than GNN eye loss from ep5 | Fusion not benefiting from richer CFG embeddings — check fwd_contains wiring |
| CFG-FUNCTION embedding similarity | Higher cosine sim between FUNCTION and its CFG children at convergence | Diagnostic: run `ml/scripts/jk_weight_hist.py` variant that computes node-pair cosine sim |
| Reentrancy + ExternalBug F1 | Higher than without IMP-G3 | CrossAttention still not routing correctly; may need to isolate further |
| No NaN in Phase 3 down pass | 0 NaN | Forward pass on CONTAINS edges with LayerNorm — check for degenerate graphs with no CONTAINS edges |

### IMP-M1 — FUNCTION Node Secondary Sort by external_call_count

**Source:** `graph_schema.py:343` (documents intended behavior, not yet implemented)
**File:** `ml/src/models/sentinel_model.py:select_prefix_nodes()`

#### Problem

`graph_schema.py:343` documents: *"FUNCTION — sorted by feature[10] descending (external_call_count)"*.
The actual `select_prefix_nodes()` implementation sorts eligible nodes by priority type only. Within
the FUNCTION priority group (priority=4), nodes are emitted in graph index order — the order Slither
happened to process them.

For the 4.5% of contracts where K=48 truncation occurs (>48 eligible declaration nodes), the most
vulnerability-relevant FUNCTION nodes (those making external calls) may be excluded in favor of
earlier-indexed but structurally less interesting ones.

#### Fix

**In `sentinel_model.py:select_prefix_nodes()`**, replace the simple `priorities.argsort()` with a
two-key sort: primary = type priority, secondary = `-external_call_count` (descending) for FUNCTION
nodes, 0 for all others.

```python
# After building `eligible_local` list, replace:
# priorities = torch.tensor([_PREFIX_NODE_PRIORITY[g_types[i].item()] for i in eligible_local], ...)
# sort_order = priorities.argsort()

# With:
_EXT_CALL_DIM = 10  # feature[10] = external_call_count (log1p-normalized)
sort_keys = []
for local_idx in eligible_local:
    t = g_types[local_idx].item()
    prio = _PREFIX_NODE_PRIORITY[t]
    # Secondary sort: higher external_call_count = lower sort key (selected first)
    # Only applies within FUNCTION nodes (prio=4); others use 0.0
    ext_calls = -g_embs[local_idx, _EXT_CALL_DIM].item() if t == NODE_TYPES["FUNCTION"] else 0.0
    sort_keys.append((prio, ext_calls, local_idx))  # (primary, secondary, tiebreak)

sort_keys.sort()  # Python stable sort on tuples
selected_indices = [sk[2] for sk in sort_keys[:K]]
selected = torch.tensor(selected_indices, device=device, dtype=torch.long)
```

Remove the old `priorities` tensor and `sort_order` variables.

#### Gate Criteria

Run diagnostic on 5 large contracts (> 50 FUNCTION nodes) from the val set:
1. Verify that among FUNCTION nodes selected, those with higher `feature[10]` appear in the prefix
2. Verify that CONSTRUCTOR/FALLBACK/RECEIVE/MODIFIER nodes still appear before FUNCTION nodes
3. Verify that for contracts with ≤ K eligible nodes, selection is identical to before

### IMP-M2 — prefix_attention_mean Diagnostic Logging

**Source:** `docs/proposal/EXECUTION_PLAN.md` — P1-TRAIN monitoring section
**File:** `ml/src/training/trainer.py`

#### Problem

After warmup ends at epoch 16, there is no mechanism to detect whether GraphCodeBERT is actually
attending to the K=48 prefix positions or learning to ignore them. A prefix ignored by the
transformer provides zero benefit — the K=48 code budget reduction becomes a pure loss.

The only signal available is indirect (F1 trend after ep16 and `gnn_to_bert_proj.weight.norm()`
stability). A zero or near-zero `prefix_attention_mean` for 5+ consecutive post-warmup epochs
would indicate the mechanism is not working, warranting early intervention.

#### Fix

Two-tier approach:

**Tier 1 (simple proxy — implement now, 30 min):** Log `gnn_to_bert_proj.weight.norm()` to MLflow
each epoch from ep≥`gnn_prefix_warmup_epochs`. A norm that is still rapidly changing at ep25+
suggests the projection has not converged; a norm that flatlines immediately may indicate the
optimizer treats it as frozen.

```python
# In trainer.py, inside the per-epoch logging block:
if model.gnn_prefix_k > 0:
    proj_norm = model.gnn_to_bert_proj.weight.norm().item()
    mlflow.log_metric("gnn_to_bert_proj_norm", proj_norm, step=epoch)
```

**Tier 2 (attention extraction — implement before GATE-GCB-4 analysis, 2–3 hrs):** Once per
validation epoch (not per training step), run one forward pass on a single val batch with
`output_attentions=True` and extract the mean attention from code positions → prefix positions.

```python
# In TransformerEncoder.forward(), add optional param:
def forward(self, input_ids, attention_mask, gnn_prefix_nodes=None, output_attentions=False):
    # Pass output_attentions to bert() call
    outputs = self.bert(..., output_attentions=output_attentions)
    if output_attentions:
        # outputs.attentions: tuple of 12 tensors, each [B, heads, L, L]
        # Code positions: [:, :, K:, :K] = attention FROM code tokens TO prefix tokens
        attn_stack = torch.stack(outputs.attentions, dim=0)  # [12, B, heads, L, L]
        K = gnn_prefix_nodes.shape[1]
        prefix_attn = attn_stack[:, :, :, K:, :K].mean().item()  # mean over all dims
        return outputs.last_hidden_state, prefix_attn
    return outputs.last_hidden_state
```

Log `prefix_attention_mean` to MLflow. Warning: adds ~15% overhead per val step when active.
Restrict to val-only with `model.eval()` and `torch.no_grad()`.

#### Gate Criteria

After warmup ends (ep ≥ 16):

| Metric | Healthy | Warning | Action |
|--------|---------|---------|--------|
| `gnn_to_bert_proj_norm` | Changing by 1–5% per epoch for 5+ epochs after warmup | Flatlines immediately (< 0.1% change from ep17–20) | Projection not learning — check LR and weight init |
| `prefix_attention_mean` (Tier 2) | > 0.005 by ep20 (code tokens allocate ≥ 0.5% attention to prefix — note K/L = 48/464 = 10.3% of positions) | < 0.002 for 5 consecutive epochs | Prefix being ignored — reduce K or check attention mask |
| `prefix_attention_mean` trend | Increasing ep16 → ep30 | Flat or decreasing after ep20 | Transformer not integrating prefix signal over time |

### IMP-M3 — Zero-Padded Prefix Attention Mask Fix

**File:** `ml/src/models/sentinel_model.py:select_prefix_nodes()` and `transformer_encoder.py:forward()`

#### Problem

`prefix_mask = torch.ones(B, K, ...)` sets attention mask to 1 for ALL K prefix positions, including
zero-padded positions (graphs with fewer than K eligible nodes). Zero-padded positions have zero
embedding values but are still attended to and attended from.

The transformer cannot distinguish real prefix tokens from zero-padded ones via the embedding
value alone (both could be near-zero after convergence). This wastes attention capacity and may
cause gradient noise through the zero-padded positions.

For the 95.5% of contracts that DO fill all K=48 slots (per PRE-4), this has zero impact.
For the 4.5% with truncation, incorrect masking persists.

#### Fix

**Option A — Return node count from `select_prefix_nodes()` and mask padded positions:**

```python
# In sentinel_model.py:select_prefix_nodes(), add to return:
node_counts = torch.zeros(num_graphs, dtype=torch.long, device=device)
# ... existing code ...
node_counts[g] = proj.shape[0]  # actual selected count per graph
return prefix, node_counts  # [B, K, 768], [B]

# In sentinel_model.py:forward(), after calling select_prefix_nodes:
gnn_prefix, prefix_counts = self.select_prefix_nodes(...)

# In transformer_encoder.py:forward(), accept prefix_counts:
def forward(self, input_ids, attention_mask, gnn_prefix_nodes=None, gnn_prefix_counts=None):
    # Build prefix mask using actual counts instead of torch.ones
    if gnn_prefix_counts is not None:
        prefix_mask = torch.zeros(B, K, dtype=attention_mask.dtype, device=device)
        for b in range(B):
            prefix_mask[b, :gnn_prefix_counts[b]] = 1
    else:
        prefix_mask = torch.ones(B, K, dtype=attention_mask.dtype, device=device)
```

**Option B (simpler) — Detect padded positions by L2 norm of prefix embedding:**

```python
# Zero-padded positions have embedding L2 norm = 0 before projection output is added
# (torch.zeros input to Linear produces bias-only output... but prefix tensor is zeros
# before assignment, so padded slots are truly zeros with no bias)
# Detect: prefix_mask[:, :] = (gnn_prefix_nodes.norm(dim=-1) > 1e-6).float()
```

Option A is preferred (explicit and correct). Option B is a reasonable fallback if signature changes
are too invasive.
