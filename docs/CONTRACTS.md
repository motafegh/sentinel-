# SENTINEL Smart Contracts — Technical Reference

## Overview

SENTINEL deploys two contracts on Ethereum:

| Contract | Type | Address (Sepolia) |
|---|---|---|
| `SentinelToken` | ERC20 + staking | `0x9d6F751AC25CE3b8D16Bff3df9c3C0BC8A463761` |
| `AuditRegistry` | UUPS proxy | `0x14E5eFb6DE4cBb74896B45b4853fd14901E4CfAf` |
| `AuditRegistry` (impl) | Implementation | `0x40A2BC1d6F53bA037b5999e21a3a02163B0BED28` |
| `ZKMLVerifier` | EZKL Halo2 | `0xB7093Be4958dd95438D6f53Ff7DF8659451CbD97` |

## Architecture

```
Agent (off-chain)
    │
    │  1. stake(1000 SENT)
    ▼
SentinelToken.sol
    │
    │  2. submitAudit(contract, score, proof, signals)
    ▼
AuditRegistry.sol ──── Guard 1: stakedBalance(agent) ≥ MIN_STAKE
    │               ──── Guard 2: ZKMLVerifier.verifyProof(proof, signals) → true
    │               ──── Guard 3: signals[64] == scoreFieldElement
    │
    │  3. verifyProof(proof, signals)
    ▼
ZKMLVerifier.sol (EZKL-generated Halo2 verifier)
    │
    └── emits AuditSubmitted(contractAddress, proofHash, agent, scoreFieldElement)
```

## SentinelToken.sol

### Purpose
ERC20 token that functions as economic collateral for audit agents. Agents must stake tokens before submitting audits — this creates skin-in-the-game incentives.

### Key constants
```solidity
uint256 public constant MIN_STAKE = 1000 * 10 ** 18;  // 1,000 SENT
```

### State
```solidity
mapping(address => uint256) private _stakedBalances;
```
Staked tokens are transferred into the contract itself (`address(this)`) but tracked separately from the public `balanceOf`. They are locked until explicitly unstaked — they cannot be transferred while staked.

### Functions

#### `stake(uint256 amount)`
Transfers `amount` tokens from `msg.sender` to the contract. Adds to `_stakedBalances[msg.sender]`. Emits `Staked(agent, amount)`.

#### `unstake(uint256 amount)`
Returns `amount` tokens from the contract to `msg.sender`. Requires `_stakedBalances[msg.sender] >= amount`. Emits `Unstaked(agent, amount)`.

#### `slash(address agent, uint256 amount)` — `onlyOwner`
Burns `amount` from an agent's staked balance. Tokens are removed from circulation permanently via `_burn(address(this), amount)`. This makes slashing economically meaningful — a slashed agent cannot recover the tokens. Emits `Slashed(agent, amount)`.

**Current limitation:** `onlyOwner` — controlled by deployer. Production replacement: multi-sig or DAO governance.

### Why tokens are burned on slash
Transferring slashed tokens to the protocol treasury would give the owner a financial incentive to slash agents. Burning removes that incentive — the owner gains nothing from slashing except protocol integrity.

## AuditRegistry.sol

### Purpose
Stores cryptographically verified vulnerability audit results on-chain. Every stored result is backed by a ZK proof confirming the ML model ran correctly.

### Upgrade pattern: UUPS

```
User → ERC1967Proxy (0x14E5...) → AuditRegistry (0x40A2...)
```

- **Proxy** holds all storage and delegates all calls to the implementation
- **Implementation** contains the logic; its storage is unused
- **Upgrades** replace the implementation address stored in the proxy
- **Access control:** only owner can call `_authorizeUpgrade()`

**Why UUPS not Transparent Proxy?**
In UUPS, upgrade logic lives in the implementation. This saves ~3KB of bytecode in the proxy (no admin slot reads on every call) and makes the upgrade path auditable alongside the business logic.

**`_disableInitializers()` in constructor:**
The implementation contract's constructor calls `_disableInitializers()`. This prevents anyone from calling `initialize()` directly on the implementation — a critical security property because the implementation's owner would have the same privileges as the proxy's owner if someone initialized it. The ERC1967Proxy calls `initialize()` atomically during deployment.

### The Three Guards

`submitAudit()` enforces three independent conditions. All must pass or the transaction reverts with no state change.

#### Guard 1 — Economic stake
```solidity
require(
    sentinelToken.stakedBalance(msg.sender) >= sentinelToken.MIN_STAKE(),
    "AuditRegistry: insufficient stake"
);
```
Prevents spam audits by requiring 1,000 SENT at risk. An agent who submits a fraudulent proof (if they somehow bypass Guard 2) loses their stake to slashing.

#### Guard 2 — ZK proof validity
```solidity
require(
    zkmlVerifier.verifyProof(proof, publicSignals),
    "AuditRegistry: invalid ZK proof"
);
```
Delegates to the EZKL-generated Halo2 verifier. If the proof was:
- generated by a different model version
- tampered with after generation
- generated for different inputs than those claimed in `publicSignals`

...this call returns false and the tx reverts. ~250K gas.

#### Guard 3 — Score consistency
```solidity
require(
    publicSignals[64] == scoreFieldElement,
    "AuditRegistry: score mismatch with proof"
);
```
The `publicSignals` array is what the verifier checks against the proof. `publicSignals[64]` is the model output field element. By requiring the caller to pass the same value as `scoreFieldElement`, we guarantee the on-chain stored score is exactly what the proof attests.

**Why not just read from `publicSignals[64]` instead of taking a `scoreFieldElement` param?**
Solidity `calldata` arrays are passed by the caller. The caller must commit to a specific value in both the `scoreFieldElement` argument and `publicSignals[64]` — if they differ, Guard 3 reverts. This makes the stored score explicit and auditable from the transaction itself, not just inferred from the signals array.

### Score encoding

The `scoreFieldElement` stored on-chain is NOT a percentage or probability directly. It is a BN254 field element:

```
scoreFieldElement = round(model_output_probability × 2^scale)
scale = 13  (from EZKL calibration, settings.json)
2^13  = 8192
```

To recover the probability off-chain:
```python
probability = scoreFieldElement / 8192
# Example: 4497 / 8192 = 0.5490  → 54.9% vulnerability probability
```

### Pausable emergency stop

The contract inherits `PausableUpgradeable`. The owner can call `pause()` to immediately halt all `submitAudit()` calls. Reads (`getLatestAudit`, `getAuditHistory`, `hasAudit`) are NOT paused.

Use cases:
- Critical bug discovered in ZKMLVerifier
- Model key compromise suspected
- Protocol governance transition

### AuditResult struct

```solidity
struct AuditResult {
    uint256 scoreFieldElement;  // model_output * 2^13 — divide by 8192 for probability
    bytes32 proofHash;          // keccak256(proof) — ~32 bytes vs ~29KB for full proof
    uint256 timestamp;          // block.timestamp at submission
    address agent;              // who submitted this audit
    bool verified;              // always true if stored (three guards enforce it)
}
```

`verified` is always `true` for stored results — a result cannot be stored without passing all three guards. It exists to make the intent explicit in the struct and allow future extension (e.g., a disputable audit state).

### Query functions

| Function | Description |
|---|---|
| `hasAudit(address)` | Check before calling getLatestAudit to avoid revert |
| `getLatestAudit(address)` | Most recent audit — reverts if none exist |
| `getAuditHistory(address)` | All audits in chronological order |
| `getAuditCount(address)` | Number of audits for a contract |

## IZKMLVerifier.sol

```solidity
interface IZKMLVerifier {
    function verifyProof(bytes calldata proof, uint256[] calldata instances) external returns (bool);
}
```

**Important details:**
- `nonpayable` not `view` — the Halo2 verifier allocates and writes memory internally, even though it doesn't persist any storage. Solidity requires `nonpayable` for functions that write memory.
- `instances` not `publicSignals` — EZKL's generated ABI uses the term "instances". Both refer to the same thing: the public inputs/outputs of the ZK circuit.
- The interface allows the verifier to be swapped when the ML model is updated (new circuit → new verification key → new `ZKMLVerifier.sol`).

## Deploy.s.sol

The deploy script handles only `SentinelToken` and `AuditRegistry`. `ZKMLVerifier` is deployed separately with `cast` (it's EZKL-generated and requires standalone compilation).

```bash
# Step 1: deploy ZKMLVerifier separately
cast send --create <bytecode> ...

# Step 2: set ZKML_VERIFIER env var, then run:
forge script script/Deploy.s.sol \
  --rpc-url $RPC_URL \
  --private-key $PRIVATE_KEY \
  --broadcast
```

**Proxy initialization:** `ERC1967Proxy` calls `initialize(verifierAddress, tokenAddress)` atomically in its constructor. This is the only call that can ever succeed on the proxy — `_disableInitializers()` in the implementation constructor prevents a second initialization.

## Testing

```bash
cd contracts && forge test --summary
```

Test suite covers:
- SentinelToken: stake, unstake, slash, fuzz tests for balance invariants
- AuditRegistry: all three guards, happy path, pause/unpause, event emission
- Multi-audit accumulation: `getLatestAudit` returns most recent; history preserves order
- Edge cases: getLatestAudit reverts on no audits; `hasAudit` returns false/true correctly

**Pattern — deploying through proxy in tests:**
```solidity
AuditRegistry impl = new AuditRegistry();
bytes memory initData = abi.encodeWithSelector(AuditRegistry.initialize.selector, ...);
ERC1967Proxy proxy = new ERC1967Proxy(address(impl), initData);
AuditRegistry registry = AuditRegistry(address(proxy));
```
This is required because `_disableInitializers()` blocks direct `initialize()` calls on the implementation — tests must go through the proxy, same as production.
