## What SENTINEL Is

A decentralized platform where AI agents autonomously audit smart contracts, generate zero-knowledge proofs of their analysis, coordinate via multi-agent systems, and serve results through blockchain oracles — continuously learning from new exploits.

**GitHub:** https://github.com/motafegh/sentinel

## System Architecture — How All 6 Modules Connect

```
USER
  │ Upload .sol contract
  ▼
MODULE 6: API GATEWAY (FastAPI)
  │ POST /v1/audit → job_id
  │ Celery task queue (Redis)
  ▼
MODULE 4: AGENT ORCHESTRATION (CrewAI)
  ├─ Agent 1: Static Analyzer (Slither/Mythril)
  ├─ Agent 2: ML Intelligence ──────────────── calls MODULE 1
  ├─ Agent 3: RAG Researcher (FAISS)
  ├─ Agent 4: Code Logic Reviewer
  └─ Agent 5: Synthesizer → structured report
  ▼
MODULE 2: ZK PROOF GENERATION (EZKL)
  │ Proxy model (distilled from MODULE 1)
  │ Generates proof π + public risk_score
  ▼
MODULE 5: BLOCKCHAIN (Ethereum Sepolia)
  │ ZKMLVerifier.verify(proof, signals) → true
  │ AuditRegistry.submitAudit(...)
  │ Emit AuditSubmitted event
  ▼
MODULE 3: MLOps LAYER
  │ MLflow: tracks all MODULE 1 experiments
  │ DVC: versions MODULE 1 datasets
  │ Evidently: monitors MODULE 1 model drift
  │ Dagster: orchestrates retraining pipeline
  ▼
USER RECEIVES
  Vulnerability report + risk score (0-100)
  + on-chain verifiable ZK proof link
  + remediation suggestions
```

### Module Dependency Map

```
MODULE 1 (ML Core)
  ← no dependencies, builds first
  → output: trained model + inference API
  → feeds: Module 2 (needs model to distill), Module 4 (Agent 2 calls inference)

MODULE 2 (ZKML)
  ← depends on: Module 1 (trained model required)
  → output: ZK proof pipeline + ZKMLVerifier.sol
  → feeds: Module 5 (verifier contract deployed here)

MODULE 3 (MLOps)
  ← depends on: Module 1 (something to track/version)
  → output: experiment tracking + retraining pipeline
  → runs parallel to: Module 1 training

MODULE 4 (Agents)
  ← depends on: Module 1 (inference API), Module 3 (FAISS index)
  → output: multi-agent audit pipeline
  → feeds: Module 6 (agents called by API)

MODULE 5 (Solidity)
  ← depends on: Module 2 (ZKMLVerifier.sol generated by EZKL)
  → output: deployed contracts on Sepolia
  → feeds: Module 6 (API calls contracts)

MODULE 6 (Integration)
  ← depends on: all modules working
  → output: running system end-to-end
```

## Interview Talking Points

**Important:** Fill in placeholders (`[specific X]`, `[N]%`, etc.) as you build. Specific numbers and real decisions are what make these stories land in senior interviews. A story with "our F1 score was 0.87 on reentrancy" is 10x stronger than one without it.

### "Tell me about a challenging ML architecture decision"
"In SENTINEL, I chose a dual-path architecture combining a Graph Attention Network with frozen
CodeBERT. The GNN path processes the contract's AST/CFG as a graph — function call relationships,
state variable reads and writes. GAT's attention mechanism learns which neighbouring nodes matter
most for vulnerability prediction; for reentrancy detection, for example, state variable writes
inside external calls receive higher attention than unrelated events. The CodeBERT path captures
code semantics — variable names, coding idioms, patterns a developer wrote. Both produce embeddings
that are concatenated and passed through an MLP fusion layer. The training dataset is BCCC-SCsVul-
2024 (68,555 Solidity contracts, 64% vulnerable) with binary labels. Production val F1-macro: 0.67."

### "How did you handle class imbalance?"
"The training data has 64.33% vulnerable contracts and 35.67% safe — about a 1.8:1 ratio, with the
vulnerable class as the majority. I used Binary Focal Loss with alpha=0.25 for the vulnerable class
and 0.75 for safe. Higher alpha means higher weight, so this deliberately down-weights the majority
vulnerable class and up-weights the harder-to-learn safe class. The weight ratio is 0.75/0.25 = 3×,
stronger than the raw imbalance, combined with gamma=2.0 which multiplies loss by (1-pt)², forcing
focus on uncertain samples. After training, I swept the decision threshold from 0.30 to 0.70 on the
val set to find the F1-macro optimal point — threshold 0.50 gave the best balance at F1-macro 0.67."

### "What is ZKML and why does it matter?"
"ZKML uses zero-knowledge proofs to cryptographically verify that an ML model produced a specific
output — without revealing the model's weights. In SENTINEL, any user can verify our vulnerability
score came from the certified SENTINEL model, not a fabricated one. I used EZKL to convert a
distilled proxy model to a ZK circuit, generate proofs per audit, and verify them on-chain in
Solidity. The hardest part was size: the full model has 239K trainable parameters (plus 125M frozen)
which is far too large for a ZK circuit. I built a small proxy MLP that takes compressed inputs and
outputs the same binary score. The EZKL pipeline generates a Groth16 proof and auto-generates a
Solidity verifier that embeds the verification key as constants."

### "Describe a multi-agent system you built"
"SENTINEL's agent system has specialised agents for comprehensive smart contract analysis. A static
analysis agent runs Slither. An ML intelligence agent calls the GNN+CodeBERT inference API. A RAG
researcher retrieves semantically similar historical exploits from a FAISS vector index (DeFiHackLabs
data, chunked and embedded). The agents are coordinated and their findings synthesized into a
structured audit report. A key design decision was using a local LLM via LM Studio with model
routing — routing simpler classification tasks to a smaller/faster model and complex reasoning to a
larger model, keeping costs zero during development."

### "What advanced Solidity patterns have you used?"
"In SENTINEL: UUPS upgradeable contracts via OpenZeppelin — chosen over transparent proxy for gas
efficiency. SentinelToken is an ERC-20 with Permit and Votes extensions for governance. AuditRegistry
uses a staking model — agents must stake SENTINEL tokens to submit audits. The ZK verifier is
auto-generated by EZKL from the verification key; it embeds BN254 curve parameters as constants and
exposes a single verify(bytes, uint256[]) → bool interface. For testing: Foundry fuzz tests with
vm.assume() for bounded inputs, and invariant tests that enforce 'no audit is ever stored without a
valid ZK proof' across thousands of random action sequences."

### "How do you manage ML models in production?"
"SENTINEL's MLOps stack: MLflow (SQLite backend) tracks every training run — hyperparameters,
per-epoch metrics (F1-macro, precision, recall for vulnerable class), and checkpoint artifacts.
Each run saves to a named checkpoint (`{run_name}_best.pt`) so experiments never overwrite each
other. The checkpoint format includes model weights, AdamW optimizer state, epoch number, and
best F1 — so training can be resumed exactly where it stopped. The new-format checkpoints
(dict with 'model' key) support full resume; old plain state_dict checkpoints are inference-only.
DVC versions the datasets. Future: Evidently AI drift detection on incoming production contracts."

### "What was the hardest data engineering challenge?"
"The training dataset (BCCC-SCsVul-2024) is genuinely multi-label — 41.2% of contracts appear in
multiple vulnerability type folders simultaneously. One contract can be simultaneously Reentrancy +
IntegerUO + Timestamp + CallToUnknown. However, the preprocessing pipeline was designed to assign
one binary label per unique contract hash. When the same hash appeared in multiple folders, only
the first-encountered folder's label was kept. This means the training data is a simplification of
a richer ground truth. A proper multi-label implementation would need a multi-hot label vector per
contract, per-class BCE loss, and per-class threshold tuning — plus handling the severe per-type
imbalance (Reentrancy: ~122 samples vs IntegerUO: ~16K samples). Binary classification was the
pragmatic first step; the CSV retains the class-level labels for a future upgrade."

## Fallback Decision Heuristics

### When to Persist vs Pivot

```
Persist (debug 1-2 more days):
  ✓ Error messages are clear and searchable
  ✓ Solution seems close (>80% working)
  ✓ Learning value is high — this is core technology
  ✓ Problem is in your code, not the external tool

Pivot to fallback:
  ✗ No clear error messages — mystery bugs
  ✗ Problem is inside the external tool/library itself
  ✗ Blocking all progress on other modules
  ✗ Alternative achieves 80% of the learning goal
  ✗ Time spent > 2 days with no visible progress
```

### Priority When Time Is Short

```
MUST HAVE (core learning — do not skip):
  Module 1: Transformer path (CodeBERT fine-tuning)
  Module 5: Basic contracts (AuditRegistry + ZKMLVerifier placeholder)

SHOULD HAVE (impressive, high interview value):
  Module 2: ZKML (even if slow/inefficient)
  Module 4: Basic RAG agent (single agent + 3 tools)

NICE TO HAVE (resume boosters):
  Module 3: MLflow + DVC (basic setup is enough)
  Module 6: Docker Compose (local is fine)

STRETCH (only if time allows):
  GNN dual-path, multi-agent CrewAI, Kubernetes, Next.js frontend,
  Chainlink CCIP, ERC-4337, Feast, full drift detection
```
