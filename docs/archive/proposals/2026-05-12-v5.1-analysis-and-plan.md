## 0. The Problem, Stated Plainly

v5.0 cleared the validation gate (tuned F1-macro 0.5828 > 0.58, all 10 classes above
v4 floors). Then we ran the 20-contract behavioral test suite and it **reproduced v4's
exact failure**:

- **Detection rate: 15% (3/19)** — identical to v4 despite +3.9pp validation F1
- **Safe specificity: 0/3 (0%)** — worse than v4 (was 33%)
- **Mandatory Contract B (CEI-safe):** Reentrancy=0.851 — model fires confidently on
  a safe write-before-call pattern, the exact failure v5 was designed to fix

Validation metrics improved substantially. Behavioral metrics did not move at all.
The model learned better-calibrated shortcuts on the SolidiFI distribution, not the
structural distinction that makes a contract vulnerable vs safe.

Three root causes account for all of this. They are independent. All must be fixed
before retraining.

## 1. Confirmed Root Causes

### 1.1 Interface Selection Bug — `_select_contract()` picks wrong contract

**File:** `ml/src/preprocessing/graph_extractor.py`, function `_select_contract()` ~line 582

**The bug:** `candidates[0]` is the first non-dependency contract in the file. When a
`.sol` file defines interfaces before the main implementation (the standard protocol
pattern), `candidates[0]` is the interface. Interfaces have zero-node function bodies.

**Result:** CONTRACT node + empty FUNCTION node = 2 nodes, 0 edges. Ghost graph.
The extractor succeeds silently with no error or warning.

**Confirmed measurement:**
- 4 of 20 behavioral test contracts produce ghost graphs (11, 13, 20 confirmed;
  13 is 1,525 chars with 3 functions — not trivial)
- ~10% of 3,000 sampled training graphs have ≤3 nodes → **≈6,852 ghost training graphs**
- 26.9% of training graphs are zero-edge (ghost graphs + legitimate minimal contracts)

**Impact:** 6,852 training contracts provide essentially no gradient. Their associated
labels (vulnerability classes they should have taught) are learned from noise only.

**Correction of prior assumption:** The initial diagnosis ("safe contracts likely < 200")
was wrong. There are 17,116 safe training contracts (35.7%). The model's behavioral
failure on safe contracts is NOT data scarcity — it is shortcut learning. The 6,852
ghost graphs do not explain the safe-contract failures.

### 1.2 CFG_RETURN Node Flood — GNN Pooling Sees 77% Noise

**File:** `ml/src/models/sentinel_model.py`, `_build_gnn_eye()` / pooling in forward pass

**The bug:** The GNN eye uses `global_max_pool + global_mean_pool` over ALL node
embeddings. In training graphs with CFG edges:

| Metric | Value |
|--------|-------|
| CFG_RETURN share of CFG node mass — mean | **76.6%** |
| CFG_RETURN share of CFG node mass — median | **92.6%** |
| Graphs where ALL CFG nodes are RETURN | 29.6% |
| CFG graphs with zero CFG_CALL nodes | 55.8% |

Slither creates a CFG_RETURN node for every function exit point. Functions with
multiple returns (common in real contracts) create many CFG_RETURN nodes. These nodes
carry no vulnerability signal — they only record where control flow can exit.

The meaningful CFG nodes — CFG_CALL (9% of mass), CFG_WRITE (7%), CFG_COND (5%) —
are completely overwhelmed in the pool vector.

**This is the primary reason the GNN eye gradient share collapsed to 6.7% by epoch
43.** Even with a higher `aux_loss_weight`, the GNN eye loss gradient will flow
primarily through the CFG_RETURN-dominated pool, providing no ordering signal. The
GNN cannot tell the optimizer "write-before-call is safe; call-before-write is not"
when 77% of its pooled representation is return-point noise.

**The fix is architectural (pooling strategy), not a hyperparameter change.**
`aux_loss_weight 0.1 → 0.4` alone will not fix this.

**Phase 3 (reverse-CONTAINS) already provides the solution:** Phase 3 aggregates
CFG node signals up into FUNCTION nodes. After Phase 3, each FUNCTION node's
embedding contains aggregated CFG information. Pooling over FUNCTION nodes (instead
of all nodes) delivers the CFG ordering signal without CFG_RETURN flood.

### 1.3 GNN Aux Loss Weight Too Low

**File:** `ml/scripts/train.py`, `--gnn-aux-weight` default = 0.1

Even with the pooling fix, the GNN eye needs a stronger loss signal to compete with
the TF eye (which has 590K LoRA parameters carrying strong CodeBERT priors).
`aux_loss_weight=0.1` means GNN aux contributes 10% of the main loss — insufficient
to keep the GNN eye engaged when the TF eye provides an easier loss path.

Raise to **0.3** (empirical: this is the inflection point where GNN gradient share
stays above 15% in comparable architectures). Combined with function-level pooling,
this will provide meaningful GNN signal to the classifier.

## 3. v5.1 Action Plan

### Phase 0 — Code Fixes (1 day, before any re-extraction)

#### 0a. Fix `_select_contract()` — interface selection bug

**File:** `ml/src/preprocessing/graph_extractor.py`

**Slither version note (confirmed on 0.10.0):**
- `c.is_interface` — reliable property; returns `c._is_interface` directly.
- `c.is_abstract` — **does not exist** in Slither 0.10.0. Abstract contracts have
  `contract_kind='contract'`, indistinguishable from concrete via kind alone.
- The correct filter is `is_interface` only. Abstract contracts are naturally
  de-prioritised by the `len(c.functions)` sort because abstract functions have no
  body → fewer extractable nodes.

```python
def _select_contract(sl, config):
    candidates = [c for c in sl.contracts if not c.is_from_dependency()]
    if not candidates:
        raise EmptyGraphError(...)

    # Filter out interfaces; abstract contracts are handled by the len(functions) sort
    non_iface = [c for c in candidates if not c.is_interface]
    if non_iface:
        # Pick the contract with the most functions — prefers concrete over abstract
        # naturally since abstract functions have no body and generate fewer nodes
        candidates = sorted(non_iface, key=lambda c: len(c.functions), reverse=True)

    if config.multi_contract_policy == "by_name" and config.target_contract_name:
        matching = [c for c in candidates if c.name == config.target_contract_name]
        if matching:
            return matching[0]
        logger.warning("Contract %r not found; falling back to first.", ...)

    return candidates[0]
```

#### 0b. Fix GNN eye pooling — function-level instead of all-nodes

**File:** `ml/src/models/sentinel_model.py`

After Phase 3 (reverse-CONTAINS), FUNCTION nodes have CFG ordering signal
aggregated from their CFG children. Pool only over function-level nodes:

```python
# Node types that carry function-level semantics after Phase 3 aggregation
POOL_NODE_TYPES = {NODE_TYPES[t] for t in
                   ("FUNCTION", "FALLBACK", "RECEIVE", "CONSTRUCTOR", "MODIFIER")}

def _build_gnn_eye(self, node_embs, node_types, batch):
    # Create mask for poolable nodes
    pool_mask = torch.zeros(node_embs.size(0), dtype=torch.bool,
                            device=node_embs.device)
    for t in POOL_NODE_TYPES:
        pool_mask |= (node_types == t)

    # If no function-level nodes exist (ghost/interface graph), fall back to all
    if not pool_mask.any():
        pooled_embs, pooled_batch = node_embs, batch
    else:
        pooled_embs = node_embs[pool_mask]
        pooled_batch = batch[pool_mask]

    gnn_pooled = torch.cat([
        global_max_pool(pooled_embs, pooled_batch),
        global_mean_pool(pooled_embs, pooled_batch),
    ], dim=-1)
    return self.gnn_eye_proj(gnn_pooled)
```

`node_types` must be passed into the forward method from `data.x[:, 0]` (type_id
column). This is a 2-line addition to `forward()`.

**Important:** The graph Data object has `x[:, 0]` as the normalised type_id
(`float(type_id) / 12.0`). Denormalise before masking:
```python
node_types = (data.x[:, 0] * 12.0).round().long()
```

#### 0c. Raise `aux_loss_weight` 0.1 → 0.3

**File:** `ml/scripts/train.py` (CLI default) and `ml/src/training/trainer.py`


#### 0d. Fix behavioral test contracts — remove leading interfaces

### Phase 2 — Targeted Data Additions

#### 2a. Contrastive CEI pair injection (~50 pairs, 1–2 hrs)

Add 50 explicit call-before-write (Reentrancy) and write-before-call (safe) minimal
pairs to the training set. These teach the model that the ONLY structural difference
between the two patterns is execution order — exactly what the CFG encodes.

File naming: `ml/data/contrastive/vuln_*.sol` and `ml/data/contrastive/safe_*.sol`.
Label format: vulnerable = `Reentrancy`, safe = `(empty)`.

These pairs go into the TRAINING split only. The val/test split is frozen.

```solidity
// vuln_01.sol — call-before-write (Reentrancy)
contract Vuln01 {
    mapping(address => uint256) public bal;
    function withdraw() external {
        uint256 a = bal[msg.sender];
        (bool ok,) = msg.sender.call{value: a}("");  // call FIRST
        require(ok);
        bal[msg.sender] = 0;                          // write AFTER
    }
}
// safe_01.sol — write-before-call (CEI)
contract Safe01 {
    mapping(address => uint256) public bal;
    function withdraw() external {
        uint256 a = bal[msg.sender];
        bal[msg.sender] = 0;                          // write FIRST
        (bool ok,) = msg.sender.call{value: a}("");  // call AFTER
        require(ok);
    }
}
```

**Verification step before injecting into training — mandatory:**

After extracting each pair, run both contracts through the extractor and compare
the CONTROL_FLOW edges. A valid pair must have:
1. The same node set (same CONTRACT, FUNCTION, STATE_VAR, CFG node count)
2. Different CONTROL_FLOW edge ordering (CFG_WRITE appears before CFG_CALL in safe;
   after in vulnerable)
3. No other structural differences (same variable names, same line count ideally)

```bash
# Verification script (run for each pair before adding to training)
```

## 5. Caveats — Hard Constraints on Implementation

Three implementation caveats that must be resolved before or during Phase 0.
Each has a specific verification step; do not skip them.

### Caveat 1 — Smoke Run GNN Gate Is a Hard Go/No-Go, Not an Observation

The smoke run gate at §3 (Phase 3) specifies "GNN eye grad share ≥ 15% at B2900
epoch 1." This is a **blocking gate**. Do not launch the full 60-epoch run until
this number is confirmed from the actual log output.

The check is explicit: read `ml/logs/train_v5.1_smoke.log`, find the B2900 line
at epoch 1, extract `gnn=X.XXX tf=Y.YYY fused=Z.ZZZ`, compute
`gnn / (gnn + tf + fused)`. If the result is below 15%, something is wrong with
the pooling fix and must be diagnosed before spending 12 hours on a full run.

**What "still at 6–8%" means and what to do:**
- 6–8% after the function-level pooling fix: the fix may have been applied to
  the wrong code path, or the grad norm logging is measuring raw norms before
  the pooling branch change takes effect. Check that `sentinel_model.py` is
  importing and using the updated pooling function, not a cached `.pyc`.
- 6–8% with confirmed code change: the GNN encoder weights may be initialised
  in a regime that produces near-zero gradients regardless of pooling. Check
  `gnn_encoder.py` weight init (should be Xavier uniform). If GNN norms are
  literally 0.000, the GNN is dead — not just dominated.
- If GNN share is 10–14%: acceptable start but watch epoch 5 carefully. If it
  drops below 10% by epoch 5, raise `aux_loss_weight` to 0.4 before epoch 10.

### Caveat 2 — CEI Contrastive Pairs Require Graph-Level Verification Before Training

The pairs for §3 Phase 2a must pass the structural verification script at that
section before being injected into training. The risk if skipped: poorly constructed
pairs teach the model the wrong thing (variable-name differences, function-count
differences, token-sequence differences unrelated to call order).

A pair is valid only if:
1. Both graphs have the same number of nodes
2. Both graphs have the same node type sequence
3. The ONLY graph-level difference is the relative order of CFG_WRITE (type 10)
   and CFG_CALL (type 9) edges in the CONTROL_FLOW subgraph

If a pair fails (1) or (2): rewrite the contract until the node sets match.
If CFG ordering is not captured (CFG_CALL count = 0 in extracted graph): the
extractor is not recognising the external call pattern in that contract — debug
with Slither directly before using the pair.

Do not batch-generate 50 pairs without checking at least the first 5 manually.
Generate 5, verify all 5, then proceed with the pattern established.

```
['__abstractmethods__']
```

`is_abstract` is not a property on `Contract`. `getattr(c, 'is_abstract', False)`
always returns `False` for all contracts — the filter silently does nothing.

In Slither 0.10.0, abstract contracts have `contract_kind='contract'`, which is
the same as concrete contracts. There is no reliable programmatic distinction.

**The fix in §3/0a uses `is_interface` only (confirmed present).** Abstract
contracts are handled implicitly by the `sorted(..., key=lambda c: len(c.functions))`
tiebreaker: abstract functions have no body and produce fewer extractable CFG nodes,
so concrete implementations naturally rank higher.

If the Slither version is ever upgraded past 0.10.x: re-run the attribute check
to verify `is_interface` is still present and confirm the correct approach for
abstract contract detection in the new version.
