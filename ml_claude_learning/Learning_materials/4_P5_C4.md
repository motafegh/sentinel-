## Warm-Up Answers + Gap-Fill

---

**W1. conv3 output shape**

`conv3 = GATConv(in=256, out=64, heads=4, concat=True)`.
Each of the 4 attention heads independently produces a 64-dim vector per node. `concat=True` stacks them side by side: `4 × 64 = 256`. Output shape: `[N, 256]`. The `hidden_dim=256` invariant is preserved across the phase boundary.

---

**W2. Phase 3 heads=1**

Phase 2's job is *specialization* — different heads learn to attend to different execution patterns (intra-function CF, call entry, return-to, def-use). Heads give you parallel views of heterogeneous signals.

Phase 3's job is *aggregation* — it answers one question: "what Phase 2 signal lives in my container subtree?" CONTAINS is a single unambiguous hierarchical relationship. There is no second or third *type* of containment to specialize over. Multiple heads would just produce redundant versions of the same aggregation, waste parameters, and add noise. One head, full `hidden_dim=256` output, is the correct design.

---

**W3. nn.ModuleList vs plain list**

🔵 **Portable recall — nn.ModuleList:** PyTorch's `.to(device)`, `.parameters()`, `.state_dict()`, and `.train()`/`.eval()` all recursively walk the module tree. They only walk `nn.Module` children registered via assignment or `nn.ModuleList`/`nn.ModuleDict`. A plain Python `list` is invisible to this walk.

Concrete failure: if `phase_norm` were a plain list `[LayerNorm(), LayerNorm(), LayerNorm()]`, calling `model.cuda()` would move every conv and linear to GPU, but the three LayerNorm layers would stay on CPU. The forward pass would then try to add a CPU LayerNorm output to a CUDA tensor → runtime device mismatch error. Worse, their parameters would not appear in `model.parameters()`, so the optimizer would never update them — silent wrong training.

---

**W4. Stale _param_dtype — failure path**

At construction, model is float32 → `_param_dtype = torch.float32`.

Caller does `model.half()` — all parameters and buffers become float16. But `_param_dtype` still says float32.

Forward pass: `if x.dtype != self._param_dtype` → compares input dtype to `torch.float32`. If input arrives as float32 (common, since many data pipelines produce float32): condition is `False`, no cast happens. `x` stays float32. Then `self.conv1(x_init, ...)` — conv1's weights are float16. On **CPU**: PyTorch upcasts float16 weights to float32 silently, computation happens in float32, memory savings from half() are lost. On **CUDA**: strict dtype matching is required; PyTorch raises a runtime error (`Expected all tensors to be on the same device / same dtype`), but the error message points at the conv operation, not the stale cache — root cause is invisible.

The fix (A3): override `.half()` / `.bfloat16()` / `.float()` / `.to()` to auto-call `refresh_dtype_cache()`.

---

**W5. phase_norm purpose**

After Phase 1 (2 layers), node embeddings have been through 2 residual + conv operations. After Phase 2 (3 layers), they've been through 3 more. After Phase 3 (3 more), 3 more. The L2 magnitude of embeddings scales roughly with depth — Phase 1 outputs will systematically have a different magnitude than Phase 2 or Phase 3 outputs, purely as a function of how many operations they passed through.

JK attention computes scores via `Linear(256→1)` applied to each phase output, then `softmax` across the three scores per node. If Phase 1's magnitude is higher, its raw score will be higher, and softmax will assign it a higher weight — not because Phase 1 is semantically more useful, but because it happens to have larger values. LayerNorm (applied once per phase before collecting for JK) normalizes each vector to have mean≈0 and std≈1, removing the magnitude bias. Now JK learns actual routing based on content, not scale.

---

**W6. IMP-G3 directional embedding distinction**

`edge_embedding` is an `nn.Embedding(11, 64)` — 11 rows, one per edge type ID.

- **Upward pass** (conv4 and conv4b, CFG→FUNCTION): uses `REVERSE_CONTAINS` = type ID 7. The embedding lookup returns `edge_embedding.weight[7]` — a 64-dim vector that was trained specifically on the "child signals rising to parent" direction.
- **Downward pass** (conv4c, FUNCTION→CFG): uses `CONTAINS` = type ID 5. The embedding lookup returns `edge_embedding.weight[5]` — a different learned vector, trained on the "parent distributes to children" direction.

Without IMP-G3 (old code used the same embedding for both), the network would see identical edge features on both directions and would have to use the *graph topology alone* to distinguish up from down. With IMP-G3, the edge embedding itself carries a directional signal — the network gets an explicit hint about which direction the message is flowing.

## Chunk 4 — `forward()`: Guards, Dtype, Edge Embeddings, Edge Masks

---

### Big Picture for This Chunk

`forward()` is where everything built in `__init__` runs. It's ~270 lines and splits naturally into two halves:

- **Chunk 4 (this chunk, lines 360–538):** Input validation, dtype normalization, edge embedding lookup, edge mask computation, Phase 3 reverse edge construction. All *setup* before the convolutions start.
- **Chunk 5 (next):** The actual 8-layer convolution sequence, JK aggregation, and return paths.

This chunk is the "routing layer" — it decides which edges go to which phase, builds the runtime REVERSE_CONTAINS edges, and safely handles every error case a production system will encounter.

---

### Signature + Return Modes

```python
# Learning mode: Master the mechanism

def forward(
    self,
    x:                    torch.Tensor,          # ← MASTER: [N, NODE_FEATURE_DIM=11] — raw node features
    edge_index:           torch.Tensor,          # ← MASTER: [2, E] — COO format: row 0 = source, row 1 = dest
    batch:                torch.Tensor,          # ← UNDERSTAND: [N] — which graph in the batch each node belongs to
    edge_attr:            torch.Tensor | None = None,  # ← MASTER: [E] int64 — edge type IDs
    return_intermediates: bool               = False,  # ← UNDERSTAND: diagnostic mode, returns detached per-phase embs
    return_phase2_embs:   bool               = False,  # ← MASTER: returns Phase 2 output WITH gradients (for aux loss)
):
```

**`x` shape — why `[N, 11]` not `[N, 27]`?**
The 27-dim `_GNN_IN_DIM` is *built at runtime* inside `forward()` — the raw `.pt` file stores 11 features per node. The `type_embedding` lookup that prepends the 16-dim type vector happens here, not at data loading time.

**`edge_index` — COO format:**

🔵 **Portable recall — COO (Coordinate) format:** PyTorch Geometric represents sparse graphs as a `[2, E]` tensor. `edge_index[0]` = all source node indices, `edge_index[1]` = all destination node indices. One column per edge. `edge_index[:, k]` = the k-th edge: `(source=edge_index[0,k], dest=edge_index[1,k])`. This format appears in every PyG codebase — it's the universal sparse graph representation in the PyG ecosystem.

**Two return flags — design decision:**
- `return_intermediates=True`: diagnostic only. Returns detached (no gradients) per-phase tensors for inspection/visualization. You can't backpropagate through them. Mutually exclusive with `return_phase2_embs`.
- `return_phase2_embs=True`: used in the training loop by `sentinel_model.py`. Returns Phase 2 output *with gradients attached* so the `aux_phase2` auxiliary head can compute its own loss and send gradients back through Phase 2 conv layers. This is the BUG-R7-1 fix — before v8.1, Phase 2 layers had no direct gradient path.

### Section 1 — Guards

Three guards run before any computation. Each targets a specific failure mode where the default behavior would produce a silent wrong result or an unhelpful crash.

```python
# Learning mode: Master the mechanism

# Guard 1 — schema version check
if x.shape[1] != NODE_FEATURE_DIM:                    # ← MASTER: catches stale v6 .pt files at the source
    raise ValueError(
        f"GNNEncoder expects {NODE_FEATURE_DIM}-dim node features (schema v8) "
        f"but got {x.shape[1]}. Likely a stale v6 .pt file — re-run reextract_graphs.py."
    )

# Guard 2 — edge_attr required when use_edge_attr=True
if self.use_edge_attr and edge_attr is None:           # ← MASTER: without this, cfg_mask = all-zeros silently
    raise ValueError(
        "GNNEncoder was built with use_edge_attr=True but edge_attr=None "
        "was passed. ..."
    )

# Guard 3 — graph integrity (gated by flag)
if self.validate_graph_integrity and edge_index.numel() > 0 and edge_index.max() >= x.shape[0]:
    raise ValueError(...)                              # ← UNDERSTAND: OOB node index — only enable for debugging
```

**Why Guard 2 matters in detail:**
`cfg_mask` is built from `edge_attr == CONTROL_FLOW`. If `edge_attr is None`, the fallback path sets `cfg_mask = torch.zeros(...)` — all False. Phase 2 runs with *zero edges*. Every Phase 2 conv call receives an empty `edge_index`. GATConv on empty edges returns all zeros. Residual adds those zeros back (no-op). The model trains for an entire epoch with Phase 2 completely disabled — no error, no warning with the old code. Guard 2 surfaces this the moment a misconfigured graph enters the pipeline.

**Why Guard 3 is gated:**
`edge_index.max()` is O(E) — it scans all E edge indices. On a batch of 32 graphs with 10K edges each, that's 320K comparisons *per forward pass*. At training time this adds measurable overhead. The guard is designed for debugging and testing only, disabled in production via `validate_graph_integrity=False` (the default). The comment also suggests the right production fix: move the check to `DualPathDataset.__getitem__` so it runs once at data load time, not every forward pass.

### Section 2 — Dtype Normalization

### Section 3 — Edge Embedding Lookup

```python
# Learning mode: Master the mechanism

e = None
if self.edge_embedding is not None and edge_attr is not None:
    if edge_attr.numel() > 0:
        _max_valid = self.edge_embedding.num_embeddings - 1   # = 10
        _oob_mask = (edge_attr < 0) | (edge_attr > _max_valid)
        if _oob_mask.any():
            logger.warning(...)                                # ← UNDERSTAND: warn, don't crash
            edge_attr = edge_attr.clamp(0, _max_valid)         # ← MASTER: clamp OOB before lookup
    e = self.edge_embedding(edge_attr)   # [E, 64]             # ← MASTER: integer IDs → 64-dim vectors
```

**Why clamp instead of raise?**
Out-of-bounds indices in `nn.Embedding` cause a CUDA illegal-memory-access error — one of the least debuggable PyTorch failures, with a stack trace that points at CUDA internals and no indication of which sample triggered it. Clamping lets the forward pass continue on a corrupted sample (with a warning logged), allowing a training run over 100K contracts to survive a handful of malformed `.pt` files without crashing at midnight. The warning includes the observed min/max, making the corrupt file traceable.

**`e` is `None` when `edge_embedding is None`** — the model supports a no-edge-attr mode where all edge type information is discarded and GATConv falls back to topology-only attention. `e` stays `None` throughout and GATConv handles `edge_attr=None` internally.

### Section 4 — Edge Mask Computation

This is the core routing logic. It splits the single flat `edge_attr` tensor into separate edge sets for each phase.

```
All edges: [E] int64 type IDs
          │
          ├── struct_mask:   type ≤ 5  → Phase 1 (structural)
          ├── cfg_mask:      types 6,8,9,10 → Phase 2 (CFG + ICFG)
          └── contains_mask: type == 5 only → Phase 3 (for reverse construction)
```

```python
# Learning mode: Master the mechanism

_CONTAINS         = EDGE_TYPES["CONTAINS"]       # 5
_CONTROL_FLOW     = EDGE_TYPES["CONTROL_FLOW"]   # 6
_CALL_ENTRY       = EDGE_TYPES["CALL_ENTRY"]     # 8
_RETURN_TO        = EDGE_TYPES["RETURN_TO"]      # 9
_DEF_USE          = EDGE_TYPES["DEF_USE"]        # 10

if edge_attr is not None:
    struct_mask   = edge_attr <= _CONTAINS        # ← MASTER: types 0–5 inclusive
    if self.phase2_edge_types is not None:
        cfg_mask = torch.zeros(..., dtype=torch.bool, ...)   # ← MASTER: ablation path — custom subset
        for _t in self.phase2_edge_types:
            cfg_mask |= (edge_attr == _t)
    else:
        cfg_mask  = (                             # ← MASTER: default — types 6,8,9,10
            (edge_attr == _CONTROL_FLOW) |
            (edge_attr == _CALL_ENTRY)   |
            (edge_attr == _RETURN_TO)    |
            (edge_attr == _DEF_USE)
        )
    contains_mask = edge_attr == _CONTAINS        # ← MASTER: type 5 only — used for Phase 3
else:
    struct_mask   = torch.ones(n_edges, dtype=torch.bool, ...)   # ← UNDERSTAND: all edges → Phase 1 only
    cfg_mask      = torch.zeros(...)                              # Phase 2 disabled
    contains_mask = torch.zeros(...)                              # Phase 3 disabled
```

**Why does `struct_mask` include CONTAINS (type 5) even though Phase 3 also uses it?**
Phase 1 needs CONTAINS to propagate *function-level properties down* into CFG nodes during the initial structural encoding. Phase 3 re-uses those same CONTAINS edges for the *downward distribution* after Phase 2 has enriched the embeddings. Same edges, different semantic purpose at different depths — nothing wrong with using them twice.

**The `phase2_edge_types` ablation path:**
When a researcher wants to test "what if we remove DEF_USE from Phase 2?", they pass `phase2_edge_types=[6, 8, 9]`. The loop builds a custom boolean mask. The default `else` branch bakes in types 6, 8, 9, 10. Both paths produce a `cfg_mask: [E] bool`.

### Section 5 — IMP-G1: Layer-Specific Phase 2 Subsets

```python
# Learning mode: Master the mechanism

if phase2_ea is not None:
    phase2_raw = edge_attr[cfg_mask]    # ← MASTER: [E_cfg] raw integer IDs — must use raw, not embedded
    _cf_mask   = (phase2_raw == _CONTROL_FLOW)
    _icfg_mask = (phase2_raw == _CALL_ENTRY) | (phase2_raw == _RETURN_TO)
    cf_only_ei   = phase2_ei[:, _cf_mask]      # ← MASTER: [2, E_cf]   → conv3 only
    cf_only_ea   = phase2_ea[_cf_mask]
    icfg_only_ei = phase2_ei[:, _icfg_mask]    # ← MASTER: [2, E_icfg] → conv3b only
    icfg_only_ea = phase2_ea[_icfg_mask]
else:
    cf_only_ei = icfg_only_ei = phase2_ei       # ← UNDERSTAND: no edge_attr → no sub-masking possible
    cf_only_ea = icfg_only_ea = phase2_ea
```

**Why use `phase2_raw` (integer IDs) and not `phase2_ea` (embedded vectors)?**
`phase2_ea` has shape `[E_cfg, 64]` — it's already been through the embedding lookup. Comparing a float `[E_cfg, 64]` tensor against an integer (e.g., `== _CONTROL_FLOW`) would produce a `[E_cfg, 64]` boolean matrix, not the `[E_cfg]` boolean vector needed to index into `phase2_ei`. The comment in the code explicitly calls this out: "Sub-masks must use raw integer edge_attr, not the embedded phase2_ea." This is a concrete example of a subtle shape bug that wouldn't raise an error — it would produce wrong indices silently.

**What this gives each Phase 2 layer:**

```
conv3  → cf_only_ei    (CONTROL_FLOW only)           — intra-function execution order
conv3b → icfg_only_ei  (CALL_ENTRY + RETURN_TO)      — cross-function call structure
conv3c → phase2_ei     (all 4 types together)         — integration: learns from nodes already enriched by 3+4
```

Layer 5 (conv3c) is the integration layer — it operates on embeddings *already updated* by layers 3 and 4, so it sees the full execution-order context in a single pass.

### Section 6 — Phase 3 Reverse Edge Construction

This is the runtime-only REVERSE_CONTAINS generation from Chunk 1/3.

```python
# Learning mode: Master the mechanism

fwd_contains_ei = edge_index[:, contains_mask]            # ← MASTER: [2, E_contains] — original CONTAINS direction
fwd_contains_ea = e[contains_mask] if e is not None else None

rev_contains_ei = fwd_contains_ei.flip(0)                 # ← MASTER🔵: flip rows → swap source/dest → runtime reversal
if self.edge_embedding is not None:
    n_rev = rev_contains_ei.shape[1]
    if n_rev > 0:
        rev_type_ids = torch.full(
            (n_rev,), _REVERSE_CONTAINS,                  # ← MASTER: fill with type ID 7 (not type 5)
            dtype=torch.long, device=edge_index.device,
        )
        rev_contains_ea = self.edge_embedding(rev_type_ids)  # ← MASTER: lookup type-7 embedding (IMP-G3)
    else:
        rev_contains_ea = None
else:
    rev_contains_ea = None
```

🔵 **Portable recall — `.flip(0)` for edge reversal:** A COO edge_index `[2, E]` stores source in row 0, destination in row 1. Flipping along dim 0 exchanges the two rows: every `(src, dst)` edge becomes `(dst, src)`. This is O(1) — no memory copy, just a view with reversed stride. The pattern `edge_index.flip(0)` is the standard PyG idiom for reversing edge direction without re-building the sparse graph.

**Why `torch.full(..., _REVERSE_CONTAINS)`?**
The reversed edges don't exist in `edge_attr` (REVERSE_CONTAINS is runtime-only, type ID 7). We need to create edge embedding vectors for them. `torch.full((n_rev,), 7)` creates a `[E_contains]` tensor where every entry is 7. Passing that through `edge_embedding` returns `edge_embedding.weight[7]` repeated E_contains times — the type-7 embedding vector for every reversed edge. This gives IMP-G3 its directional distinction: upward edges get the type-7 vector, downward edges (conv4c) get the type-5 vector from `fwd_contains_ea`.

### 3 Things to Lock In

1. **Guard 2 is the most important guard.** `use_edge_attr=True` with `edge_attr=None` produces an all-False `cfg_mask`, silently disabling Phase 2. Without the guard, the model trains without error while the architecture's most important phase does nothing.

2. **`phase2_raw` (not `phase2_ea`) for sub-masking.** You cannot use the 64-dim embedded vectors to filter edges — you need the raw integer IDs. Wrong tensor here = wrong shape boolean mask = silent wrong graph routing.

3. **REVERSE_CONTAINS is `flip(0)` + type-7 embedding.** Two operations: (1) swap source/dest rows to reverse direction, (2) look up type ID 7 (not 5) so the network knows it's an upward pass. Both are needed for IMP-G3 to work — topology alone isn't enough.
