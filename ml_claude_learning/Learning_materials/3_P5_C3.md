### Part 1 — Phase 2 Layers: `conv3`, `conv3b`, `conv3c`

```python
# Learning mode: Master the mechanism

        _p2_heads    = 4                              # ← MASTER: fixed at 4 for Phase 2; not the same as Phase 1's `heads` param
        _p2_head_dim = hidden_dim // _p2_heads        # ← MASTER: 256//4 = 64 per head
        self.conv3 = GATConv(
            in_channels  = hidden_dim,                # 256 — Phase 1 output feeds directly in
            out_channels = _p2_head_dim,              # ← MASTER: 64 per head
            heads        = _p2_heads,                 # ← MASTER: 4 heads, concat=True → 4×64=256
            concat       = True,
            add_self_loops = False,                   # ← MASTER🔵: CRITICAL — directional signal requires no self-loops
            edge_dim     = _edge_dim,
        )
        self.conv3b = GATConv(...)   # identical config — CONTROL_FLOW subset differs at runtime
        self.conv3c = GATConv(...)   # identical config — joint subset at runtime
```

#### IMP-R7-1: Why heads=4? (was heads=1 in v8)

In v8, Phase 2 used `heads=1, concat=False` — one attention function processing all execution-order edges. The single head had to simultaneously learn:
- "which predecessor executes before me in this function?"
- "which callee does this CALL_ENTRY lead to?"
- "which DEF feeds this USE?"

One head = one learned relevance function. It has to compress all three signals into the same attention weight computation — they fight each other.

With `heads=4, concat=True`:
- Head 1 can specialize on write-order (what writes to state before this node?)
- Head 2 can specialize on call depth (what CALL_ENTRY leads into this execution path?)
- Head 3 can specialize on the CEI sequence pattern
- Head 4 can specialize on data-flow timing (DEF_USE)

Critically: **total parameters are unchanged**. `heads=1, out=256` has the same weight matrix size as `heads=4, out=64` — both are a `256×256` matrix, just split differently. Same FLOPS, four parallel attention functions instead of one combined one.

> ⚠️ **CRITICAL** — `_p2_heads=4` and `_p2_head_dim=hidden_dim//4` are **local variables** — not stored as `self._p2_heads`. They only exist during `__init__` to build the GATConv layers. The GATConv layers themselves carry the correct configuration after construction. This means you can't read the Phase 2 head count from a live model instance without inspecting `self.conv3.heads`.

---

#### The Three-Layer Progressive Structure (IMP-G1)

All three Phase 2 layers have **identical construction** — same `in_channels`, same `heads`, same `concat`. Their difference is **runtime**: each layer processes a different edge subset. That distinction lives in the forward pass (Chunk 4), but understanding it now matters for grasping why three separate layers exist.

```
conv3  → processes CONTROL_FLOW (type 6) only
         "what node executes before me within this function?"
         1 hop into intra-function execution order

conv3b → processes CALL_ENTRY (8) + RETURN_TO (9) only
         "what callee does this call site lead into? where does execution return?"
         1 hop into cross-function call structure

conv3c → processes CF(6) + CALL_ENTRY(8) + RETURN_TO(9) + DEF_USE(10) jointly
         "integration layer — combine all flow signals"
         1 hop seeing the complete picture after conv3 and conv3b have enriched embeddings
```

Why three separate hops rather than one layer on the full set? Because each hop **builds on the previous**: by the time conv3c runs, every node's embedding already reflects one hop of intra-function order (from conv3) and one hop of call structure (from conv3b). The integration layer combines already-enriched information, not raw features.

---

#### BUG-H1: Why 3 hops instead of 2?

The comment in the code explains a real vulnerability-detection requirement:

```
2 hops (conv3 + conv3b):  can see CALL → TMP → WRITE
                          "an external call happened 2 steps before this state write"
                          → catches the classic reentrancy pattern

3 hops (+ conv3c):        can see ENTRY → CALL → TMP → WRITE
                          "from the very start of this function's execution,
                           there is a call followed by a state write 3 steps later"
                          → catches the full CEI (Call-Entry-ICFG) sequence
                          → without conv3c, functions that immediately start with
                             an external call before any guard were missed
```

CEI = Call-Entry-ICFG — the 3-node vulnerability fingerprint: execution enters a function's ENTRY node, hits a CALL node (external call), then reaches a WRITE node (state update). The 3-hop design was introduced specifically to capture this complete sequence in one aggregation pass.

---

🔵 **Portable recall — `add_self_loops=False` for directional graphs:**
We established this in Chunk 1, and it's worth anchoring again here as a pattern you'll see in any directed GNN. Self-loops let each node attend to itself during message passing. For structural edges (Phase 1) that's fine — a node should incorporate its own features. For directed flow edges (Phase 2), self-loops destroy the directional signal: node A can no longer distinguish "B executed before me" from "I executed before myself." Every GNN that processes ordered sequences — execution traces, dependency chains, timelines — will have `add_self_loops=False` on those edges. The pattern is universal.

### Part 2 — Phase 3 Layers: `conv4`, `conv4b`, `conv4c`

```python
# Learning mode: Master the mechanism

        self.conv4 = GATConv(
            in_channels  = hidden_dim,    # 256
            out_channels = hidden_dim,    # ← MASTER: 256 — NOT divided across heads
            heads        = 1,             # ← MASTER: single head; full 256-dim capacity
            concat       = False,         # ← UNDERSTAND: with heads=1, concat vs avg makes no difference
            add_self_loops = False,
            edge_dim     = _edge_dim,
        )
        # conv4b — identical (second REVERSE_CONTAINS hop upward)
        # conv4c — identical (IMP-G3: CONTAINS downward pass)
```

#### Why heads=1 for Phase 3?

Phase 3 has one specific job: **route information vertically through the containment hierarchy**. Conv4 and conv4b send execution-order signal upward (CFG → FUNCTION). Conv4c sends enriched context downward (FUNCTION → CFG).

This is a routing problem, not an attention diversity problem. You don't need four parallel "what is relevant?" functions — you need one clean channel that asks "what does this parent/child node know that I should incorporate?" A single head with full 256-dim capacity is more powerful for this than four heads each with 64-dim capacity.

Compare across phases:

| Phase | Layers | heads | out per head | total out | Why |
|-------|--------|-------|-------------|-----------|-----|
| 1 | conv1, conv2 | 8 | 32 | 256 | Diverse structural relationships need diverse attention functions |
| 2 | conv3–conv3c | 4 | 64 | 256 | Multiple CFG signal types need multi-function specialization |
| 3 | conv4–conv4c | 1 | 256 | 256 | Single routing direction — full capacity in one channel |

> ⚠️ **CRITICAL** — With `heads=1, concat=False`: output is `out_channels=256` directly (not `1×256`). The `concat` flag only matters when `heads > 1`. When `heads=1`, concat and average produce identical output. The `concat=False` here is set for clarity and convention, not because it changes behavior.

#### The three Phase 3 layers are NOT symmetric

Despite having identical GATConv configurations, the three layers do very different things at runtime:

```
conv4:   REVERSE_CONTAINS edges — signal flows CFG → FUNCTION (upward)
conv4b:  REVERSE_CONTAINS edges — second upward hop (multi-function depth)
conv4c:  CONTAINS edges         — signal flows FUNCTION → CFG (downward, IMP-G3)
```

The direction reversal happens at the edge level, not the layer level. The same GATConv architecture is pointed at different edge subsets. This is the key design insight of Phase 3: **one layer type, two traversal directions, three hops**.

### Part 3 — Per-Phase LayerNorm, JK, and dtype cache

```python
# Learning mode: Master the mechanism | 🔵 Portable

        self.phase_norm = nn.ModuleList([       # ← MASTER🔵: ModuleList not plain list — see below
            nn.LayerNorm(hidden_dim),           # ← MASTER🔵: LayerNorm normalizes each embedding to mean=0, std=1
            nn.LayerNorm(hidden_dim),           # ← UNDERSTAND: one per phase — each learns its own scale/shift
            nn.LayerNorm(hidden_dim),
        ])

        self.jk = _JKAttention(hidden_dim) if use_jk else None   # ← UNDERSTAND: constructed here; taught in Chunk 2

        self.relu    = nn.ReLU()
        self.dropout = nn.Dropout(dropout)

        self._param_dtype: torch.dtype = next(self.parameters()).dtype  # ← UNDERSTAND: [A26] cached dtype optimization
```

---

🔵 **Portable recall — LayerNorm:**
First appeared in Chunk 1's rationale. Now seeing the construction. LayerNorm normalizes across the **feature dimension** of each embedding independently: for each node, it computes mean and std across its 256 features, then shifts/scales to mean=0, std=1. Two learnable parameters (γ scale, β shift) then allow the network to restore any scale it needs.

**Why three separate instances matter here:** Each instance has its own γ and β. Phase 1 embeddings (just structural aggregation) may need different rescaling than Phase 2 embeddings (execution-order enriched) or Phase 3 (fully hierarchical). Using one shared LayerNorm would force all three phases to share the same learned rescaling — wrong, because their distributions differ. Three instances → three independent normalizations.

**The broader portable pattern:** Any time you concatenate outputs from multiple processing paths before a softmax (or any comparison operation), normalize first. Magnitude differences in raw outputs will dominate softmax weights before the network has any chance to learn useful routing. This is why LayerNorm before JK attention is not specific to Sentinel — it's the correct pattern for any multi-path aggregation.

---

🔵 **Portable recall — `nn.ModuleList` vs plain Python list:**
First saw this principle when `phase_norm` was described. Now seeing it constructed. The rule: any time you store PyTorch modules in a collection on a class, that collection must be a `nn.ModuleList` (or `nn.ModuleDict`). Plain Python lists are invisible to PyTorch's module system — `.to(device)` won't move them, `state_dict()` won't save them, `model.parameters()` won't yield their parameters. This applies universally: any `nn.Module` stored in any collection on any `nn.Module` subclass needs a proper container.

---

#### `_param_dtype` cache — [A26]

```python
self._param_dtype: torch.dtype = next(self.parameters()).dtype
```

`next(self.parameters())` iterates the parameter generator and returns the first parameter tensor, then `.dtype` reads its type (usually `torch.float32`). This is called once at construction and cached.

**Why cache it?** In the forward pass, the code checks `if x.dtype != self._param_dtype: x = x.to(self._param_dtype)`. Without the cache, this would call `next(self.parameters())` on every forward pass — rebuilding the generator, iterating it, reading dtype — small but measurable overhead on a training loop that runs millions of times. Cache once, read always.

**The `refresh_dtype_cache()` companion method:**
```python
def refresh_dtype_cache(self) -> None:
    self._param_dtype = next(self.parameters()).dtype
```
If you call `model.half()` or `model.bfloat16()` after construction, all parameters change dtype but the cache is stale. `refresh_dtype_cache()` exists specifically to update it. The risk: forgetting to call it after a dtype cast means the forward pass silently casts every input to the wrong dtype. This is a defensive design — the method exists because the failure mode is silent.

> **[AUDIT] A3** — `refresh_dtype_cache()` must be called manually after any dtype cast, but nothing enforces this. A user who calls `model.half()` and forgets `refresh_dtype_cache()` gets silent wrong-dtype inputs on every forward pass. A safer pattern: override `__setattr__` or the `.to()` method to invalidate the cache automatically. Alternatively, use a Python `property` that calls `next(self.parameters()).dtype` lazily with a simple `_dtype_dirty` flag.
> **File:** `ml/src/models/gnn_encoder.py` | **Location:** `refresh_dtype_cache` / `_param_dtype` | **Severity:** Medium

```
                 PHASE 1              PHASE 2                  PHASE 3
               (Layers 1+2)        (Layers 3+4+5)           (Layers 6+7+8)
             ─────────────────   ────────────────────     ─────────────────────
in_channels       27                   256                       256
out/head          32                    64                       256
heads              8                     4                         1
concat           True                  True                     False
self_loops       True                 False                     False
output         [N, 256]             [N, 256]                  [N, 256]
               (8×32)               (4×64)                    (1×256)

             aggregation         (directed flow)           (up CFG→FN, down FN→CFG)
─────────────────────────────────────────────────────────────────────────────
after phase  LayerNorm[0]         LayerNorm[1]              LayerNorm[2]
             saved for JK         saved for JK              saved for JK
                    │                   │                        │
                    └───────────────────┴────────────────────────┘

                              _JKAttention(hidden_dim=256)

```

Every phase outputs the same shape `[N, 256]`, despite different head configurations. This is the shape invariant that makes JK aggregation possible.

### 3 Things to Lock In

1. **Phase 2: heads=4 (IMP-R7-1) enables attention specialization.** Four heads can independently learn write-order, call-depth, CEI sequence, and data-flow patterns simultaneously. Same parameter count as heads=1 — just split into four parallel functions. The 3-layer progressive structure (CF only → ICFG only → joint) gives three distinct hops on three distinct edge subsets, capturing the full ENTRY→CALL→TMP→WRITE sequence.

2. **Phase 3: heads=1, full capacity for clean routing.** REVERSE_CONTAINS (up) and CONTAINS (down) are routing passes, not attention diversity problems. One head with 256-dim capacity routes hierarchical signal cleanly. The three Phase 3 layers are configured identically but pointed at different edge subsets at runtime — same architecture, two directions.

3. **`nn.ModuleList` + LayerNorm three separate instances.** Storing modules in ModuleList makes them visible to `.to(device)`, `state_dict()`, and `parameters()`. Three separate LayerNorm instances matter because each phase's embedding distribution differs — they need independent γ/β rescaling before JK attention combines them.
