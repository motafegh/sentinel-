### Part 1 — Module-Level Constants (the numbers everything else depends on)

```python
# Learning mode: Master the detail — these flow into every layer's shape

from ml.src.preprocessing.graph_schema import NODE_FEATURE_DIM, NODE_TYPES, NUM_EDGE_TYPES, EDGE_TYPES

_TYPE_EMB_DIM:   int = 16
_NUM_NODE_TYPES: int = int(max(NODE_TYPES.values())) + 1   # 13
_GNN_IN_DIM:     int = NODE_FEATURE_DIM + _TYPE_EMB_DIM    # 11 + 16 = 27

SENTINEL_GNN_NUM_LAYERS: int = 8
```

**`NODE_FEATURE_DIM = 11`** — the number of features stored per node in every `.pt` graph file on disk. Includes things like: is this node payable? does it write state? what is its normalized type ID? These come from `graph_schema.py`, already taught in Phase 1 of the roadmap.

**`_TYPE_EMB_DIM = 16`** — the dimension of the new categorical type embedding added in v8.1 (BUG-R7-2). 16 dims per node type.

**`_NUM_NODE_TYPES = 13`** — how many distinct node types exist in the v8 schema (IDs 0–12: FUNCTION, CFG_CALL, CFG_WRITE, etc.). The `+1` accounts for Python's `max()` returning the highest ID; the total count is ID_max + 1.

> ⚠️ **CRITICAL — Why 27, not 11?** This is the most important v8.1 change at the data level. The old schema stored node type as `float(type_id) / 12.0` inside the 11-feature vector — a continuous scalar in [0, 1]. GATConv treats this as just another continuous feature: it can learn "higher values are more important" but it **cannot** learn "type 3 and type 9 are categorically different things that deserve different representations." That's the bug. The fix: at runtime, read that scalar back out, recover the integer type ID, look it up in a learned embedding table, and concatenate the resulting 16-dim vector to the 11 features. Now GATConv sees 27 dims where types have real representation space. No graph re-extraction needed — the integer is recoverable from the stored float.

**`SENTINEL_GNN_NUM_LAYERS = 8`** — the 2+3+3 phase architecture is fixed. This constant plus a runtime check in `__init__` ensure nobody accidentally builds a 6-layer GNNEncoder expecting it to work.

### Part 2 — `_JKAttention` — The Phase Blender

This is a standalone `nn.Module` defined before `GNNEncoder`. It has exactly one job: given the three phase outputs, combine them into one final embedding per node using learned attention.

#### `__init__`

```python
# Learning mode: Master the detail

class _JKAttention(nn.Module):

    def __init__(self, channels: int, num_phases: int = 3) -> None:
        super().__init__()
        self.attn = nn.Linear(channels, 1, bias=False)
        self.register_buffer("last_weights",     torch.zeros(num_phases))
        self.register_buffer("last_weight_stds", torch.zeros(num_phases))
        self.last_node_weights: "torch.Tensor | None" = None
```

**`self.attn = nn.Linear(channels, 1, bias=False)`**
This is the entire scoring mechanism. One linear layer: 256 inputs → 1 scalar output. Given any phase embedding, it produces a score saying "how much should this node rely on this phase?" With 3 phases, you get 3 scores → softmax → 3 weights summing to 1. Only 256 parameters total.

`bias=False` because a constant bias added to all three scores cancels perfectly in softmax — it changes nothing. No point training it.

**`register_buffer` vs plain attribute — the critical distinction (P14):**

`register_buffer("last_weights", ...)` registers a tensor as **state** but not a **parameter**:
- Moves with the model when you call `.to(device)` or `.cuda()`
- Saved and restored by `state_dict()` / `load_state_dict()`
- **Not** in `model.parameters()` — no gradient, never updated by the optimizer

`last_node_weights` is a **plain Python attribute**, deliberately not a buffer. Why? Because N (total nodes in the batch) changes every batch. You cannot register a variable-shape tensor as a buffer — PyTorch requires fixed shapes at registration time. So it lives as a plain attribute, immediately moved to `.cpu()` when set, and only populated in eval mode where it's safe to hold for diagnostic reading.

The `last_weight_stds` buffer (standard deviation of per-phase weights across nodes) answers the diagnostic question: "is the model actually routing different nodes to different phases (`std > 0.10`) or treating all nodes identically (`std ≈ 0`)?" The trainer logs this.

#### `forward()` — step by step (P14)

```python
# Learning mode: Master the detail — this is the JK computation core

def forward(self, xs: list[torch.Tensor]) -> tuple[torch.Tensor, torch.Tensor]:
    stacked = torch.stack(xs, dim=1)         # [N, K, 256]   K=3 phases
    scores  = self.attn(stacked)              # [N, K, 1]
    weights = torch.softmax(scores, dim=1)    # [N, K, 1]   sums to 1 over K
    w_nk    = weights.squeeze(-1)             # [N, K]

    self.last_weights.copy_(w_nk.mean(0).detach())
    self.last_weight_stds.copy_(
        w_nk.std(0, unbiased=False).nan_to_num(0.0).detach()   # [A23]
    )
    if not self.training:
        self.last_node_weights = w_nk.detach().cpu()

    output     = (weights * stacked).sum(dim=1)                # [N, 256]
    jk_entropy = -(w_nk * (w_nk + 1e-8).log()).sum(dim=1).mean()
    return output, jk_entropy
```

**Step 1 — `torch.stack(xs, dim=1)`**
`xs` is a Python list of 3 tensors, each `[N, 256]`. `stack` inserts a new axis at position 1 and stacks along it: result is `[N, 3, 256]`. Every node now has its 3 phase embeddings laid out side by side.

**Step 2 — `self.attn(stacked)` → `[N, 3, 1]`**
`nn.Linear` broadcasts over any leading dimensions — it treats the input as N×3 independent 256-dim vectors and applies the same weight matrix to each. Each gets one scalar score.

**Step 3 — `torch.softmax(scores, dim=1)` → `[N, 3, 1]`**
Softmax over `dim=1` (the phase axis). For each node independently: turns 3 raw scores into 3 non-negative weights that sum exactly to 1.0. This is the attention distribution over phases.

**Step 4 — `weights.squeeze(-1)` → `[N, 3]`**
Removes the trailing size-1 dimension. Now `w_nk[i, j]` is "how much does node `i` rely on phase `j`?"

**Step 5 — `(weights * stacked).sum(dim=1)` → `[N, 256]`**
Broadcasting: `[N, 3, 1] * [N, 3, 256]` = `[N, 3, 256]` (each phase scaled by its weight). `.sum(dim=1)` collapses the phase axis. This is exactly: `α₁×phase1 + α₂×phase2 + α₃×phase3` per node.

**Step 6 — JK entropy: `-(w_nk * (w_nk + 1e-8).log()).sum(dim=1).mean()`**
Shannon entropy H = -Σ pᵢ log(pᵢ) computed per node, averaged to a scalar.
- `+1e-8`: numerical guard — `log(0) = -∞`, which would produce NaN in the loss
- High H (≈ log(3) ≈ 1.10): weights spread uniformly across phases → diverse routing
- Low H (≈ 0): one phase gets weight ≈ 1 → collapsed attention → model ignores two phases

> ⚠️ **CRITICAL — entropy has no `.detach()`**. This is intentional. The trainer adds it to the main loss: `total_loss = main_loss - λ * jk_entropy` (maximizing entropy = penalizing collapse). Gradients flow back through `self.attn` and discourage the model from locking onto one phase. If `.detach()` were added here, the regularizer becomes a floating diagnostic with zero training effect — silent and dangerous.

**The `[A23]` comment on `unbiased=False`:**
`torch.std(..., unbiased=True)` (the default) uses Bessel's correction (divides by N-1 instead of N). When N=1 (a batch with a single node), this produces `NaN` — you can't compute a sample standard deviation with one sample. `unbiased=False` divides by N, returning 0 when N=1. `.nan_to_num(0.0)` is a belt-and-suspenders guard for any NaN that still slips through.

#### The `_head_dim` shape invariant

```python
# Learning mode: Master the detail — this invariant propagates through every phase

_head_dim = hidden_dim // heads   # 256 // 8 = 32
if _head_dim * heads != hidden_dim:
    raise ValueError(...)
```

With `concat=True`, GATConv concatenates all head outputs: total output = `heads × out_channels`. For the output to stay `hidden_dim=256`, each head must output `256 / 8 = 32` dims. Python's `//` truncates, so `hidden_dim=260, heads=8` gives `_head_dim=32`, then `32×8=256 ≠ 260` — caught. Without this check, you'd get a silent shape mismatch on the residual addition later.

#### `edge_embedding` — Edge type lookup table

```python
# Learning mode: Understand the pattern

if use_edge_attr:
    self.edge_embedding = nn.Embedding(NUM_EDGE_TYPES, edge_emb_dim)  # 11 types × 64 dims
    _edge_dim = edge_emb_dim
else:
    self.edge_embedding = None
    _edge_dim = None
```

`nn.Embedding(11, 64)` is a lookup table: 11 rows (one per edge type), 64 columns. Pass in an integer type ID 0–10, get back a 64-dim learned vector. `_edge_dim` is then passed as `edge_dim=_edge_dim` to every GATConv layer — it tells PyG the incoming edge feature shape so the attention mechanism can factor in edge type when computing weights.

#### `type_embedding` — BUG-R7-2 categorical fix (NEW in v8.1)

```python
# Learning mode: Master the detail — new in v8.1, always present (no gate)

# BUG-R7-2: categorical node-type embedding.
self.type_embedding = nn.Embedding(_NUM_NODE_TYPES, _TYPE_EMB_DIM)
# 13 node types × 16 dims = 208 parameters total
```

Unlike `edge_embedding`, this has **no `use_edge_attr` gate** — it's always created. Why? Because it's not a feature option; it's a **correctness fix**. The original scalar encoding was fundamentally wrong. There's no scenario where you'd want to revert to it.

At runtime (in `forward()`), the type embedding is looked up and prepended to the 11-dim features before any conv layer sees the input:

```python
# (you'll see this in Chunk 5 — forward pass)
type_ids = x[:, 0].mul(12).round().long().clamp(0, _NUM_NODE_TYPES - 1)
type_emb  = self.type_embedding(type_ids)       # [N, 16]
x = torch.cat([x, type_emb], dim=1)             # [N, 27]
```

The type ID was stored as `float(type_id)/12.0` in feature slot 0. To recover: multiply by 12, round, cast to integer, clamp for safety. Then embed. The 27-dim result is `_GNN_IN_DIM` — what every Phase 1 layer actually receives.

> **[AUDIT] A1** — `phase2_edge_types` accepts arbitrary integers with no bounds check. If you pass `[99]`, the forward pass silently builds an all-False `cfg_mask` — Phase 2 is completely disabled, no edges match, no warning is logged. The model trains on structural signal alone. Fix: validate in `__init__`: `if any(t < 0 or t >= NUM_EDGE_TYPES for t in phase2_edge_types): raise ValueError(...)`.

#### `input_proj` — IMP-G2 skip connection

```python
# Learning mode: Master the detail

self.input_proj = nn.Linear(_GNN_IN_DIM, hidden_dim, bias=False)
# Linear(27, 256, bias=False) → 27 × 256 = 6,912 parameters
```

Projects the full 27-dim enriched input directly to `hidden_dim=256`, parallel to `conv1`. Its output is added to `conv1`'s output before the ReLU (forward pass detail). This ensures raw feature information (including the type embedding) always survives the first layer regardless of how well GAT attention is initialized.

`bias=False` — `conv1` already has its own bias; adding another would just learn an equivalent combined constant at double the parameter cost.

#### `conv1` and `conv2` — Phase 1 GAT layers

```python
# Learning mode: Master the detail — shapes are the invariant

self.conv1 = GATConv(
    in_channels  = _GNN_IN_DIM,   # 27
    out_channels = _head_dim,      # 32 per head
    heads        = 8,
    concat       = True,           # 8 × 32 = 256 total output
    add_self_loops = True,
    edge_dim     = _edge_dim,      # 64
)
self.conv2 = GATConv(
    in_channels  = hidden_dim,     # 256
    out_channels = _head_dim,      # 32 per head
    heads        = 8,
    concat       = True,           # 8 × 32 = 256 → residual-compatible with input
    add_self_loops = True,
    edge_dim     = _edge_dim,
)
```

**GAT (Graph Attention Network) — what it does (P11):**
Standard message passing averages all neighbor embeddings equally. GAT replaces that with a learned attention weight per neighbor: `new_A = Σ α(A,B) × W × B_embedding` where `α(A,B)` is a scalar the network learns, representing "how relevant is B to understanding A?" Multi-head runs this K times in parallel, each head potentially learning different relevance criteria.

**The concat=True shape invariant:**
With `concat=True`, outputs of all 8 heads are concatenated (not averaged). Each head outputs `out_channels=32` dims → concatenated = `8 × 32 = 256`. This equals `hidden_dim`, which is required for the **residual connection** on `conv2` (its input and output are both 256, so `x = x + conv2(x)` is shape-safe).

**Why Phase 1 uses 8 heads:**
Structural edges (CALLS, READS, WRITES, INHERITS, CONTAINS) are diverse — each carries a different kind of relationship. Eight heads let the model learn eight different "what makes this neighbor relevant?" functions simultaneously. Some heads may specialize for inheritance, others for call structure, others for data dependency.

> **[AUDIT] A2** — Phase 2 hardcodes `_p2_heads=4` with no divisibility guard. Phase 1 validates `_head_dim * heads != hidden_dim` (line 208), but Phase 2 does `_p2_head_dim = hidden_dim // _p2_heads` with no equivalent check. If `hidden_dim` is not divisible by 4, `GATConv` will raise an unhelpful shape error at forward-pass time, not at construction. Fix: add `if hidden_dim % _p2_heads != 0: raise ValueError(...)` immediately after `_p2_heads = 4`.
> **File:** `ml/src/models/gnn_encoder.py` | **Severity:** Medium

#### Per-phase LayerNorm and JK construction

```python
# Learning mode: Understand the pattern

self.phase_norm = nn.ModuleList([
    nn.LayerNorm(hidden_dim),   # after Phase 1
    nn.LayerNorm(hidden_dim),   # after Phase 2
    nn.LayerNorm(hidden_dim),   # after Phase 3
])

self.jk = _JKAttention(hidden_dim) if use_jk else None
```

`nn.ModuleList` is a PyTorch-aware list. If you stored these as a plain Python list, `.to(device)`, `state_dict()`, and the optimizer would not see the LayerNorm parameters. ModuleList makes them proper sub-modules. Three separate LayerNorm instances — each learns its own scale and bias for its phase.

The `_JKAttention(hidden_dim)` instantiation is straightforward: channels=256, num_phases=3 (default).

### 3 Things to Lock In

1. **`_GNN_IN_DIM = 27` is a runtime construction, not a disk format.** The `.pt` graph files still store 11-dim features. At runtime, the type ID (already encoded as a float in slot 0) is recovered, looked up in `type_embedding` (13×16), and concatenated. GATConv now sees categorical structure instead of a meaningless scalar — that's BUG-R7-2's fix.

2. **`_JKAttention` has one 256→1 linear layer that scores all three phase embeddings per node.** The scores go through softmax (producing 3 weights summing to 1), then the weighted sum of the three phase outputs is returned. The entropy of those weights is returned gradient-attached so the trainer can penalize collapsed attention.

3. **`register_buffer` vs plain attribute is a deliberate choice, not style.** Buffers survive `.to(device)`, `save/load`, and DDP. `last_node_weights` is a plain attribute because N varies per batch — fixed-shape requirement rules out buffer registration. Always ask: "does this tensor need to travel with the model?" If yes, use `register_buffer`.
