## Answering All 5 Questions

---

### W1 — What problem does JK solve, and why keep all three phase outputs?

**The problem: oversmoothing.**

When you stack multiple GNN layers, each layer aggregates from neighbors. After enough layers, every node's embedding starts to look like the average of the entire graph — all nodes converge toward the same representation. This is called **oversmoothing**. The deeper the network, the worse it gets.

In this architecture: Phase 1 produces structural embeddings. Phase 2 then runs on top of those — each node again aggregates from neighbors. Then Phase 3 runs on top of Phase 2. By the end, Phase 3's output has been averaged 8 times through the graph. Phase 1's signal — the raw structural identity of each node — has been diluted through 8 rounds of neighbor mixing.

If you kept only Phase 3's output, you'd lose whatever Phase 1 captured directly. Some nodes carry useful information *because of their structural type*, not because of what runs before or after them. A `MODIFIER` node, for example, carries access-control information that's most clearly expressed after Phase 1, before CFG averaging dilutes it.

**JK's answer:** Don't throw away earlier phases. Keep the output of each phase, then let the model decide per-node how to blend them:

```
final = α₁×phase1_out + α₂×phase2_out + α₃×phase3_out
        where α₁+α₂+α₃ = 1, learned per-node
```

A `MODIFIER` node might learn `α₁ ≈ 0.7` (heavy weight on structural phase). A `CFG_CALL` node at a reentrancy site might learn `α₂ ≈ 0.8` (heavy weight on execution-order phase). The model routes each node type to the phase that best encodes its relevant information.

### W2 — What failure mode does `input_proj` (IMP-G2) prevent?

**The problem: uniform attention initialization.**

At the start of training, GAT's attention weights are random and often near-uniform — every neighbor gets roughly equal weight. In that regime, `conv1` is essentially averaging all neighbors' 27-dim features together.

For node A with 10 neighbors: the output of `conv1` is approximately `(1/10) × Σ neighbor_features`. Node A's own features are only 1/10 of its output. Its raw information is diluted immediately at layer 1 — before the network has had any chance to learn what A's features mean.

`input_proj` adds a direct path:
```
conv1_output = GATConv(x) + Linear(x)    # both [N, 256]
```

No matter how bad the GAT attention is at init, the raw 27-dim features of each node are always directly present in the layer 1 output. The network starts from a better position — it hasn't lost the identity of each node before training even begins.

The name "skip connection" is the same concept as in ResNet: bypass a layer to ensure gradient and information flow don't depend entirely on that layer being well-initialized.

### W3 — FUNCTION node with no CFG children in Phase 3 — correct or bug?

**What happens mechanically:**

Phase 3 runs `REVERSE_CONTAINS` edges — these go from CFG nodes **up** to their parent FUNCTION nodes. If a FUNCTION node has zero CFG children, then zero edges point to it in the REVERSE_CONTAINS graph. GATConv has nothing to aggregate from — it returns a zero tensor.

The residual connection then does:
```
x = x + dropout(zero_tensor) = x + 0 = x
```

The node's embedding is unchanged. Phase 3 is a no-op for this node.

**Why this is correct:**

Phase 3's purpose is "aggregate execution-order signal from CFG nodes up to their parent function." A function with no CFG children has no execution-order signal to aggregate — it's a pure/view/abstract function that doesn't have implementation instructions tracked in the graph. The correct behavior is: that function's embedding stays exactly where Phase 1+2 left it. The zero-message produces a no-op, which is exactly the right result.

If you "fixed" this by adding artificial edges or padding, you'd be injecting false information — pretending that empty functions have execution context they don't have. That would be the actual bug.

The comment in the code says: `# Zero-message behaviour (correct — do not "fix")` — the explicit warning exists because this pattern looks like a bug to anyone who hasn't thought it through.

### Q1 — Why is `conv1`'s output `[N, 256]` and not `[N, 32]`?

The answer lives in two parameters working together: `heads=8` and `concat=True`.

**Step by step:**

`GATConv` with `out_channels=32, heads=8, concat=True` runs the attention mechanism 8 times in parallel, independently. Each of the 8 "heads" produces its own 32-dim output for every node. Then `concat=True` tells PyG: don't average these 8 outputs — concatenate them end to end.

```
Head 1 output: [N, 32]
Head 2 output: [N, 32]
...
Head 8 output: [N, 32]
concat → [N, 8 × 32] = [N, 256]
```

So the total output dimension is `heads × out_channels = 8 × 32 = 256 = hidden_dim`.

**Why not `[N, 32]`?** That would be the output if `concat=False` (average mode). With averaging, all 8 heads collapse back to 32-dim. With `concat=True`, each head's output is preserved separately.

**Why not `[N, 27]`?** The input dimension (`in_channels=27`) has no relationship to the output dimension. The GATConv projection matrix transforms 27 → 32 (per head) internally. Input and output sizes are independent choices.

**The invariant to remember:**
```
concat=True  → output = heads × out_channels
concat=False → output = out_channels  (heads averaged together)
```

Phase 1 uses `concat=True` → `8 × 32 = 256`. Phase 2 (v8.1) uses `concat=True` → `4 × 64 = 256`. Phase 3 uses `concat=False, heads=1` → `1 × 256 = 256`. All three phases output exactly 256. That's what makes the residual connections and JK aggregation possible — every phase outputs the same shape.

### Q2 — Why `w_nk + 1e-8` before `.log()`, and what does `.detach()` break?

**The `+1e-8` guard:**

Shannon entropy is `H = -Σ pᵢ × log(pᵢ)`. When any weight `pᵢ = 0`, you need `log(0)`, which is negative infinity (`-∞`). In floating point, `log(0.0) = -inf`. Multiplying `0 × (-inf)` produces `NaN` in IEEE arithmetic (not 0, as mathematics would say). That NaN propagates through the loss and corrupts the entire training step — gradients become NaN, weights become NaN, training collapses.

The fix: add a tiny constant before taking the log. `w_nk + 1e-8` ensures the argument is always at least `1e-8 > 0`. `log(1e-8) = -18.4` — large but finite, not NaN. The impact on actual entropy values is negligible (a weight of exactly zero is rare at training time and mathematically would contribute zero to the entropy anyway).

**What `.detach()` breaks:**

Without `.detach()`, the entropy value carries PyTorch's autograd computation graph — the chain of operations that connects it back to `self.attn`'s weights (`w` matrix). When the trainer does `loss = main_loss - λ * jk_entropy` and calls `loss.backward()`, gradients flow backward through the entropy calculation and update `self.attn`.

The direction: `loss.backward()` wants to increase the total loss. Since entropy appears as `-λ * jk_entropy`, increasing the loss means *decreasing* entropy — but that's wrong. The trainer actually adds it as `loss - λ * jk_entropy` to encourage high entropy, so minimizing this total loss means *maximizing* entropy. The gradient on `self.attn` pushes it toward weights that produce more uniform phase distributions.

With `.detach()`: `jk_entropy` becomes a constant float from PyTorch's perspective. No gradient flows back through it. `self.attn` receives zero gradient from the entropy term — the regularizer is completely disconnected from optimization. The model can collapse its attention to one phase and nothing stops it. The `last_weight_stds` diagnostic would show `std ≈ 0` (all nodes routed to one phase) and the model would be effectively using only one phase, discarding two-thirds of its architecture.

This is the class of bug that's hardest to catch: training runs, loss decreases, metrics look reasonable — but the model is silently ignoring two of its three phases.
