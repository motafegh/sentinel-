## Gap-Fill — Warm-Up W1–W5

**W1 — `conv1` output shape** `[Mechanism]`

The answer is `[N, 256]`, and the key is `concat=True`.

With `concat=True`, GATConv runs `heads=8` attention functions in parallel and **concatenates** (not averages) their outputs end to end. Each head produces `out_channels=32` dims → `8 × 32 = 256`. The result is always `heads × out_channels` when `concat=True`.

`[N, 32]` would only be the output with `concat=False` (averaging the 8 heads back to 32) or with `heads=1`. The input dimension `in_channels=27` has no bearing on output shape — the internal projection matrix transforms 27 → 32 per head before concatenation.

---

**W2 — `register_buffer` vs trainable parameter** `[Portable🔵]`

Two concrete consequences:

1. **Gradient / optimizer:** Parameters are in `model.parameters()` → the optimizer updates them on every backward pass. Buffers are NOT in `model.parameters()` → optimizer ignores them, zero gradient ever flows to them.

2. **Device and persistence:** Both survive `.to(device)` (move to GPU) and `state_dict()` (save/load checkpoint). This is the thing people miss — buffers ARE saved and loaded, they just aren't trained. A buffer is persistent diagnostic or state information that needs to travel with the model but must never be touched by gradient descent.

---

**W3 — BUG-R7-2: why 27 fixes what 11 couldn't** `[Pattern]`

The 11-dim encoding stored node type as `float(type_id) / 12.0` — a continuous scalar. GATConv treats this as just another number in the feature vector. It can learn "higher values tend to be more important" but it cannot learn "type 3 and type 9 are categorically distinct things that need completely different representations." The gap between `0.25` and `0.33` is just a numeric distance, not a semantic distinction.

The 27-dim fix gives each of the 13 node types its own **16-dim learned representation vector**. After training, FUNCTION nodes and CFG_CALL nodes can be as far apart in embedding space as the network needs them to be — the distance is not constrained by a scalar encoding. GATConv now sees genuine categorical structure.

---

**W4 — `.detach()` on JK entropy** `[Portable🔵]`

Without `.detach()`: the entropy scalar carries PyTorch's autograd graph connecting it back to `self.attn`'s weight matrix. When the trainer computes `loss = main_loss - λ × jk_entropy` and calls `loss.backward()`, gradients flow backward through the entropy computation and update `self.attn`. The gradient direction pushes toward more uniform phase weights (higher entropy), actively discouraging collapsed attention.

With `.detach()`: `jk_entropy` is severed from the computation graph — it becomes a floating constant as far as autograd is concerned. No gradient reaches `self.attn` from this term. The model can collapse all attention onto one phase, ignore the other two, and the regularizer does absolutely nothing to stop it. Training still runs, loss still decreases, there is no crash — it's a silent architectural neutering.

---

**W5 — `input_proj` uses `bias=False`** `[Mechanism]`

The combined output at the end of the IMP-G2 skip is:
```
result = GATConv(x) + input_proj(x)
```

`GATConv` already has its own internal bias term. If `input_proj` also had `bias=True`, the effective combined bias in the output would be `bias_conv + bias_proj` — two separate learned constants that the optimizer adjusts independently but whose effect is identical to a single constant. You'd be spending parameters to learn the same thing twice. `bias=False` removes that redundancy at zero cost to capacity.

## Gap-Fill — Challenge Questions Q1–Q6

---

**Q1 — Phase 2 vs Phase 3 output shape** `[Mechanism]`

Both produce `[N, 256]`:
- Phase 2: `out_channels=64, heads=4, concat=True` → `4 × 64 = 256` → `[N, 256]`
- Phase 3: `out_channels=256, heads=1, concat=False` → `1 × 256 = 256` → `[N, 256]`

With `heads=1`, `concat=True` and `concat=False` produce identical results (there is nothing to concatenate vs average when there's only one head). `concat=False` in Phase 3 is a convention choice, not a functional one. This shape consistency across all three phases (`[N, 256]` always) is what makes JK aggregation possible — you can't stack tensors of different shapes.

---

**Q2 — Why 3 separate Phase 2 layers with distinct edge subsets?** `[Pattern]`

If you ran one layer on all Phase 2 edge types simultaneously, the attention mechanism would have to learn "CONTROL_FLOW relevance" and "CALL_ENTRY relevance" and "DEF_USE relevance" in one combined weight matrix — they compete. A node that's a callee entry point wants to heavily weight its CALL_ENTRY predecessors; a node that's a state-writing CFG_WRITE wants to heavily weight CONTROL_FLOW predecessors. One attention function can't cleanly serve both.

The three-layer structure separates the learning problems:
- conv3 learns purely on CF edges — embeddings become enriched with intra-function order
- conv3b receives those CF-enriched embeddings and learns on ICFG edges — cross-function structure layered on top
- conv3c receives doubly-enriched embeddings and integrates all signals

Each layer builds on what the previous one established. This is the IMP-G1 insight: ordered, non-overlapping subsets accumulate context more effectively than one pass over everything.

---

**Q3 — CEI sequence and why 3 hops are needed** `[Mechanism]`

CEI (Call-Entry-ICFG) is the 4-node reentrancy fingerprint: `ENTRY → CALL → TMP → WRITE`.

- ENTRY: the function begins executing
- CALL: an external contract is called (`call.value()`, `transfer()`)
- TMP: intermediate computation (optional but common)
- WRITE: contract state is updated (`balances[msg.sender] -= amount`)

The vulnerability: the external call happens **before** the state write. If CALL happens at function entry and WRITE happens 3 control-flow hops later:

- **2 hops** from the WRITE node sees: `TMP → WRITE` and `CALL → TMP → WRITE` (2 predecessors back). Catches contracts where the call is 2 steps before the write.
- **3 hops** additionally sees `ENTRY → CALL → TMP → WRITE`. Catches contracts that immediately call an external function at the very start of the function body, before any state guard — the most dangerous pattern, since there's no guard at all between function entry and the external call.

Without conv3c's third hop, those contracts (call right at ENTRY) were passing through Phase 2 without the ENTRY node's context reaching the WRITE node. BUG-H1 (the comment in the code) documents this specific missing case.

---

**Q4 — `nn.ModuleList` vs plain Python list** `[Portable🔵]`

Three concrete things break with a plain list:

1. **`.to(device)` / `.cuda()`**: PyTorch walks the module tree to move parameters and buffers to the target device. Plain lists are invisible to this walk — the LayerNorm parameters stay on CPU while everything else moves to GPU. First forward pass crashes with a device mismatch.

2. **`state_dict()` / `load_state_dict()`**: Checkpoint saving enumerates all registered submodules. Plain list items are not registered → their `weight` and `bias` parameters are not saved → loading a checkpoint restores a model missing its LayerNorm parameters (randomly re-initialized).

3. **`model.parameters()` / optimizer**: The optimizer's parameter groups are populated from `model.parameters()`. LayerNorms in a plain list are never yielded → their γ and β are never updated by the optimizer → they stay at their initialization values (γ=1, β=0) for the entire training run, no normalization learning happens.

All three failures are silent until they catastrophically surface. `nn.ModuleList` (and `nn.ModuleDict`) are the solution for any collection of modules.

---

**Q5 — Why LayerNorm before JK softmax is necessary** `[Portable🔵]`

The JK scorer is `self.attn = nn.Linear(256, 1)`. It maps a 256-dim embedding to a scalar score. The score for any phase is determined by how well the embedding's feature values "project" onto the learned weight vector.

Now consider: Phase 1 runs 2 conv layers (each potentially scaling magnitudes up). Phase 2 runs 3. Phase 3 runs 3 more. Without normalization, Phase 1's embeddings may have significantly higher L2 norm than Phase 2 or 3 — not because Phase 1 is more informative, but because the random initialization of the first two layers happened to produce larger activations.

The JK softmax then sees larger raw scores for Phase 1 embeddings and routes most nodes there — not because Phase 1 is the right phase for them, but because it's louder. The model appears to learn routing but it's just learning to track magnitude, which is irrelevant to the actual task.

LayerNorm zero-centers and unit-normalizes each embedding before the scorer sees it. All three phases enter the JK competition with equal magnitude. Now the learned `self.attn` weight vector actually has to find meaningful structural differences between phases to route nodes correctly.

The portable pattern: **normalize before any operation that compares representations from different processing paths** — whether it's JK attention, cross-attention, or feature concatenation before a classifier.

---

**Q6 — Stale `_param_dtype` after `model.half()`** `[Mechanism]`

After `model.half()`: all parameters and buffers are now `float16`. `_param_dtype` still holds `torch.float32` (stale).

Forward pass check: `if x.dtype != self._param_dtype`. Input `x` arrives as `float32` (from the data pipeline). `self._param_dtype` says `float32`. Condition is `False` → **no cast happens**. `x` remains `float32`.

Now `x` (float32) flows into `GATConv` whose weight matrices are `float16`. PyTorch behavior depends on version and backend:
- On CPU: implicit upcasting to float32 — the half-precision model silently runs at full precision, wasting memory and gaining nothing from the cast
- On CUDA: runtime error — mixed-precision matmul is not supported without explicit `autocast`

Either way it's a failure: the model doesn't work correctly in half precision because the input dtype guard is frozen at the wrong value. The fix is calling `refresh_dtype_cache()` after any dtype cast — but nothing enforces that, which is why A3 flags it as a design issue.
