## Gap-Fill — Q1: Why `add_self_loops=False` in Phase 2?

**The core concept: self-loops fight directionality.**

In a Graph Attention Network (GAT), when computing a new embedding for node A, the mechanism aggregates messages from A's *neighbors*. A self-loop makes node A one of its own neighbors — so A's own current embedding is included in the aggregation.

In Phase 1 this is fine. Why? Because Phase 1 is doing **structural aggregation** — "what do my neighbors look like?" Self-inclusion is appropriate: a FUNCTION node should incorporate its own features when building a richer embedding. Direction doesn't matter for structural questions.

In Phase 2 this breaks everything. Why? Because Phase 2 is answering a **directional question**: *"what executes before me?"*

Here's the reentrancy pattern:
```
call.value()  →  state_write     (VULNERABLE: external call before state update)
state_write   →  call.value()    (SAFE:       state update before external call)
```

A CONTROL_FLOW edge `A → B` means "A executes before B." When the GNN aggregates at node B, it should see **A's embedding** (predecessor). That predecessor context tells B: "an external call just happened before me." That signal is the vulnerability fingerprint.

Now add a self-loop at B. GAT computes attention weights over all neighbors **including B itself**. In attention, the self-weight can dominate — the "loudest" signal is often the node's own current embedding. When that happens, B stops listening to its predecessors and just reinforces itself. The directional "A before B" signal is diluted or lost entirely.

> ⚠️ **CRITICAL** — If you added `add_self_loops=True` to Phase 2, the model could no longer reliably detect execution order, making reentrancy detection structurally impossible. The comment `# CRITICAL` in the code is not stylistic emphasis — it is a correctness invariant.

**What would break if flipped?**
If Phase 1 had `add_self_loops=False`: raw features wouldn't be incorporated into a node's own update — each node only hears from its structural neighbors (callers, contained nodes). The `input_proj` skip (IMP-G2) partially compensates, but you'd lose the structural self-context.

If Phase 2 had `add_self_loops=True`: execution order signal degrades silently. Training loss would still converge, but the model would be learning a weaker signal. You'd likely see lower recall on reentrancy patterns without knowing why.

## Gap-Fill — Q2: Why REVERSE_CONTAINS isn't stored on disk?

Your intuition is correct on both points. Let me make it precise.

**What you got right:**
- It's derived at runtime, not from new information
- Storing it would cause edge redundancy

**The precise mechanism:**

REVERSE_CONTAINS is computed in the forward pass with a single operation:

```python
# Learning mode: Master the detail
contains_mask = edge_attr == _CONTAINS       # boolean mask: which edges are type 5
reverse_ei = edge_index[:, contains_mask].flip(0)   # swap row 0 and row 1
reverse_ea = e[contains_mask]                # same edge embeddings
```

`flip(0)` on a `[2, E]` tensor swaps rows: `[sources, targets]` → `[targets, sources]`. That's the entire runtime cost. No new information — just direction reversal.

**Why not store on disk — the three real reasons:**

1. **Redundancy:** Every CONTAINS edge `F → C` (function → CFG node) would have a twin REVERSE_CONTAINS edge `C → F` in the .pt file. The file would be ~10–15% larger for every contract. At 68,000 contracts, that matters.

2. **The key insight — it's not a separate relationship.** CONTAINS means "function F contains CFG node C." REVERSE_CONTAINS means "CFG node C belongs to function F." These are the **same fact** traversed in opposite directions. Storing it twice would be semantically incorrect — there's one containment relationship, not two. The code makes this explicit: REVERSE_CONTAINS has no independent meaning apart from CONTAINS reversed.

3. **Sync correctness:** If you stored both and ever updated one without the other (e.g., if graph extraction code changed), they'd silently diverge. Deriving REVERSE_CONTAINS at runtime from CONTAINS at runtime *by definition* keeps them in sync.

> ⚠️ **CRITICAL** — REVERSE_CONTAINS (type 7) is listed in the `EDGE_TYPES` dict and the `nn.Embedding` covers all 11 types. But the forward pass only creates type-7 edges at runtime. If you ever find a .pt file that has stored type-7 edges, the graph extractor has a bug — those edges will be double-counted in Phase 3.
