**Q1. Guard 2 — what happens without it**

`edge_attr=None` → code reaches the `else` branch:
```python
cfg_mask = torch.zeros(n_edges, dtype=torch.bool, device=...)
```
All False. `phase2_ei = edge_index[:, cfg_mask]` → shape `[2, 0]` — an empty edge tensor.

Phase 2: `conv3(x, empty_ei, empty_ea)`. GATConv on an empty `edge_index` returns a zero tensor `[N, 256]`. Residual: `x = x + dropout(zeros) = x`. Same for conv3b and conv3c. Phase 2 is a complete no-op.

Phase 3: `contains_mask` is also all-False → `fwd_contains_ei = [2, 0]`. Same no-op chain through conv4/4b/4c.

The model trains on Phase 1 structural signal only — structurally identical to a v5 model before any of the CFG/ICFG improvements were added. No error, no NaN, loss decreases normally. The experiment appears to run correctly. You'd only notice if you compared validation metrics to a correctly configured run.

**Q2. `phase2_ea` instead of `phase2_raw` — shape failure**

`phase2_ea` has shape `[E_cfg, 64]` — a float tensor with 64 columns.

`phase2_ea == _CONTROL_FLOW` (where `_CONTROL_FLOW = 6`) compares each of the 64 floats to the integer 6. Result shape: `[E_cfg, 64]` bool — a 2D mask.

Then `phase2_ei[:, _cf_mask]` tries to use this 2D boolean mask to index columns of `phase2_ei [2, E_cfg]`. PyTorch would either raise a dimension mismatch error or produce a completely wrong result depending on version — column indexing with a 2D boolean mask is not the intended operation. You'd get either a crash or a mangled `[2, X]` tensor where X is some unintended count. Either way, wrong edges go to conv3.

The raw integer IDs in `phase2_raw [E_cfg]` produce a 1D boolean mask, which correctly selects columns.

---

**Q3. `.flip(0)` on `[2, E]`**

🔵 **Portable recall:** A `[2, E]` COO tensor:
```
row 0: [src_0, src_1, ..., src_E]   ← source nodes
row 1: [dst_0, dst_1, ..., dst_E]   ← destination nodes
```

`.flip(0)` reverses along dimension 0 — swaps the two rows:
```
row 0: [dst_0, dst_1, ..., dst_E]   ← was destination, now source
row 1: [src_0, src_1, ..., src_E]   ← was source, now destination
```

Every `(src→dst)` edge becomes `(dst→src)`. No data is copied — `.flip()` returns a view with a negative stride on dim 0, so it's O(1).

Where else you'd use this: any time you need a bidirectional graph from a directed one without re-building it. Common patterns — (1) building symmetric adjacency for undirected GNNs from directed input, (2) constructing reverse residual paths in encoder-decoder GNNs, (3) propagating gradient signal "upward" in hierarchical graphs (exactly what Phase 3 does). The pattern is widespread in PyG codebases.

---

**Q4. Why `validate_graph_integrity` defaults to False**

`edge_index.max()` scans all E values on the GPU or CPU. For a batched forward pass with B=32 graphs averaging 10K edges: 320K comparisons, synchronization point that stalls the GPU pipeline, measurable throughput loss per step across millions of training steps.

The right production alternative (called out in the comment): move the check to `DualPathDataset.__getitem__`. That runs once when the `.pt` file is first loaded, not once per training step. Corrupted files are caught at data loading time, logged with the specific filename, and skipped — without adding overhead to every forward call.

The flag exists so you can enable it during debugging and integration tests, where catching the error with a clear message is worth the overhead.

---

**Q5. `return_phase2_embs` — why gradients must be attached**

The `aux_phase2` head in `sentinel_model.py` computes:
```
aux_loss = cross_entropy(aux_phase2_head(_phase2_x), labels)
total_loss = main_loss + λ * aux_loss
total_loss.backward()
```

`backward()` walks the computation graph from `aux_loss` through `aux_phase2_head` → through `_phase2_x` → through `conv3c → conv3b → conv3` → all the way back to the input. This is the *only* direct gradient path to Phase 2 conv layers.

The main loss flows through JK → Phase 3 → Phase 2 (via residual connections), but that path is long and gradients decay. The aux head provides a short, direct path specifically to ensure Phase 2 gets strong gradient signal during training. This is BUG-R7-1's fix: Phase 2 was gradient-starved before v8.1.

If `.detach()` were used: `_phase2_x.detach()` breaks the computation graph at that point. `aux_loss.backward()` only updates the `aux_phase2_head` MLP parameters — the gradient stops at the detach boundary and never reaches conv3/3b/3c. The aux head becomes a useless classifier trained on frozen Phase 2 features. Phase 2 gradient starvation persists exactly as in v8.0.

---

**Q6. Why `rev_contains_ea ≠ fwd_contains_ea`**

`fwd_contains_ea = e[contains_mask]` — these are the edge embedding vectors for CONTAINS (type 5). Each row is `edge_embedding.weight[5]` — the learned 64-dim representation for "parent contains child."

If you used these same vectors for the reversed edges, every reversed edge `(CFG→FUNCTION)` would carry the type-5 "parent contains child" signal — the *wrong direction semantics*. The network's edge-conditioned attention in conv4 would receive the same embedding regardless of whether it's doing an upward or downward pass.

IMP-G3's point is precisely that upward aggregation (Phase 2 signal rising from CFG to FUNCTION) and downward distribution (FUNCTION broadcasting to CFG children) are semantically different operations. `rev_contains_ea` must use type-7 embeddings (`edge_embedding.weight[7]`) — a separate learned vector trained specifically on "child belongs to parent" signal. Only then can the network's attention heads distinguish the two directions.

### Big Picture

The code is repetitive by design — each phase follows the same `conv → relu → x + dropout(x2)` residual pattern — but the *why* behind each step and the subtle differences between phases are what matter.

Chunk 5:    Prepend type_emb → [N, 27]

### Section 1 — `_live` and `_intermediates` Setup

```python
# Learning mode: Master the mechanism

_live: list[torch.Tensor] = []      # ← MASTER: collects gradient-attached phase outputs for JK
_intermediates: dict      = {}      # ← UNDERSTAND: diagnostic only — detached clones, not used in backward
```

⚠️ **CRITICAL** — `_live` must never contain detached tensors. The JK attention parameters (`self.jk.attn = nn.Linear(256, 1)`) receive gradients *only* through the tensors in `_live`. If any phase output were detached before being appended, JK's parameters would receive zero gradient for that phase — the attention weights couldn't learn to route away from it.

The `_intermediates` dict is the opposite contract: always detached. It exists for visualization and debugging — you can inspect per-phase embeddings post-training without inadvertently keeping large intermediate tensors alive in the computation graph.

---

### Section 2 — Type Embedding Prepend (BUG-R7-2 at Runtime)

```python
# Learning mode: Master the mechanism

# Recover integer type ID from the continuous float stored in feat[0]
_type_int = (x[:, 0].float() * _NUM_NODE_TYPES).round().long().clamp(0, _NUM_NODE_TYPES - 1)
#            ↑ feat[0] was stored as float(type_id)/12.0        ↑ clamp guards float rounding edge cases

x_init = torch.cat([x, self.type_embedding(_type_int).to(x.dtype)], dim=1)  # [N, 27]
#                      ↑ still [N, 11]    ↑ [N, 16] — 16-dim learned type vector per node
```

**The recovery math:**
`feat[0]` was stored as `float(type_id) / 12.0` (old BUG-R7-2 encoding). To get back the integer: multiply by `_NUM_NODE_TYPES=13`, round (undoes floating-point imprecision), cast to `long`. Example: type 5 → stored as `5/12.0 = 0.4167` → `0.4167 * 13 = 5.417` → `.round() = 5` → `.long() = 5`. The `.clamp(0, 12)` handles any edge case where rounding produces 13.

**Why not fix the `.pt` files and re-extract?**
Re-extracting all graphs is expensive (hours of compute). The fix is applied at runtime — a cheap 3-operation decode. The 16-dim type vector then gives GATConv something it can actually learn categorical structure from, replacing the useless `0.4167` scalar.

**Why `.to(x.dtype)`?**
`self.type_embedding` parameters are in `_param_dtype` (likely float32). `x` was already cast to `_param_dtype` in the dtype guard. But being explicit about the cast prevents subtle BF16/FP32 mismatches at the concat boundary.

---

### Section 3 — Phase 1: Structural Aggregation (Layers 1+2)

```python
# Learning mode: Master the mechanism

# IMP-G2: input_proj skip — save x_init before conv1 overwrites x
x_skip = self.input_proj(x_init.to(self._param_dtype)).to(x_init.dtype)  # [N, 256]  ← MASTER: 27→256 linear, no bias
x  = self.conv1(x_init, struct_ei, struct_ea)   # [N, 27] → [N, 256]    ← MASTER: 8 heads × 32 dims, concat
x  = self.relu(x + x_skip)                      # ← MASTER: skip added BEFORE relu — IMP-G2
x  = self.dropout(x)

# Layer 2: residual
x2 = self.conv2(x, struct_ei, struct_ea)         # [N, 256] → [N, 256]  ← MASTER: 8 heads × 32
x2 = self.relu(x2)
x  = x + self.dropout(x2)                        # ← MASTER🔵: residual — identity + branch

x  = self.phase_norm[0](x)                       # ← MASTER: LayerNorm before JK collection
_live.append(x)
_intermediates["after_phase1"] = x.detach().clone()
```

**The IMP-G2 skip in detail:**

```
x_init [N,27] ──→ conv1 ──→ [N,256] ─┐
         │                            ├─ + ──→ relu ──→ dropout ──→ x [N,256]
         └──→ input_proj ──→ [N,256] ─┘
                (linear, no GAT attention)
```

The skip runs in parallel to conv1, not after it. This means: even if conv1's attention weights initialize to near-uniform (averaging all neighbor features, losing the raw signal), the skip preserves a direct linear transformation of `x_init`. After the add, the relu fires on `conv1_output + linear(x_init)`. The raw 27-dim feature information is always present in the output, independent of whether attention learned anything useful yet.

🔵 **Portable recall — residual connections:** `x = x + f(x)` (or `x = f(x) + projection(x)` when dimensions change) is the foundational building block of deep networks. The identity path guarantees that gradients flow directly back through `x` without passing through the conv operation — solving the vanishing gradient problem in deep networks. It also means the conv only needs to learn the *delta* (what to add) rather than the full transformation. Used universally: ResNets, Transformers, GNNs. The exact same pattern appears in Layer 2: `x = x + dropout(x2)`.

**Layer 2 residual — why `dropout` on the branch, not on the identity?**
`x = x + self.dropout(x2)` — only the conv output is dropped, never the identity path. If dropout were applied to `x` instead: `x = self.dropout(x) + x2`, then at training time with 20% dropout, some of the accumulated Phase 1 signal in `x` would be zeroed out at each step. The identity is the "safe" path — it carries the accumulated history of all previous layers. Dropping the identity would destabilize training. Only the new branch signal is regularized.

---

### Section 4 — Phase 2: CFG + ICFG (Layers 3+4+5)

```python
# Learning mode: Master the mechanism

# Layer 3 (conv3): CONTROL_FLOW only
x2 = self.conv3(x, cf_only_ei, cf_only_ea)        # [N,256] → [N,256]  ← MASTER: 4 heads × 64
x2 = self.relu(x2)
x  = x + self.dropout(x2)

# Layer 4 (conv3b): CALL_ENTRY + RETURN_TO only
x2 = self.conv3b(x, icfg_only_ei, icfg_only_ea)   # [N,256] → [N,256]  ← MASTER: 4 heads × 64
x2 = self.relu(x2)
x  = x + self.dropout(x2)

# Layer 5 (conv3c): joint CF + ICFG
x2 = self.conv3c(x, phase2_ei, phase2_ea)          # ← MASTER: all Phase 2 edges; x already enriched by 3+4
x2 = self.relu(x2)
x  = x + self.dropout(x2)

x  = self.phase_norm[1](x)                         # ← MASTER: LayerNorm after all 3 Phase 2 layers
_live.append(x)
_intermediates["after_phase2"] = x.detach().clone()
_phase2_x = x                                       # ← MASTER: gradient-attached ref for aux loss
```

**The layered enrichment strategy:**

After Layer 3: each node's embedding encodes "which nodes execute immediately before/after me within my function."

After Layer 4: each node additionally encodes "which functions call me, and where execution returns after I return." These two pieces of information now coexist in the same embedding vector via the residual additions.

Layer 5 (conv3c) then runs with `phase2_ei` — all CF + ICFG edges — on embeddings that *already contain both signals*. A 2-hop pattern like "A calls B, and A's CONTROL_FLOW predecessor is C" can now be captured in a single Layer 5 attention step, because C's effect is already baked into A's embedding from Layer 3.

This is why the integration layer comes last: it leverages the intermediate representations built by Layers 3 and 4. Running it first (before the others) would only see the Phase 1 output, missing the specialized enrichment.

**`_phase2_x = x`** — this line assigns the gradient-attached Phase 2 tensor right after `phase_norm[1]`. The normalized version (not the raw pre-norm) is what gets passed to the auxiliary head. LayerNorm doesn't break gradients — it's a differentiable operation — so the gradient path is intact.

---

### Section 5 — Phase 3: Bidirectional CONTAINS (Layers 6+7+8)

```python
# Learning mode: Master the mechanism

# Layer 6 (conv4): CFG→FUNCTION up (REVERSE_CONTAINS)
x2 = self.conv4(x, rev_contains_ei, rev_contains_ea)   # ← MASTER: reversed edges; type-7 embs
x2 = self.relu(x2)
x  = x + self.dropout(x2)

# Layer 7 (conv4b): second up hop
x2 = self.conv4b(x, rev_contains_ei, rev_contains_ea)  # ← MASTER: same reversed edges, second pass
x2 = self.relu(x2)
x  = x + self.dropout(x2)

# Layer 8 (conv4c): FUNCTION→CFG down (CONTAINS, type-5 embs — IMP-G3)
x2 = self.conv4c(x, fwd_contains_ei, fwd_contains_ea)  # ← MASTER: original CONTAINS direction, type-5 embs
x2 = self.relu(x2)
x  = x + self.dropout(x2)

x  = self.phase_norm[2](x)                              # ← MASTER: LayerNorm after full bidirectional Phase 3
_live.append(x)
_intermediates["after_phase3"] = x.detach().clone()
```

**Why two upward passes (Layers 6 and 7)?**

One upward pass: each FUNCTION node aggregates the Phase 2 embeddings of its *direct* CFG children. This gives functions 1-hop CEI context.

Two upward passes: after Layer 6, FUNCTION nodes carry aggregated CFG child context. In Layer 7, when a CFG node aggregates again from its parent function (wait — the edges are still reversed: CFG→FUNCTION), it sees the *function's updated embedding*, which already incorporates Layer 6's aggregation. In graphs where functions call other functions (nested call stacks), the two hops allow a callee-function's CEI pattern to propagate into a caller-function's embedding: `CFG_in_callee → FUNCTION_callee → ...`. A single hop would only reach 1 level of the call hierarchy.

**The downward pass (Layer 8) — the "broadcast":**
After two upward passes, FUNCTION nodes now carry rich embeddings: their own Phase 1 structural context + aggregated Phase 2 CEI patterns from all their CFG children. Layer 8 sends this *back down* to CFG nodes. After Layer 8, each CFG node's embedding includes:
- Its own Phase 1 structural context
- Its own Phase 2 execution-order context
- Its parent function's aggregated view of the entire function's vulnerability pattern

This is why every node has "uniform depth" before being passed to `CrossAttentionFusion` — they've all gone through the same 8 layers and carry context at all three levels.

### Section 6 — JK Aggregation

```python
# Learning mode: Master the mechanism

if self.use_jk and self.jk is not None:
    x, _jk_entropy = self.jk(_live)           # ← MASTER: _live = [phase1_x, phase2_x, phase3_x]
else:
    _jk_entropy = x.new_zeros(1).squeeze()    # ← UNDERSTAND: use_jk=False → Phase 3 output only; entropy = 0
```

`_live` contains three tensors, each `[N, 256]`, each LayerNorm-normalized:
- `_live[0]` = Phase 1 output: pure structural signal
- `_live[1]` = Phase 2 output: execution-order signal (also in `_phase2_x`)
- `_live[2]` = Phase 3 output: bidirectionally enriched signal

`self.jk(_live)` calls `_JKAttention.forward(_live)`. Recall from Chunk 2:
1. Stack → `[N, 3, 256]`
2. Score via `Linear(256→1)` → `[N, 3, 1]`
3. Softmax across dim=1 → `[N, 3, 1]` (sum-to-1 per node)
4. Squeeze → `[N, 3]`
5. Weighted sum: `(scores.unsqueeze(-1) * stacked).sum(dim=1)` → `[N, 256]`
6. Shannon entropy of scores → scalar regularizer

`use_jk=False` path: `x` is already Phase 3's output (last assignment). Return it as-is. This is the v5.0 behavior — preserved for checkpoint compatibility with pre-JK models.

### Section 7 — Return Paths

```python
# Learning mode: Master the mechanism

if return_intermediates:
    return x, batch, _jk_entropy, _intermediates   # ← UNDERSTAND: diagnostic; _intermediates has detached tensors
if return_phase2_embs:
    return x, batch, _jk_entropy, _phase2_x        # ← MASTER: _phase2_x has gradients — for aux_phase2 head
return x, batch, _jk_entropy                       # ← MASTER: standard training path
```

**Standard return — `[N, 256]`, `[N]`, scalar:**
- `x`: final node embeddings. Every node in every graph in the batch is in this single `[N, 256]` matrix (N = total nodes across all graphs in the batch).
- `batch`: `[N]` tensor of graph indices. Node `i` belongs to graph `batch[i]`. Used downstream in `sentinel_model.py` for graph-level pooling (mean/max pool over nodes belonging to the same graph index).
- `_jk_entropy`: scalar added to the main loss as a regularizer. Encourages the JK weights to be non-degenerate (not concentrated on one phase).

**Why return `batch`?**
`batch` enters `forward()` as a parameter and is returned unchanged. It could be accessed by the caller who already has it — but passing it through makes `GNNEncoder.forward()` self-contained as a module: the caller just unpacks the tuple and has everything needed for the next stage without keeping track of the original batch tensor separately.

### Complete Forward Pass — Unified Data Flow

```
Input: x [N,11], edge_index [2,E], batch [N], edge_attr [E]
         │
    ─── Guards ─── (schema, use_edge_attr, integrity)
    ─── Dtype cast ─── (x → _param_dtype)
    ─── Edge embedding ─── (edge_attr → e [E,64])
    ─── Mask split ─── (struct / cfg / contains)
    ─── IMP-G1 sub-masks ─── (cf_only / icfg_only)
    ─── Reverse edge construction ─── (flip(0) + type-7 emb)
         │
    x_init = cat([x, type_emb(_type_int)]) [N,27]   ← BUG-R7-2 fix
         │
    Phase 1 (struct_ei):
      conv1 [N,27→256] + input_proj skip (IMP-G2)
      conv2 [N,256→256] + residual
      LayerNorm → _live[0]
         │
    Phase 2 (cf/icfg/phase2_ei):
      conv3  [CF only]     + residual
      conv3b [ICFG only]   + residual
      conv3c [CF+ICFG]     + residual   ← integration
      LayerNorm → _live[1], _phase2_x
         │
    Phase 3 (rev/fwd contains_ei):
      conv4  [rev, up]     + residual
      conv4b [rev, up ×2]  + residual
      conv4c [fwd, down]   + residual   ← IMP-G3 downward broadcast
      LayerNorm → _live[2]
         │
    JK(_live) → x [N,256], entropy (scalar)
         │
    return x [N,256], batch [N], entropy
```

---

### 3 Things to Lock In

1. **`_live` must never be detached.** JK's attention parameters only receive gradients through these tensors. `_intermediates` is the detached diagnostic copy — keep the contracts separate.

2. **The integration layer (conv3c) runs last in Phase 2 deliberately.** It operates on embeddings already enriched by Layers 3 and 4, enabling multi-type multi-hop patterns in a single attention step. Order matters — running it first would miss the specialized enrichment.

3. **`_phase2_x = x` is assigned after `phase_norm[1]` on the gradient-attached tensor.** It's the normalized Phase 2 output with full gradient connectivity back to conv3/3b/3c. This is the sole direct gradient path to Phase 2 layers — the BUG-R7-1 fix.

## Chunk 5 Gap-Fill

---

**Q1. Type ID recovery for `feat[0] = 0.3077`**

`0.3077 * 13 = 3.9997` → `.round() = 4` → `.long() = 4`.

Type ID 4 = `INHERITS`. The math correctly reverses the `float(4)/13.0 ≈ 0.3077` encoding.

`.clamp(0, 12)` guards one specific failure: floating-point arithmetic on `float(12)/13.0 = 0.9231` → `0.9231 * 13 = 12.0003` → `.round() = 12`. Safe. But `float(12)/12.0` (old v6 code used `/12.0` not `/13.0`) → `1.0 * 13 = 13` → `.round() = 13` → `.long() = 13`. That's out of bounds for a 13-row embedding table (valid indices: 0–12). Clamp to 12. Without it: CUDA illegal-memory-access, no useful traceback.

---

**Q2. Dropout on branch, not identity**

🔵 **Portable recall — residual dropout placement:** `x = x + dropout(branch)` is the standard. The identity path `x` carries the accumulated history of every prior layer. Dropping it would randomly zero out already-learned representations at training time, forcing the network to relearn from scratch on unlucky steps — destabilizes training and wastes the depth you built. The branch `x2` is the new delta being proposed by the current layer — that's what should be regularized.

If `x = dropout(x) + x2`: at 20% dropout probability, roughly 20% of the 256 embedding dimensions are zeroed on every forward pass. The gradient on those zeroed dimensions is also zero — the optimizer doesn't update weights connected to them on those steps. Effective learning rate becomes stochastic in an uncontrolled way. Worse, at inference time when dropout is disabled, `x` suddenly carries full magnitude — the scale of `x + x2` differs between train and eval, producing a train/eval distribution shift on top of all the conv layers.

---

**Q3. Why conv3c (integration layer) runs last**

After Layer 3 (CF only): each node's embedding encodes intra-function execution order — "which nodes precede and follow me within my function."

After Layer 4 (ICFG only): each node additionally encodes cross-function call structure — the residual add means both signals coexist in `x`.

When conv3c runs with all Phase 2 edges on this enriched `x`, its attention mechanism can compute things like: "node A has a CONTROL_FLOW edge to node B, and A's embedding already contains CALL_ENTRY context from Layer 4." A single attention step in conv3c can now capture a 2-hop CEI pattern (CF within function + CALL_ENTRY cross-function) because the multi-hop information is already pre-aggregated in the node embeddings.

If conv3c ran first (on Phase 1 output): it would only see structural context, with no execution-order enrichment yet. The joint pass would produce the same result as running CF and ICFG independently, then combining — losing the synergistic 2-hop capture. The sequential enrichment is the mechanism.

---

**Q4. Why two identical-looking upward passes produce different results**

After Layer 6 (conv4): FUNCTION nodes aggregate Phase 2 embeddings from their direct CFG children. Call the result `x_after_6`.

In Layer 7 (conv4b): the input is `x_after_6`. FUNCTION nodes now aggregate from their CFG children again — but those CFG children now carry `x_after_6` embeddings, which already contain the Layer 6 aggregation.

In a call graph where Function F1 calls Function F2: after Layer 6, F2 has aggregated its own CFG children's Phase 2 embeddings. CFG nodes *inside F2* now carry residual-updated embeddings that include F2's Layer 6 context. When conv4b runs, F1 aggregates from its own CFG children — and those children may include CALL_ENTRY nodes that connect to F2's CFG nodes. The second hop allows F1 to indirectly see F2's Phase 2 pattern.

Same edges, different node embeddings at input — that's why the results differ. The embedding `x` evolves between every layer.

---

**Q5. Why `batch` is returned**

`GNNEncoder.forward()` is designed to be a composable module — its output tuple is a self-contained "everything downstream needs." The caller (`sentinel_model.py`) does:

```python
node_embs, batch, jk_entropy = self.gnn(x, edge_index, batch, edge_attr)
# immediately uses batch for graph-level pooling:
graph_emb = global_mean_pool(node_embs, batch)
```

If `batch` were not returned, the caller would need to keep the original `batch` tensor in scope, correctly pair it with the returned `node_embs`, and ensure device/dtype consistency between them. For a module used inside `nn.DataParallel` or across async pipelines this becomes fragile — the `batch` tensor from the original forward call might be on a different device replica. Returning it from `forward()` ensures the caller always has the correctly-paired, correctly-placed version without external bookkeeping.

**Q6. LayerNorm before JK — magnitude bias**

🔵 **Portable recall — magnitude bias in learned aggregation:** Softmax over raw scores amplifies the highest-magnitude inputs regardless of their semantic content. If Phase 1 output has L2 norm ~4.0 and Phase 2 output has L2 norm ~1.5 (a realistic difference after 2 vs 5 conv layers), the `Linear(256→1)` scorer will output a larger value for Phase 1 by default. Softmax → Phase 1 gets weight ~0.8, Phases 2+3 share ~0.2. The JK mechanism degenerates to "always prefer Phase 1" — not because Phase 1 is more useful, but because its numbers are bigger.

LayerNorm normalizes each `[256]` vector to have mean≈0, variance≈1 *before* scoring. Now all three phase outputs enter `Linear(256→1)` with the same magnitude scale. Softmax weights are determined by the *direction* of the vector in 256-dim space — i.e., learned content — not by its length. JK can now genuinely learn "this node type relies on Phase 2 execution-order context more than Phase 1 structure" and route accordingly.

Without LayerNorm: JK weights encode "which phase has higher magnitude" — a function of network depth, not vulnerability signal. The attention is learning the wrong thing, and Phase 1 effectively dominates the entire JK mechanism throughout training.
