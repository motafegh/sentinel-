## **Phase 5, Chunk 1: gnn_encoder.py — Module Overview, Architecture, Edge Types, Phase Design**

### **Big Picture First (P5)**

`gnn_encoder.py` solves a critical problem in Sentinel: **how to propagate vulnerability patterns through a smart contract's code structure**.

**What it does:**
- Takes a control-flow graph (CFG) representation of a Solidity smart contract (nodes = code instructions, edges = data flow + control flow)
- Runs it through a specialized Graph Neural Network (GNN) called a **Graph Attention Network** (GAT)
- Outputs learned numerical embeddings for each node that encode both **structural position** (where it sits in execution order) and **semantic context** (function-level properties trickling down)

**Why it's complex:**
Smart contracts have two interleaved challenge problems:
1. **Execution order matters.** A `call` followed by a `write` to contract state is the Reentrancy Vulnerability signature. A `write` followed by a `call` is safe. The GNN must track **directionality** and **multi-hop neighborhoods** to distinguish these.
2. **Functions are containers.** CFG nodes (instructions) live *inside* function definitions. Vulnerability patterns span the function boundary — a `withdraw()` function's semantics propagate into its CFG nodes. The GNN must **propagate downward** (function→CFG) and **aggregate upward** (CFG→function) symmetrically.

This file's answer: a **three-phase, eight-layer GAT** (v8 architecture) that handles both problems in sequence.

---

### **System Role**

Within Sentinel's three-eye classifier:
```
Code (Solidity) → Extractor → Graph (.pt file) → GNNEncoder → node embeddings [N, 256]
```

GNNEncoder produces the **structural eye's opinion**. Other files (transformer_encoder.py, fusion_layer.py, sentinel_model.py) take these embeddings plus token embeddings from BERT and fuse them into a final vulnerability classification.

### **Part 1: The 11 Edge Types**

⚠️ **CRITICAL** — Edge types are the **language** by which the GNN understands the graph. Each edge carries a semantic label that controls which layers process it. Confusing these = silent wrong computation.

From `graph_schema.py`:

| Type | ID | Name | Meaning | Phase(s) |
|------|----|----|---------|----------|
| 0 | CALLS | Function A calls Function B | Structural; Phase 1 |
| 1 | READS | Instruction reads a variable | Structural; Phase 1 |
| 2 | WRITES | Instruction writes a variable | Structural; Phase 1 |
| 3 | EMITS | Instruction emits an event | Structural; Phase 1 |
| 4 | INHERITS | Contract A inherits Contract B | Structural; Phase 1 |
| 5 | CONTAINS | Function contains CFG node OR Contract contains Function | Structural; Phase 1 *and* Phase 3 |
| 6 | CONTROL_FLOW | One instruction → next in execution order | Phase 2 only |
| 7 | REVERSE_CONTAINS | CFG node belongs to Function (reverse of 5) | **Runtime-only**, Phase 3 |
| 8 | CALL_ENTRY | Function call → entry node of callee | Phase 2 only |
| 9 | RETURN_TO | Return node → next node after call | Phase 2 only |
| 10 | DEF_USE | Instruction defines a value → instruction uses it | Phase 2 (data flow) |

**Key split:**
- **Types 0–5 (Structural):** Built during graph extraction and stored in .pt files. Capture the *container hierarchy* (what function contains what node) and *static dependency* (what calls what).
- **Types 6–10 (Directional flow):** Also stored. Capture *execution order* (CONTROL_FLOW) and *interprocedural call structure* (CALL_ENTRY, RETURN_TO, DEF_USE).
- **Type 7 (REVERSE_CONTAINS):** Generated **at runtime** in the forward pass — the reverse direction of CONTAINS. Used in Phase 3 to propagate signal upward.

### **Part 2: Three-Phase Design Rationale (P5, P11)**

The 8-layer architecture doesn't process all edge types equally. Instead, it splits into **three phases**, each with a different job:

#### **Phase 1 — Structural Aggregation (Layers 1+2)**
**Edges:** Types 0–5 (CALLS, READS, WRITES, EMITS, INHERITS, CONTAINS)

**Goal:** Propagate function-level properties *down* into CFG nodes, and establish *structural context*.

```
Layer 1:  [11 features] → [256 dims]    (raw features → embeddings, 8 heads concat)
          + input_proj skip (IMP-G2)    ← stays ← added as a shortcut
Layer 2:  [256] → [256]                 (residual connection, 8 heads concat)
```

**IMP-G2 (Improvement #2):** When GAT (Graph Attention Network) starts training, its learned attention weights may initialize uniformly (all neighbors weighted equally). In that regime, Layer 1 might lose the raw 11-dimensional feature signal entirely through aggressive averaging. The **input_proj** layer creates a **learned skip connection** that directly feeds raw features into the first output, preventing catastrophic feature loss. This is a 11→256 linear layer that runs in parallel to the GAT conv; its output is added to conv1's output before the ReLU.

**Result after Phase 1:**
- Node embeddings are now 256-dimensional.
- Function nodes carry properties describing what they do (e.g., "this is a payable function that can transfer funds").
- CFG nodes carry those properties *inherited* from their parent function.
- Inter-function CALLS edges establish that node A in function F1 can reach node B in function F2.

#### **Phase 2 — Control Flow + ICFG (Layers 3+4+5)**
**Edges:** Types 6, 8, 9, 10 (CONTROL_FLOW, CALL_ENTRY, RETURN_TO, DEF_USE)

**Goal:** Build **execution-order context** — "what instruction can affect what other instruction"?

```
Layer 3 (conv3):   CONTROL_FLOW (type 6) only
                   → intra-function execution ordering
                   → "node A executes before node B within one function"

Layer 4 (conv3b):  CALL_ENTRY (8) + RETURN_TO (9) only
                   → cross-function call structure
                   → "node in function F1 calls function F2, execution enters F2's entry node"

Layer 5 (conv3c):  CF(6) + CALL_ENTRY(8) + RETURN_TO(9) + DEF_USE(10) joint
                   → integration layer
                   → "combines all flow signals into a unified execution-order representation"
```

⚠️ **CRITICAL — add_self_loops=False:** Phase 1 adds self-loops (every node attends to itself), which is fine for structural information. Phase 2 **must not** add self-loops. Why? Self-loops cancel the directionality signal. If node A has a self-loop in a directed CONTROL_FLOW edge, the attention mechanism can't distinguish "A → B" (A before B) from "B → A" (B before A). Removing self-loops forces the network to learn the actual direction. This is not an optimization — it's a **correctness requirement**.

**IMP-G1 (Improvement #1):** Each Phase 2 layer processes a *different subset* of edge types:
- conv3 sees CONTROL_FLOW only
- conv3b sees CALL_ENTRY + RETURN_TO only
- conv3c sees all four together

This is more flexible than processing the same set three times (older code did that). Now you can ablate individual edge types or test hypotheses like "ICFG alone is enough" by passing `phase2_edge_types=[8, 9]`.

**Result after Phase 2:**
- Node embeddings now encode "what other nodes will execute next" — the full CEI pattern.
- CEI = CALL + ENTRY + ICFG, a 3-hop pattern that captures the reentrancy signature.

#### **Phase 3 — Bidirectional CONTAINS (Layers 6+7+8)**
**Edges:** Type 5 (CONTAINS) and Type 7 (REVERSE_CONTAINS, runtime-generated)

**Goal:** Distribute Phase 2's execution-order signal *back up* to function nodes, then *back down* to CFG nodes so they're all enriched with both types of information.

```
Layer 6 (conv4):   REVERSE_CONTAINS (type 7) up
                   → CFG nodes send their Phase 2 embeddings UP to parent functions
                   → functions now aggregate their children's execution patterns

Layer 7 (conv4b):  REVERSE_CONTAINS again
                   → multi-hop: functions aggregate through multiple CFG levels
                   → captures patterns like "this function calls another that has the vulnerability"

Layer 8 (conv4c):  CONTAINS (type 5) down
                   → FUNCTION nodes now send enriched embeddings back DOWN to CFG children
                   → CFG nodes now see both their own execution order (Phase 2) and
                     their function's broader role (Phase 3 aggregation)
```

**IMP-G3 (Improvement #3):** Phase 3 uses *distinct embeddings* for the up and down directions:
- Layers 6+7 use REVERSE_CONTAINS (type 7) embeddings
- Layer 8 uses CONTAINS (type 5) embeddings

This prevents the network from conflating "propagate child signal UP" with "propagate parent signal DOWN" — they're semantically different operations.

**Special case — zero-message behavior:**
If a FUNCTION node has *no CFG children*, Phase 3 layers 6+7 will receive zero messages (nothing to aggregate up). The convolution returns zero. The residual connection adds this zero to the node's embedding: `x = x + dropout(0) = x`, which is a no-op. This is **correct and intentional**, not a bug. A function with no CFG nodes (e.g., `pure` functions that just read storage) doesn't accumulate execution order information.

**Result after Phase 3:**
- Both FUNCTION and CFG nodes carry enriched embeddings blending structural + execution-order signals.
- Ready for pooling (next file, sentinel_model.py).

### **Part 3: JK Connections & Per-Phase LayerNorm**

**JK (Jumping Knowledge) — Phase 1-A1:**
The three phases produce three distinct embeddings per node. Rather than throwing away Phase 1 and Phase 2 (keeping only Phase 3), we **learned-aggregate all three**. A small neural network (_JKAttention) computes a scalar attention weight per phase and combines them:

```
final_node_emb = α₁ × phase1_emb + α₂ × phase2_emb + α₃ × phase3_emb
                 (where Σαᵢ = 1, learned via softmax)
```

Why? Phase 1 captures *pure structural signal* before it gets oversmoothed by Phases 2+3. JK lets the network route different nodes to different phases — some nodes might rely purely on structure (modifiers, state variables), others on execution order (vulnerable operations).

**Per-Phase LayerNorm — Phase 1-A2:**
Phase 1 runs two conv layers (depth 2), while Phases 2 and 3 also run three conv each. After two convolutions, embeddings can have higher magnitude (L2 norm) than after three. This causes the JK softmax to be *dominated by Phase 1's magnitude* rather than learned phase routing. Solution: **apply LayerNorm once after each phase** before collecting for JK. This normalizes the magnitude across phases, letting attention learn actual routing rather than magnitude artifacts.

---

### **Summary: 3 Things to Lock In**

1. **11 edge types encode different graph semantics** — Structural (0–5, always on-disk), Directional (6–10, on-disk + runtime). The GNN uses different subsets in each phase because different phases solve different problems.

2. **Three-phase design:** Phase 1 propagates structure downward (function→CFG), Phase 2 captures execution order (CFG paths), Phase 3 propagates order back up to functions then back down to CFG. This is symmetrical: functions now influence their children's embeddings, children's execution patterns now influence functions.

3. **JK + LayerNorm solve oversmoothing:** Throwing away Phases 1 and 2 would lose important signals. JK lets the model blend all three phases with learned weights. LayerNorm ensures magnitude doesn't confound that learning.
