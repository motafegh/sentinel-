## 2. The Core Problem: Single Score vs Multi-Class Output

### What exists

```solidity
struct AuditResult {
    uint256 scoreFieldElement;  // ONE score (single float * 8192)
    bytes32 proofHash;
    uint256 timestamp;
    address agent;
    bool verified;
}

function submitAudit(
    address contractAddress,
    uint256 scoreFieldElement,   // one scalar
    bytes calldata proof,
    uint256[] calldata publicSignals
) external whenNotPaused {
    // Guard 3: publicSignals[64] == scoreFieldElement
    require(publicSignals[64] == scoreFieldElement, "score mismatch");
    ...
}
```

### What we need (after Run 13, 9-class output)

A way to store and verify 9 class probabilities (or a compact representation of them), linked to a ZK proof that confirms the proxy model produced those probabilities given the 128-dim fusion embedding.

The contracts README itself flags this: *"Known Limitation: AuditResult stores one scoreFieldElement — designed for a single risk scalar. Track 3 produces 10 per-class scores; a migration design is needed before enabling submit_audit in production."*

---

## 3. The Design Decision: What Goes On-Chain

This is the key decision that shapes everything else in contracts AND zkml. Three options:

### Option A: Full 9-class scores on-chain (RECOMMENDED)

```solidity
struct AuditResult {
    uint256[9] classScores;      // Each = prob * 8192, little-endian BN254 field element
    bytes32 proofHash;
    uint256 timestamp;
    address agent;
    bool verified;
}

function submitAudit(
    address contractAddress,
    uint256[9] calldata classScores,
    bytes calldata proof,
    uint256[] calldata publicSignals   // [128 fusion inputs..., class_0, ..., class_8]
) external whenNotPaused {
    // Guard 3 must verify ALL 9 scores against publicSignals
    for (uint256 i = 0; i < 9; i++) {
        require(publicSignals[128 + i] == classScores[i], "score mismatch");
    }
    ...
}
```

**Pros:** Full verifiability; on-chain consumers can check any class; aligns with SENTINEL's multi-label value proposition.

**Cons:** Gas cost increases (~9 requires vs 1 in Guard 3). calldata is larger (9 uint256 vs 1).

**Estimate:** submitAudit gas ~150k → ~200k. Still well within Ethereum limits. Acceptable.

### Option B: Single "risk score" + flagged class bitmask

```solidity
struct AuditResult {
    uint256 riskScore;        // weighted aggregation of per-class probs * 8192
    uint16  flaggedClasses;   // bitmask: bit i = class i is flagged (prob > threshold)
    bytes32 proofHash;
    ...
}
```

**Pros:** Cheap gas, compact. Backward-compatible with existing consumers.

**Cons:** Information loss — callers can't verify individual class probabilities. The ZK proof would only prove the aggregation, not per-class scores. Not aligned with SENTINEL's multi-label purpose.

### Option C: Commitment on-chain, scores off-chain

```solidity
struct AuditResult {
    bytes32 scoresCommitment;  // keccak256(abi.encode(classScores[9]))
    bytes32 proofHash;
    ...
}
```

**Pros:** Minimal gas, minimal calldata.

**Cons:** Verifying a specific class score requires the caller to fetch off-chain data and hash. Reduces the "trustless" property significantly — defeats the purpose of on-chain storage.

---

**Recommendation: Option A.** Full multi-class storage. SENTINEL's core product claim is per-class vulnerability detection — weakening this at the on-chain layer undermines the value proposition. The gas increase is acceptable for an oracle use case (audits are infrequent, not high-frequency transactions).

## 4. AuditRegistry Redesign (Option A Implementation)

### Struct migration

UUPS allows storage slot extension (add at the end). The current struct occupies:
- slot 0: `scoreFieldElement` (uint256)
- slot 1: `proofHash` (bytes32)
- slot 2: `timestamp` (uint256)
- slot 3: `agent` (address, 20 bytes, packed with bool)
- slot 3 cont: `verified` (bool, 1 byte)

**New struct (backward-compatible via new mapping):**

```solidity
// New mapping alongside old one (don't delete _audits — it has existing data)
mapping(address => AuditResultV2[]) private _auditsV2;

struct AuditResultV2 {
    uint256[9] classScores;    // index matches CLASS_NAMES order from graph_schema.py
    bytes32 proofHash;
    uint256 timestamp;
    address agent;
    bool verified;
}
```

The class index mapping (must match `graph_schema.py` CLASS_NAMES order after GasException removal):
```
0: Reentrancy
1: DenialOfService
2: ExternalBug
3: IntegerUnderOverflow
4: Timestamp
5: TransactionOrderDependence
6: UnusedReturn
7: CallToUnknown
8: MishandledException
```

**Note:** After GasException is dropped in Run 13, CLASS_NAMES will have 9 entries. The on-chain index must exactly match the order used by the zkml proxy output. Coordinate this with `graph_schema.py`.

### submitAudit redesign

```solidity
function submitAuditV2(
    address contractAddress,
    uint256[9] calldata classScores,
    bytes calldata proof,
    uint256[] calldata publicSignals
) external whenNotPaused {
    require(
        sentinelToken.stakedBalance(msg.sender) >= sentinelToken.MIN_STAKE(),
        "AuditRegistry: insufficient stake"
    );
    require(
        zkmlVerifier.verifyProof(proof, publicSignals),
        "AuditRegistry: invalid ZK proof"
    );

    // Guard 3: verify all 9 class scores match publicSignals
    // publicSignals layout: [128 fusion inputs, class_0_score, ..., class_8_score]
    uint256 INPUT_OFFSET = 128;  // after the 128-dim proxy input
    for (uint256 i = 0; i < 9; i++) {
        require(
            publicSignals[INPUT_OFFSET + i] == classScores[i],
            "AuditRegistry: class score mismatch"
        );
    }

    _auditsV2[contractAddress].push(AuditResultV2({
        classScores: classScores,
        proofHash: keccak256(proof),
        timestamp: block.timestamp,
        agent: msg.sender,
        verified: true
    }));

    emit AuditSubmittedV2(contractAddress, keccak256(proof), msg.sender, classScores);
}
```

### New events

```solidity
event AuditSubmittedV2(
    address indexed contractAddress,
    bytes32 proofHash,
    address indexed agent,
    uint256[9] classScores     // full per-class vector for off-chain indexing
);
```

The old `AuditSubmitted` event (single `scoreFieldElement`) should be kept for backward compatibility during transition.

### New query functions

```solidity
function getLatestAuditV2(address contractAddress) external view
    returns (AuditResultV2 memory);

function getAuditHistoryV2(address contractAddress) external view
    returns (AuditResultV2[] memory);

// Convenience: get decoded probability for a specific class (0-8)
function getClassProbability(
    address contractAddress,
    uint8 classIndex
) external view returns (uint256 scoreFieldElement, uint256 probability_times_1e4) {
    AuditResultV2 memory audit = getLatestAuditV2(contractAddress);
    uint256 raw = audit.classScores[classIndex];
    return (raw, (raw * 10000) / 8192);   // probability * 10000 (4 decimal places)
}
```

## 5. ZKMLVerifier Changes

The `ZKMLVerifier.sol` is auto-generated by EZKL. When zkml regenerates the circuit (after Run 13 proxy retrain), a new `ZKMLVerifier.sol` is generated with embedded BN254 verification keys.

**The `verifyProof` interface stays the same:**
```solidity
interface IZKMLVerifier {
    function verifyProof(
        bytes calldata proof,
        uint256[] calldata instances   // this is what changes in length
    ) external returns (bool);
}
```

The `instances` array (= `publicSignals` in Python) will change length when the proxy output changes from 10 to 9 classes. EZKL handles this automatically — the interface ABI stays compatible.

**After zkml regeneration:**
1. Copy `zkml/ezkl/ZKMLVerifier.sol` → `contracts/src/ZKMLVerifier.sol`
2. Compile with solc 0.8.17: `solc-select use 0.8.17 && forge build --contracts src/ZKMLVerifier.sol`
3. Deploy new ZKMLVerifier to Sepolia
4. Call `AuditRegistry.upgradeVerifier(newZKMLVerifierAddress)` — but this function doesn't exist yet! Add it.

**Missing function to add:**
```solidity
function upgradeVerifier(address newVerifierAddress) external onlyOwner {
    require(newVerifierAddress != address(0), "AuditRegistry: zero address");
    address old = address(zkmlVerifier);
    zkmlVerifier = IZKMLVerifier(newVerifierAddress);
    emit VerifierUpgraded(old, newVerifierAddress);
}

event VerifierUpgraded(address indexed oldVerifier, address indexed newVerifier);
```

## 10. Critical Technical Constraints (do not change)

| Constraint | Value | Why |
|---|---|---|
| solc — AuditRegistry, SentinelToken, IZKMLVerifier | **0.8.20** | Standard modern compiler |
| solc — ZKMLVerifier.sol ONLY | **≤ 0.8.17** | Halo2 assembly uses deprecated JUMP/PC opcodes |
| EZKL scale factor | **8192** (2¹³) | score = prob * 8192; both zkml and contracts must agree |
| BN254 little-endian decode | `int.from_bytes(..., byteorder='little')` | EZKL stores field elements little-endian |
| UUPS upgrade authorization | `onlyOwner` via `_authorizeUpgrade()` | Standard UUPS security |
| AuditRegistry `_disableInitializers()` | In constructor | Prevent re-initialisation attack |
| MIN_STAKE | 1000 SNTL (1e21 with decimals) | Economic guard; change requires community consensus |
